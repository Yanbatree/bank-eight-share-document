设计模式八股文

银行面试中的设计模式考察重点：设计模式是银行科技岗面试的高频考点，尤其关注单例模式、代理模式、策略模式和模板方法模式。面试官不仅会问"是什么"，更会追问"为什么这样设计"、"在 Spring 框架中哪里用到了"、"你在项目中怎么用的"。掌握每种模式背后的设计原则（开闭原则、单一职责、依赖倒置等）同样重要。

GoF 设计模式的六大原则：

1. 单一职责原则 (SRP)：一个类只负责一项职责。如果类承担过多职责，任意一个职责的变更都可能影响到其他职责。Spring 的 Controller-Service-Dao 分层就是 SRP 的体现。

2. 开闭原则 (OCP)：对扩展开放，对修改关闭。应通过扩展（新增类/方法）而非修改已有代码来实现新功能。策略模式、装饰器模式都是 OCP 的典型应用。

3. 里氏替换原则 (LSP)：子类必须能够替换其父类。任何使用父类引用的地方，替换为子类实例后行为应保持一致。

4. 接口隔离原则 (ISP)：不应强迫客户端依赖它不使用的方法。一个大接口应拆分为多个小接口。

5. 依赖倒置原则 (DIP)：依赖抽象而非具体实现。高层模块不应依赖低层模块，两者都应依赖抽象。Spring 的 IoC/DI 就是这个原则的最佳实践。

6. 迪米特法则 (LoD / 最少知识原则)：一个对象应对其他对象有尽可能少的了解，只与直接朋友通信。

二、创建型模式

1. 单例模式 (Singleton) 必问

定义：确保一个类只有一个实例，并提供一个全局访问点。

核心要点：构造器私有化 + 静态变量持有唯一实例 + 静态方法返回实例。

五种实现方式对比

实现方式线程安全是否懒加载防反射防序列化推荐度
饿汉式 (Eager)是否否否★★☆☆☆
实现方式： 
public class Singleton {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() {}
    public static Singleton getInstance() { return INSTANCE; }
}

原理：类加载时由 JVM 在 <clinit> 阶段初始化 static 变量。JVM 保证类初始化过程是线程安全的（加锁），多线程同时触发类加载时只有一个线程执行初始化。所以饿汉式天然线程安全。

缺点：非懒加载——无论用不用，类一加载实例就创建了。如果单例类很重（初始化耗时长、占内存大），会造成资源浪费。此外，饿汉式无法通过参数控制实例的创建过程。

防反射？可以通过反射调用私有构造器破坏单例。防序列化？反序列化会创建新对象。

懒汉式 (Lazy)否是否否☆☆☆☆☆
实现方式： 
public class Singleton {
    private static Singleton instance;
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();  // 非原子操作！
        }
        return instance;
    }
}

为什么不安全？假设线程 A 和线程 B 同时调用 getInstance()：A 执行到 if (instance == null) 时发现为 null，进入创建逻辑；此时 CPU 切换到 B，B 也发现 instance 为 null，也进入创建逻辑。结果：两个线程各自创建了一个实例，违背了单例原则。

一个简单的改进是给 getInstance 方法加 synchronized——但性能极差，每次获取实例都要竞争锁。这就是 DCL 要解决的问题。

DCL (Double-Checked Locking)是是否否★★★★☆
实现方式： 
public class Singleton {
    private static volatile Singleton instance;  // volatile 必不可少！
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {                  // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {          // 第二次检查（加锁）
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

为什么需要 volatile？——指令重排序问题：

instance = new Singleton(); 这一行代码在 JVM 层面不是一个原子操作，它分解为三个步骤：

① 分配内存空间（allocate memory）
② 初始化对象（调用构造器，init object）
③ 将引用指向内存地址（assign reference to variable）

由于 CPU 和 JIT 编译器的指令重排序优化，步骤②和③可能被交换：先执行③（将引用指向未初始化完成的内存），再执行②（初始化对象）。

如果不用 volatile 会怎样？线程 A 执行到 instance = new Singleton()，由于指令重排序，先执行了步骤①→③→②。此时 instance 已经不为 null（步骤③已完成），但对象尚未初始化（步骤②未执行）。线程 B 恰好此时调用 getInstance()，第一次检查发现 instance 不为 null，直接返回了一个未初始化完成的"半成品"对象——轻则读到 null 字段，重则程序崩溃。

volatile 的作用：volatile 禁止指令重排序——保证了 instance = new Singleton() 的三步操作严格按顺序执行，且 volatile 写操作之前的动作不会被重排到写操作之后。当线程 B 读到 volatile 变量 instance 的值时，一定也能看到 volatile 写操作之前的所有操作（Happens-Before 语义）。

面试扩展：JDK 5 之前的 volatile 语义不完整，DCL 模式在当时是有 bug 的。JDK 5 引入了新的内存模型（JSR-133），修复了 volatile 的语义，使 DCL 正确工作。

静态内部类 (Holder)是是否否★★★★★
实现方式： 
public class Singleton {
    private Singleton() {}
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}

原理：利用了 JVM 类加载机制的"延迟初始化"特性。外部类 Singleton 加载时，内部类 Holder 不会被加载。只有当 getInstance() 首次被调用、访问 Holder.INSTANCE 时，JVM 才加载 Holder 类并初始化 INSTANCE。而 JVM 保证类初始化过程是线程安全的（<clinit> 方法被 JVM 隐式加锁）。

为什么推荐？它同时实现了懒加载 + 线程安全，且代码简洁无锁，没有 volatile 的认知负担，没有 synchronized 的性能开销。这是 Effective Java 作者 Joshua Bloch 推荐的首选单例实现方式（当他没推荐枚举时）。

缺点：仍然无法防止反射攻击和序列化破坏。可以给构造器加判断抛出异常来防反射，但这不是完美方案。

枚举 (Enum)是是是是★★★★★
实现方式： 
public enum Singleton {
    INSTANCE;
    public void doSomething() { /* 业务逻辑 */ }
}
// 使用：Singleton.INSTANCE.doSomething();

为什么枚举是最好的单例实现？

1. 防反射：反射的 newInstance() 内部对枚举类型做了特殊判断，会抛出 IllegalArgumentException——"Cannot reflectively create enum objects"。这是 JVM 层面的保护。

2. 防序列化：Java 的序列化机制对枚举做了特殊处理——反序列化时不会创建新实例，而是通过 Enum.valueOf() 获取已存在的枚举常量。这是由 ObjectInputStream 的 readEnum() 方法保证的，无需实现 readResolve()。

3. 线程安全：枚举的实例创建由 JVM 在类加载时完成，自然线程安全。

4. 简洁：比任何其他实现都简洁，没有样板代码。

局限性：枚举不能被继承（因为 enum 隐式继承 java.lang.Enum，Java 是单继承）。如果单例类需要继承其他父类，就不能使用枚举。但大多数情况下，单例不需要继承。

Spring 的态度：Spring 框架中较少使用枚举单例（Spring 更倾向于通过容器管理 Bean 来实现单例），但在 JDK 源码和日常开发中，枚举是最安全的选择。

面试答题口诀：
饿汉式：类加载即创建，线程安全但不懒，浪费资源。
懒汉式：用才创建但不安全，加 synchronized 太慢。
DCL：双重检查 + volatile，高效但稍复杂，volatile 防止指令重排导致半成品对象。
静态内部类：利用类加载机制延迟初始化，简洁高效，推荐但不防反射/序列化。
枚举：最完美——防反射、防序列化、天然线程安全，唯一缺点是不能继承类。
Spring 框架中的单例模式：
Spring 容器默认管理的 Bean 都是单例（singleton scope）。Spring 采用ConcurrentHashMap（三级缓存：singletonObjects、earlySingletonObjects、singletonFactories）来存储单例实例，通过 DCL + synchronized 实现线程安全的 Bean 创建。这是在"容器层面"实现单例，而非在类层面，因此 Bean 类本身可以是非单例设计——这比传统的单例模式更灵活。

三种工厂模式的核心区别：

简单工厂：一个工厂类根据参数创建不同类型的产品。不符合开闭原则——新增产品需要修改工厂类的 if-else。严格来说不是 GoF 的 23 种设计模式之一，更像一种编程习惯。

工厂方法：定义一个创建对象的接口，让子类决定实例化哪个类。每个具体工厂只创建一种产品。符合开闭原则——新增产品只需新增一个具体工厂类。

抽象工厂：创建一组相关或相互依赖的对象（产品族），而不需要指定它们的具体类。例如：同一个 UI 工厂可以创建相匹配的 Button 和 TextField（Windows 风格或 Mac 风格）。

Spring BeanFactory 与工厂模式：

Spring 的 BeanFactory 可以看作是一个"超级简单工厂"——它通过 Bean 名称或类型从容器中获取对象，但产生对象的逻辑比传统工厂复杂得多（涉及依赖注入、AOP、生命周期管理等）。

更准确的类比：Spring 的 FactoryBean 接口是工厂方法模式的体现——getObject() 方法返回产品，getObjectType() 返回产品类型。MyBatis-Spring 的 SqlSessionFactoryBean 就是通过 FactoryBean 创建 SqlSessionFactory 的。

Spring 的 ApplicationContext 层次结构则体现了抽象工厂模式的思想——不同的 ApplicationContext 实现（ClassPathXmlApplicationContext、AnnotationConfigApplicationContext）创建出整套 Bean 实例（产品族）。

三、结构型模式

3. 代理模式 (Proxy) 必问

定义：为其他对象提供一种代理，以控制对这个对象的访问。代理对象和真实对象实现相同的接口。

银行面试最常问的代理模式考点：JDK 动态代理 vs CGLIB 代理的区别，以及 Spring AOP 如何选择代理方式。

三种代理方式

▶ 展开详解
JDK 动态代理的执行流程：
  1. 调用者通过代理对象调用方法（实际调用的是生成的代理类的方法）
  2. 代理类内部持有 InvocationHandler 引用，将调用委托给 handler.invoke()
  3. 在 invoke 方法中，我们可以织入增强逻辑（AOP 的 advice）
  4. 通过反射 method.invoke(target, args) 调用真实目标对象的方法
  5. 返回结果给调用者

为什么 JDK 动态代理必须基于接口？因为 JDK 动态代理生成的代理类 extends java.lang.reflect.Proxy，而 Java 是单继承的——代理类已经继承了 Proxy，不能再继承目标类，只能通过实现目标类所实现的接口来完成代理。这也是 Spring AOP 在目标类没有实现接口时需要切换到 CGLIB 的原因。

维度JDK 动态代理CGLIB 代理
原理通过实现目标接口生成代理类通过继承目标类生成子类作为代理

JDK 动态代理：使用 java.lang.reflect.Proxy.newProxyInstance() 在运行时动态生成一个实现了指定接口的代理类。这个代理类继承自 Proxy，方法调用通过反射转发到 InvocationHandler.invoke()。

CGLIB：使用 ASM 字节码框架在运行时动态生成目标类的子类。方法调用通过 FastClass 机制（不使用反射，而是通过索引直接定位方法）转发到 MethodInterceptor.intercept()。

目标要求必须实现至少一个接口不需要实现接口（但不能是 final 类）

JDK 代理要求目标类必须实现接口——因为生成的代理类 extends Proxy，只能通过 implements 接口来"伪装"成目标类型。

CGLIB 通过继承目标类生成代理，所以目标类不能是 final（final 类不能被继承）。同样，代理的目标方法不能是 final（final 方法不能被重写）。如果方法被 final 修饰，CGLIB 无法代理该方法，增强逻辑不会生效——这是一个常见的开发坑。

性能创建开销小，调用开销大（反射）创建开销大（字节码生成），调用快

JDK 动态代理在方法调用时依赖反射（Method.invoke），有性能损耗。虽然 JDK 内部对反射做了优化（如 MethodAccessor 的 inflation 机制，切换为生成的字节码访问器），但仍有一定开销。

CGLIB 使用 FastClass 机制——为代理类和目标类各自生成一个 FastClass，其中每个方法有一个 index，调用时直接用 index 查找而非反射。此外，CGLIB 还可以利用 MethodProxy 的 invokeSuper 直接调用父类方法，进一步减少开销。在现代 JDK（8+）中，由于反射的持续优化，两者差距已经不大。CGLIB 的劣势在于生成字节码耗时较长，代理类多时启动较慢。

Spring 中的选择目标有接口时默认使用目标无接口时使用
Spring AOP 的代理选择策略：

Spring AOP 默认使用 JDK 动态代理（如果目标类实现了至少一个接口）。如果目标类没有实现接口，则自动使用 CGLIB。

从 Spring Boot 2.0 / Spring Framework 5 开始，默认的代理策略发生了变化：spring.aop.proxy-target-class 默认值改为 true，即默认使用 CGLIB 代理。这是因为现代项目中，很少有人为每个 Service 都抽取接口，CGLIB 更符合实际开发习惯。

可以通过配置强制使用 CGLIB：@EnableAspectJAutoProxy(proxyTargetClass = true)。

注意：Spring AOP 是方法级别的代理，不能拦截方法内部的 self-invocation（同一个类中方法 A 调用方法 B，B 的增强逻辑不会生效）。这是因为调用 this.methodB() 时，this 指向的是目标对象本身，而非代理对象。解决方式：通过 AopContext.currentProxy() 获取当前代理对象，或者将方法拆分到不同的 Bean 中。

静态代理 vs 动态代理：
静态代理：代理类在编译时已确定，需要为每个目标类手动编写一个代理类。类文件数量随业务增长爆炸。
动态代理：代理类在运行时动态生成，一个 InvocationHandler 可以处理多个目标类的方法调用。Spring AOP 的核心就是动态代理。

4. 装饰器模式 (Decorator)

定义：动态地给一个对象添加额外的职责，比继承更灵活。

核心思想：装饰器和被装饰对象实现同一个接口，装饰器内部持有被装饰对象的引用，在调用前后可以添加额外的行为。

Java IO —— 装饰器模式的经典应用

// 最内层：提供基本的字节读取能力
InputStream raw = new FileInputStream("data.txt");

// 第一层装饰：添加缓冲，减少系统调用，提升性能
InputStream buffered = new BufferedInputStream(raw);

// 第二层装饰：添加基本类型读取能力（readInt、readDouble 等）
DataInputStream data = new DataInputStream(buffered);

// 使用：可以一行行读取，也可以读取 int/double 等类型化数据
int value = data.readInt();  // 缓冲读取 + 类型化解析
维度装饰器模式代理模式
关注点增强对象功能（添加新行为）控制对象访问（权限/延迟/日志）

装饰器模式的目的是"给对象增加新能力"——比如给 InputStream 添加缓冲、加密、压缩等能力。每个装饰器层增加一项独立功能，装饰器之间可以任意组合。

代理模式的目的是"控制对目标对象的访问"——比如远程代理（RMI）、虚拟代理（延迟加载大对象）、保护代理（权限控制）。代理通常不改变原有行为，只在调用前后做一些控制。

关系装饰器持有被装饰对象引用（外部传入）代理内部自行创建/持有目标对象

装饰器模式的典型特征：装饰器通过构造器接收被装饰对象（依赖注入），调用方决定装饰层的组合顺序。例如 new DataInputStream(new BufferedInputStream(new FileInputStream("f")))。

代理模式中，代理对象通常对调用方隐藏目标对象的存在——调用方以为自己直接在使用目标对象，实际上使用的是代理。例如 Spring AOP 返回的代理 Bean，调用方无需感知代理的存在。

典型例子Java IO 流体系Spring AOP、MyBatis Mapper 代理

Java IO 流是装饰器模式的最经典案例：

InputStream 是抽象组件，FileInputStream、ByteArrayInputStream 是具体组件，FilterInputStream 是抽象装饰器（继承 InputStream 并持有一个 InputStream），BufferedInputStream、DataInputStream、CheckedInputStream 等是具体装饰器。每一层装饰器都在前一层的基础上增加一项新的能力，各层职责单一且可任意组合。

如果不用装饰器模式而用继承，就需要为每种功能组合创建子类（BufferedFileInputStream、BufferedDataFileInputStream、DataBufferedFileInputStream...），会导致类爆炸。

5. 适配器模式 (Adapter)

定义：将一个类的接口转换成客户希望的另一个接口，使原本因接口不兼容而不能一起工作的类可以协同工作。

比喻：电源适配器——把 220V 转换成手机需要的 5V。

Spring MVC HandlerAdapter —— 适配器模式的经典应用

Spring MVC 中，不同类型的 Controller（或 Handler）有不同的接口：普通 Controller 实现了 Controller 接口，注解驱动的 Controller 使用 @RequestMapping 注解，Servlet 本身也可以作为 Handler。DispatchServlet 如何统一调用这些不同类型的 Handler？

答案就是 HandlerAdapter：

public interface HandlerAdapter {
    boolean supports(Object handler);           // 是否支持该 handler
    ModelAndView handle(HttpServletRequest req,
                        HttpServletResponse res,
                        Object handler);        // 调用 handler
}

// 实现类示例：
// SimpleControllerHandlerAdapter  → 处理普通的 Controller 接口
// RequestMappingHandlerAdapter   → 处理 @RequestMapping 注解
// HttpRequestHandlerAdapter      → 处理 HttpRequestHandler 类型
// SimpleServletHandlerAdapter    → 处理 Servlet 类型

DispatchServlet 遍历所有 HandlerAdapter，找到支持当前 Handler 的一个，然后调用 adapter.handle()——这就是适配器模式："将不同接口的 Handler 适配成统一的 handle() 调用方式"。

适配器模式 vs 装饰器模式 vs 代理模式：
适配器：改变接口——让不兼容的接口变得兼容（220V → 5V）。适配前后的接口不同。
装饰器：增强功能——保持原有接口不变，添加新能力。装饰前后的接口相同。
代理：控制访问——保持原有接口不变，在访问前后做控制。代理前后的接口相同。

四、行为型模式

6. 策略模式 (Strategy) 高频

定义：定义一系列算法，把它们封装起来，并使它们可以相互替换。策略模式让算法的变化独立于使用算法的客户端。

核心价值：消除大量 if-else 或 switch-case 的分支判断，符合开闭原则。

▶ 展开详解
策略模式的标准实现步骤：
  1. 定义策略接口（如 PaymentStrategy，有一个 pay(amount) 方法）
  2. 实现具体策略类（AliPayStrategy、WechatPayStrategy、CardPayStrategy）
  3. Context 类持有策略引用，通过构造器或 setter 注入
  4. Context 调用 strategy.pay(amount) 时，实际执行的是具体策略的支付逻辑

消除 if-else 的方式：可以配合工厂模式或依赖注入，在系统初始化阶段将所有策略实现注册到一个 Map 中（如 paymentType → Strategy 的映射），运行时通过 key 直接获取对应的策略实例，完全消除分支判断。

策略模式在 Spring 中的应用

1. Spring 的 Resource 抽象：org.springframework.core.io.Resource 接口可以有多种策略实现——ClassPathResource、FileSystemResource、UrlResource、ServletContextResource。ApplicationContext 根据资源路径的前缀（classpath: / file: / http:）自动选择合适的 Resource 实现。这是策略模式 + 工厂模式的组合应用。

2. Spring Security 的 PasswordEncoder：BCryptPasswordEncoder、SCryptPasswordEncoder、NoOpPasswordEncoder 等不同编码策略，运行时根据配置选择。

3. Spring 事务管理器的选择：DataSourceTransactionManager、JtaTransactionManager、HibernateTransactionManager——都是 PlatformTransactionManager 接口的不同策略实现。

银行项目实战场景：
在银行支付系统中，不同支付渠道（网银支付、快捷支付、代扣、代付）的核心流程相似但细节各异。使用策略模式将每个支付渠道封装为独立的策略类，新增渠道只需新增一个策略类并注册到策略工厂，不修改已有代码。配合 Spring 的依赖注入，可以使用 Map<String, PaymentStrategy> 自动收集所有策略实现（Spring 会将所有 PaymentStrategy 类型的 Bean 注入到这个 Map 中，key 为 Bean 名称）。

7. 观察者模式 (Observer)

定义：定义对象间的一对多依赖关系，当一个对象的状态发生改变时，所有依赖它的对象都会得到通知并自动更新。

别名：发布-订阅模式 (Publish-Subscribe)。

Spring 事件机制的使用步骤：
  1. 定义事件类：继承 ApplicationEvent（或直接使用任意对象，Spring 4.2+ 支持）
  2. 发布事件：通过 ApplicationEventPublisher.publishEvent()
  3. 监听事件：使用 @EventListener 注解或实现 ApplicationListener 接口
  4. 默认同步执行（在同一个线程中顺序调用所有监听器），可通过 @Async + @EnableAsync 切换为异步

实际应用场景：用户注册后——发送欢迎短信、赠送新人优惠券、记录注册日志。这三件事与注册核心逻辑解耦，通过事件发布让各模块独立处理。

JDK 内置的 Observable/Observer：已被标记为 deprecated（JDK 9+）。因为它基于类而非接口（Observable 是类，Java 单继承限制了灵活性），且不支持事件类型的区分和异步处理。Spring 的事件机制是更好的替代方案。

观察者模式在银行业务中的应用：
转账完成后，需要触发一系列后续操作：通知收款方、更新账户余额缓存、上报反洗钱系统、记录交易流水、发送短信通知。使用 Spring 事件机制，转账核心服务只需发布一个 TransferEvent，各后续操作通过 @EventListener 异步处理，实现业务解耦。

8. 模板方法模式 (Template Method) 高频

定义：在父类中定义一个操作中的算法骨架，将一些步骤延迟到子类中实现。子类可以不改变算法结构，重新定义某些步骤。

核心思想："好莱坞原则"——"Don't call us, we'll call you."父类调用子类，而不是子类调用父类。

Spring 中的模板方法模式

Spring 类模板方法子类可重写的钩子方法
JdbcTemplateexecute() / query() / update()RowMapper.mapRow() / PreparedStatementSetter

JdbcTemplate 是如何运用模板方法模式的：

JdbcTemplate 的 query() 方法定义了固定的算法骨架：获取连接 → 创建 PreparedStatement → 设置参数 → 执行查询 → 遍历 ResultSet（模板骨架：驱动查询流程）→ 将 ResultSet 的每一行映射为对象（变化点，由 RowMapper 实现） → 关闭资源。所有资源管理、异常处理、事务同步代码都在模板中固定写好，开发者只需实现 RowMapper 这一个变化点。

在传统的 JDBC 编码中，获取连接、创建 Statement、关闭资源这些样板代码需要重复编写，容易遗漏资源关闭。JdbcTemplate 封装了这些不变的部分，让开发者专注于业务逻辑——这正是模板方法模式的核心价值。

RestTemplate 的模板骨架：准备 URI → 创建请求 → 执行 HTTP 调用 → 提取响应 → 处理异常 → 转换响应结果。开发者只需提供 URL、请求参数、期望的响应类型。底层的连接管理、超时重试等逻辑由模板统一处理。

TransactionTemplate 定义了事务管理的模板骨架：获取事务 → 执行业务逻辑 → 提交/回滚 → 清理资源。开发者只需在 doInTransaction() 中编写业务逻辑，事务的开启/提交/回滚/异常处理由模板处理。

RedisTemplate 的模板骨架：获取连接 → 序列化 key/value → 执行 Redis 命令 → 反序列化结果 → 释放连接。开发者只需关注"执行什么 Redis 命令"，连接管理、序列化、异常处理等由模板统一处理。

模板方法模式 vs 策略模式：
模板方法：用继承实现，父类控制流程，子类填充细节——适合"主流程固定、部分步骤可变"的场景。
策略模式：用组合实现，Context 持有策略接口，运行时替换整个算法——适合"整个算法可以完全替换"的场景。
Spring 中两个模式经常组合使用：JdbcTemplate 通过不同的 RowMapper 策略来处理结果集的映射。

9. 责任链模式 (Chain of Responsibility)

定义：将请求的发送者和接收者解耦，让多个对象都有机会处理请求。将这些对象连成一条链，沿着链传递请求，直到某个对象处理它为止。

核心结构：每个处理器（Handler）持有下一个处理器的引用。如果当前处理器不能处理，则传递给下一个；如果能处理，可以选择是否继续传递。

401/403 等"]
▶ 展开详解
Servlet Filter 链是责任链模式的标准实现：

每个 Filter 都实现 doFilter(ServletRequest, ServletResponse, FilterChain) 方法。在方法内部调用 chain.doFilter(request, response) 将请求传递给下一个 Filter。如果某个 Filter 决定不传递（如认证失败直接返回 401），责任链在这一点就中断了——请求不会到达目标 Servlet。

这是一个"纯责任链"——请求在链上传递，直到被某个处理器处理或被拒绝。Spring Security 的 SecurityFilterChain 在此基础上增加了更丰富的安全逻辑。

Spring 中的责任链模式应用

组件模式体现说明
Servlet FilterChain标准的责任链多个 Filter 按顺序过滤请求，每个 Filter 决定是否放行

在 web.xml 或 @WebFilter 中配置 Filter 和它们的执行顺序。每个 Filter 在 doFilter 方法中：1) 对请求做预处理，2) 调用 chain.doFilter() 传递给下一个，3) 对响应做后处理。当某个 Filter 不调用 chain.doFilter() 时（如认证失败），请求不会到达目标 Servlet，而是在该 Filter 处被拦截返回。

Spring Security FilterChain扩展的责任链多个 Security Filter（认证、鉴权、CSRF、Session管理等）组成链

Spring Security 的过滤器链比 Servlet Filter 更复杂——它内部可能包含多条链（通过 SecurityFilterChain 接口），根据请求 URL 匹配不同的链。例如 /api/** 走一条无状态的 JWT 认证链，/admin/** 走另一条有 Session 的表单登录链。

Security 的拦截器链采用"投票机制"——每个 Filter 可以对请求投赞成/反对/弃权票，最终根据投票结果决定是否放行。

Spring InterceptorHandlerInterceptor 链DispatcherServlet 到 Controller 之间，由多个拦截器组成链

Spring 的 HandlerInterceptor 有 3 个拦截点：preHandle（Controller 执行前）、postHandle（Controller 执行后、视图渲染前）、afterCompletion（视图渲染后、请求完成时）。多个 Interceptor 按注册顺序依次执行 preHandle，再按逆序执行 postHandle 和 afterCompletion。

与 Filter 的区别：Filter 是 Servlet 层面的（在到达 DispatcherServlet 之前），Interceptor 是 Spring 层面的（在 DispatcherServlet 内部）。Filter 优先级更高。

AOP 的拦截器链AspectJ MethodInvocation多个 Advice 环绕目标方法，按优先级形成调用链

当一个 Bean 的方法被多个切面拦截时，Spring 会将这些 Advice 组织成一条"责任链"——ReflectiveMethodInvocation 持有拦截器链，按顺序执行每个 Advice 的 before、proceed（传递）、after 等逻辑。proceed() 方法的调用本质上就是责任链的"传递给下一个"。

面试技巧——如何回答"Spring 中哪里用到了责任链模式"：
优先答 Servlet FilterChain（最经典、最容易被认可）→ 其次答 Spring Security SecurityFilterChain → 再答 Spring Interceptor 执行链 → 最后可以补充 AOP 的 Advice 链（加分项，体现对 Spring 的深入理解）。银行面试官一般听到前两个答案就满意了。

五、易混淆模式对比

模式对关键区别一句话总结
代理 vs 装饰器代理控制访问，装饰器增强功能代理是"有人替我开门"，装饰器是"开门前先消毒"

代理模式的目标是控制访问——延迟创建（虚拟代理）、权限检查（保护代理）、远程调用（远程代理）、缓存（缓存代理）。代理对象通常与目标对象同时创建。

装饰器模式的目标是增强功能——给对象增加新能力，而且装饰器可以自由叠加组合（如 BufferedInputStream + DataInputStream）。装饰器的组合顺序由调用方控制。

代理 vs 适配器代理保持接口不变，适配器改变接口代理是"同一个接口下的中间人"，适配器是"接口转换器"

代理模式中，代理类和目标类实现同一个接口——调用方无需知道自己拿到的是代理还是目标对象。

适配器模式中，适配器的目的是让"不兼容的接口"可以一起工作——例如将 220V 电流适配为 5V 供电给手机。适配器实现了客户端期望的接口，内部持有被适配者的引用，完成接口转换。

模板方法 vs 策略继承（父类控制骨架）vs 组合（接口定义算法）模板方法是"骨架你定，细节我填"；策略是"整个算法都可以换"

模板方法模式使用继承——父类定义算法骨架，子类重写特定步骤。控制权在父类，子类只能填充但不能改变流程。

策略模式使用组合——Context 持有策略接口引用，可以运行时切换整个算法实现。控制权在 Strategy 实现类，Context 只是一个调用者。

选择原则：如果算法的主流程相对稳定、只有部分步骤会变化，用模板方法。如果整个算法都可能被替换，用策略模式。

工厂方法 vs 模板方法关注对象创建 vs 关注行为流程工厂方法是"生产一个产品的模板"，模板方法是"执行一个任务的模板"

工厂方法模式是模板方法模式的一种特殊应用——"创建对象"是"算法"的一种。工厂方法中，父类定义了获取对象的模板流程，子类决定创建哪种具体的产品。

在很多框架中（如 Spring 的 AbstractApplicationContext），工厂方法（getBeanFactory()、refreshBeanFactory()）被嵌入到模板方法（refresh()）中协同工作。

责任链 vs 装饰器链中某一节点处理即终止 vs 每一层都执行责任链是"找到能处理的人就停下"，装饰器是"每层都过一遍"

责任链模式中，请求沿链传递，某个处理器处理完成后可能不再传递（纯责任链）。例如 Spring Security 中的某个 Filter 认证失败，直接返回 401，后续 Filter 不再执行。

装饰器模式中，请求会经过每一层装饰器——最外层装饰器调用内层装饰器，直到最内层的真实对象，然后层层返回。每一层都会执行。

各原则的面试考察方式：

单一职责原则：面试官会让你分析一段"大而全"的代码有什么问题。要点：一个类做了多个不同的职责，任何一方的变更都可能影响其他方。

开闭原则：银行面试最爱结合策略模式考察——"如何在不修改现有代码的前提下增加一个新的支付方式？"答案就是策略模式 + 依赖注入。

里氏替换原则：经典的"正方形是否是长方形的子类"问题。数学上正方形是长方形，但编程中如果正方形重写 setWidth/setHeight 导致行为不一致，就违反了 LSP。这告诉我们——继承关系不应只看"is-a"的语义，还要看行为兼容性。

依赖倒置原则：Spring 的核心理念。传统的做法是 Service 自己 new 一个 Dao；依赖倒置后，Service 声明需要什么类型的 Dao（接口），Spring 容器负责注入具体实现。这就是"控制反转 (IoC)"。

银行面试 SOLID 经典问题：
问："依赖倒置原则和依赖注入有什么关系？"
答：依赖倒置原则 (DIP) 是一种设计原则——高层模块不应依赖低层模块，两者都应依赖抽象。依赖注入 (DI) 是实现 DIP 的具体技术手段——通过外部容器将依赖（通常是实现了抽象接口的具体类）注入到对象中，而非在对象内部自行创建。Spring 的 IoC 容器就是 DI 的实现。三者关系：DIP 是原则 → DI 是实现方式 → Spring IoC 是框架。
