A股AI投资顾问 项目八股文

基于 DeepSeek ReAct Agent 的实时A股智能分析系统 — 从幻觉问题到架构设计的完整实践

一、项目概述

一句话：基于大语言模型（DeepSeek）的实时A股智能分析系统。用户输入股票代码/名称或自然语言问题，系统自动从5个数据源获取实时行情、K线、估值、公告、新闻和全网舆情，由4个并行ReAct Agent进行多维度分析并生成投资建议报告。 核心能力：AI诊股 / 智能选股 / 行情一览 / 自然语言交互

1.1 项目亮点速览

维度方案亮点
核心架构ReAct Agent（LangChain + LangGraph）LLM主动调工具获取数据，而非被动读取
传统方案是"代码拉数据→拼Prompt→LLM读"，问题是LLM不信任被动接收的数据。ReAct让LLM自己调工具获取数据，从认知层面解决了数据信任问题。
AI 幻觉5层防御体系从Prompt→工具→代码注入，逐层拦截
Layer1时间注入→Layer2时间工具→Layer3 Prompt覆盖→Layer4 报告锚点→Layer5 代码注入Header（终极防线）。任何一层失效都有下一层兜底。
查询理解两层匹配策略正则快匹(<1ms) + LLM改写(~1s)，覆盖口语化查询
"BYD怎么样"、"宁王什么价"、"大哥们茅台还能上吗"等非标准查询都能正确解析。第1层正则处理65%的标准化查询，第2层LLM+NER处理别名/缩写/昵称。

二、核心理念：为什么用 ReAct Agent 而不是传统 Prompt 注入？

核心技术洞察：当LLM被动接收数据时（"请分析以下数据：最新价93.51元"），它倾向于用训练记忆覆盖。但当LLM主动调用工具拿到返回结果时，它将结果视为"自己获取的事实"而信任使用。数据获取的"主动性"决定了LLM对数据的信任程度。这是本项目经过4轮迭代才发现的深刻洞察。

2.1 4轮迭代过程

轮次方案原理结果
1Prompt 中强调"使用给定数据"寄希望于 LLM 听指令❌ LLM 不信任被动接收的数据

3两阶段: Stage1 提取事实 → Stage2 分析让 LLM 先"读"一遍数据⚠️ Stage2 仍然会覆盖

▶ 展开详解
为什么用4个Agent并行而不是1个大Agent？

关注点分离：每个Agent的Prompt更短、更精确，LLM不容易"走神"。容错：一个Agent失败不影响其他三个。交叉验证：Summary Agent可以发现一致点和分歧点（基本面看好但技术面看空本身就是有价值的信号）。并行加速：总耗时≈最慢的那个Agent，而不是4个串行之和。

为什么用 asyncio.gather 而不是 LangGraph StateGraph？

我们的场景是纯并行无依赖（4个Agent之间没有数据传递），asyncio.gather比StateGraph的fan-out/fan-in更简洁直接。StateGraph适合有复杂依赖关系的DAG，这里用更轻量的方案就够。

五、工具层 — 11个LangChain工具

5.1 设计原则

三个原则：
  1. 单一职责：一个工具只做一件事（获取行情/获取K线/搜索新闻），LLM自由组合调用顺序
  2. 返回结构化字符串：工具返回人类可读的格式化文本，LLM直接从中提取数值
  3. 带描述性docstring：docstring就是给LLM看的"使用手册"，LLM根据描述决定何时调用

5.2 工具清单

#工具名输入数据来源返回内容
1get_current_datetime无datetime.now()当前日期时间+星期
2get_realtime_quote6位代码新浪直连 + 东方财富最新价、涨跌额/幅、今开昨收最高最低、成交量/额、PE、PB、市值、ROE、换手率等
最核心的工具。新浪API提供价格和成交量，东方财富API提供PE/PB/ROE/市值等估值数据，两者合并为完整的数据结构，共20+个字段。如果东方财富不可用，估值字段显示"--"而不是编造。
3get_kline_data6位代码 + 天数新浪K线APIN日OHLCV数据（日期/开/收/高/低/量）
新浪K线API返回日K线数据。默认30天，缓存1小时（K线一天才变一次）。
4get_moving_averages6位代码 + 天数计算（基于K线）MA5/MA10/MA20 + 股价相对位置 + 偏离百分比
不是从外部API获取，而是基于K线数据在代码中纯数学计算。数学计算是确定性的，不会出错。LLM拿到的是已经算好的精确数值（如"MA5: 94.74元 (股价下方, 偏离-1.3%)"），不需要自己做任何计算。
5get_support_resistance6位代码 + 天数计算（基于K线）近N日最低点(支撑) + 最高点(压力)
同样基于K线纯计算：support = min(近N日最低价), resistance = max(近N日最高价)。
6get_stock_news6位代码东方财富(AKShare)个股最新5条新闻标题
7get_stock_announcements6位代码巨潮资讯网(AKShare)官方公告：日期+标题+链接
8get_cls_flash_news无财联社(AKShare)盘中突发快讯，最新10条
9search_web_bocha搜索关键词博查API全网舆情：标题+AI摘要
10get_market_indices无新浪直连上证/深证/创业板指数
11search_stock名称/代码本地缓存(AKShare股票列表)匹配的股票代码+名称

5.3 工具实现示例

@tool
def get_realtime_quote(code: str) -> str:
    """Get real-time stock quote. Input: 6-digit stock code like '600519' or '002594'.
    Returns: name, latest price, change amount, change%, open, prev close, high, low,
    volume, amount, PE, PB, market cap, turnover rate, ROE, amplitude.
    All numbers are from exchange API — use them directly, do NOT substitute with training data.
    """
    q = _get_quote(code)
    if not q:
        return f"未找到股票 {code} 的实时行情数据"
    lines = [
        f"股票代码: {q.get('代码', code)}",
        f"股票名称: {q.get('名称', '')}",
        f"最新价: {q.get('最新价', '--')} 元",
        f"涨跌额: {q.get('涨跌额', '--')} 元",
        f"涨跌幅: {q.get('涨跌幅', '--')}%",
        # ... 20+ 字段
    ]
    return "\n".join(lines)
关键设计点：
  • code: str — LLM通过type hints知道需要传6位代码字符串
  • -> str — LLM知道返回的是文本，可以直接阅读
  • docstring中的 "do NOT substitute with training data" — 反幻觉的关键指令
  • @tool 装饰器自动将函数转为LangChain工具，读取type hints生成参数schema，读取docstring生成工具描述

6.1 ReAct模式原理

ReAct = Reasoning + Acting。LLM在每一步思考需要什么信息→决定调用哪个工具传什么参数→接收工具返回结果→继续思考→信息足够时输出最终答案。langgraph.prebuilt.create_react_agent(llm, tools)封装了整个ReAct循环逻辑。

6.3 System Prompt设计

⚠️ 核心规则：

_FUNDAMENTAL_PROMPT = _BASE_SYSTEM + """

1. get_realtime_quote("{code}") — 获取估值数据

  • 每个Agent专注一个领域：Prompt更短更精准，LLM不容易"走神"

def _get_em_session() -> requests.Session:
    """东方财富直连Session — 绕过系统代理"""
    session = requests.Session()
    session.trust_env = False                    # 不信任系统代理设置
    session.proxies = {"http": None, "https": None}  # 显式禁用代理
    session.headers["Referer"] = "https://quote.eastmoney.com/"
    return session

# 新浪API天然不需要绕过(.cn域名)
def _get_sina_session() -> requests.Session:

#         "rewritten": "分析宁德时代(300750)当前投资价值"}

System Prompt中包含常见别名映射（BYD→比亚迪、宁王→宁德时代、茅台→贵州茅台），LLM举一反三。温度0确保输出稳定，耗时~1s，补充至85%。

防缓存击穿：每个key有独立的threading.Lock。当缓存过期时，第一个请求拿到锁并更新缓存，其他请求在锁外等待缓存就绪。这避免了高并发下所有请求同时打到外部API。 存储：SQLite持久化缓存。简单可靠，零外部依赖。

面试常见追问准备

▶ 展开追问与回答
Q: ReAct Agent 和 RAG 有什么区别？

RAG是"先检索，再生成"——代码预检索相关内容注入Prompt，LLM被动阅读。ReAct Agent是"边想，边做"——LLM自主决定什么时候需要什么数据，主动调用工具获取。RAG适用于知识问答场景（文档检索），ReAct适用于需要多轮交互和工具调用的场景。

Q: 为什么用4个Agent而不是1个？

三个原因：(1)关注点分离——每个Agent的Prompt更短更精准，不容易"走神"；(2)容错——一个Agent失败不影响其他三个；(3)交叉验证——Summary Agent可以发现一致点和分歧点，分歧本身就是有价值的信号。

Q: DeepSeek 和 GPT 在你这个项目里有什么区别？

项目用的是DeepSeek V4。选择原因：(1)成本极低——1M token输入约1元；(2)中文理解能力强；(3)支持Function Calling（工具调用）。缺点是对长上下文的指令跟随略弱于GPT-4，需要通过严格的System Prompt和多层防御来弥补。

Q: 如果数据源全部不可用怎么办？

目前的核心降级策略是：估值失败显示"--"（不编造）、公告失败跳过公告分析、搜索失败降级到DuckDuckGo、行情失败返回None给用户提示。理想情况下应该加入"最后交易日收盘数据"作为第三级降级——但这需要持久化每日收盘快照。

Q: asyncio.gather 和 LangGraph StateGraph 怎么选？

纯并行无依赖的场景用asyncio.gather（更轻量），有复杂依赖关系和条件路由的场景用StateGraph。本项目4个Agent完全独立并行，所以用gather就够了。如果以后需要条件分支（比如技术面看空时跳过估值分析），就应该用StateGraph。

Q: 缓存为什么用SQLite而不是Redis？

这是个人项目，SQLite零配置、零外部依赖、一个文件搞定。如果做成生产级服务，应该用Redis——支持分布式、连接池、更精细的过期策略。

Q: Temperature 为什么选0.3？

金融分析需要准确性和一致性，不需要创造性。0.3是一个平衡点——既保证了分析多样性（不同Agent可能得出不同结论），又不会因为过度创造而编造数据。实际测试中0.3产生最稳定的输出质量。
