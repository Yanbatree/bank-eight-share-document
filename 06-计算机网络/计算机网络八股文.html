3. TCP 三次握手

sequenceDiagram participant C as 客户端 participant S as 服务器 Note over C,S: 第一次握手 C->>S: SYN=1, seq=x Note over S: 收到 SYN，状态 → SYN_RCVD Note over C,S: 第二次握手 S->>C: SYN=1, ACK=1, seq=y, ack=x+1 Note over C: 收到 SYN+ACK，状态 → ESTABLISHED Note over C,S: 第三次握手 C->>S: ACK=1, seq=x+1, ack=y+1 Note over S: 收到 ACK，状态 → ESTABLISHED
为什么是三次，不是两次？防止已失效的连接请求到达服务端导致错误建立连接。如果是两次握手，服务端收到过期的 SYN 就会建立无用的连接。 为什么不是四次？三次已经可以确认双方收发能力正常（客户端收/发正常，服务端收/发正常），四次没有必要。

4. TCP 四次挥手

sequenceDiagram participant C as 主动关闭方 participant S as 被动关闭方 C->>S: FIN=1, seq=u Note over S: 收到 FIN → CLOSE_WAIT S->>C: ACK=1, seq=v, ack=u+1 Note over C: 收到 ACK → FIN_WAIT_2 Note over S: 处理剩余数据... S->>C: FIN=1, ACK=1, seq=w, ack=u+1 Note over C: 收到 FIN → TIME_WAIT C->>S: ACK=1, seq=u+1, ack=w+1 Note over S: 收到 ACK → CLOSED Note over C: 等待 2MSL → CLOSED
为什么是四次？TCP 全双工，双方都需要发送 FIN + ACK。服务端收到 FIN 后可能还有数据要发，所以 ACK 和 FIN 分开发送。

5. TIME_WAIT 为什么是 2MSL？

MSL = Maximum Segment Lifetime（报文最大存活时间，Linux 默认 30s）。 2MSL 的原因： ① 确保最后的 ACK 能到达：如果服务端没收到最后的 ACK，会重发 FIN，客户端需要在 TIME_WAIT 期间能重发 ACK ② 让旧连接的所有报文在网络中消失：防止旧连接的报文影响新连接 大量 TIME_WAIT 的问题：占用端口和系统资源。解决方案：net.ipv4.tcp_tw_reuse、SO_REUSEADDR。

7. 流量控制 & 拥塞控制

流量控制（端到端）：接收方通过窗口大小（rwnd）告知发送方自己能接收多少，防止发送方太快把接收方"淹没"。 拥塞控制（网络层面）： 
① 慢启动：cwnd 从 1 开始，指数增长（每 RTT 翻倍）
② 拥塞避免：cwnd ≥ ssthresh 后，线性增长（每 RTT +1）
③ 快重传：连续收到 3 个重复 ACK 立即重传（不等超时）
④ 快恢复：收到 3 个重复 ACK，ssthresh = cwnd/2，cwnd = ssthresh，进入拥塞避免

3. HTTPS 握手过程（TLS 1.2）

sequenceDiagram participant C as 客户端 participant S as 服务端 Note over C,S: ① Client Hello C->>S: TLS 版本 + 随机数1 + 密码套件列表 Note over C,S: ② Server Hello S->>C: 选中的 TLS 版本 + 随机数2 + 选中的密码套件 S->>C: ③ Certificate: 服务器证书（含公钥） S->>C: ④ Server Hello Done Note over C: ⑤ 验证证书（CA 签名），生成随机数3 C->>S: ⑥ Client Key Exchange：用服务器公钥加密随机数3 C->>S: ⑦ Change Cipher Spec：后续通信加密 C->>S: ⑧ Finished（加密的） Note over S: 用私钥解密得到随机数3 Note over C,S: 双方用随机数1+2+3 生成对称密钥 S->>C: ⑨ Change Cipher Spec + Finished Note over C,S: ✅ 握手完成，对称加密通信
混合加密：非对称加密（RSA/ECDHE）交换密钥 + 对称加密（AES）传输数据。原因是非对称加密计算量大，不适合大量数据。

5. HTTP 状态码

类别常见状态码
1xx 信息100 Continue
2xx 成功200 OK, 201 Created, 204 No Content
3xx 重定向301 永久（搜索引擎换URL）, 302 临时, 304 Not Modified（缓存）
4xx 客户端错误400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 405 Method Not Allowed
5xx 服务端错误500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout
301 vs 302：301 永久重定向（浏览器缓存，下次直接走新地址）；302 临时重定向（每次都要问原地址）。

6. Cookie / Session / Token (JWT)

graph TD subgraph "Cookie-Session 模式" A["登录"] --> B["服务端创建 Session 存内存/Redis"] B --> C["Set-Cookie: sessionId=xxx"] C --> D["每次请求自动带 Cookie"] D --> E["服务端查 Session 验证身份"] end subgraph "JWT Token 模式" F["登录"] --> G["服务端签发 JWT
{header}.{payload}.{signature}"] G --> H["客户端存 localStorage/Cookie"] H --> I["每次请求手动带 Authorization: Bearer token"] I --> J["服务端验证签名，无需查存储"] end
Session 优缺点：服务端存储 → 拓展性差（需 Redis 共享）但更安全（可主动失效）。
JWT 优缺点：无状态 → 好拓展，但签出后 token 仍有效（需黑名单），payload 过长影响性能。
银行场景：通常用 Session + Redis 集中存储，可主动踢人、设超时，更安全可控。
