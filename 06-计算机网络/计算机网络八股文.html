为什么 OSI 七层模型没有成为实际标准？

OSI 七层模型由 ISO 制定，设计过于理想化和复杂。表示层（数据加密/压缩/格式转换）和会话层（建立/管理/终止会话）在实际实现中被合并到了应用层。TCP/IP 协议栈先于 OSI 模型出现并大规模部署（ARPANET 项目），形成了事实标准。OSI 模型的真正价值在于它提供了一个清晰的分层参考框架，帮助理解网络协议的职责划分。

数据封装与解封装的实际过程：

发送数据时，每一层在上一层传来的数据前面加上自己的头部（Header）：应用层加上 HTTP Header → 传输层加上 TCP Header（含端口号）→ 网络层加上 IP Header（含 IP 地址）→ 数据链路层加上帧头（含 MAC 地址）和帧尾。接收方反过来逐层剥掉头部，直到还原出原始数据。这个过程就像俄罗斯套娃——每层只关心自己那层的协议字段。

表示层和会话层在 TCP/IP 中去了哪里？

表示层的功能（如 SSL/TLS 加密、数据压缩 Content-Encoding: gzip、字符编码 UTF-8）实际发生在应用层。会话层的功能（如 HTTP 的 Cookie/Session 管理、RPC 的会话保持）也由应用层协议自行处理。这就是为什么 HTTPS 实际上是在应用层和传输层之间插入了一个 TLS 子层——可以看作 OSI 模型的"补丁"。

各层常见协议：
层协议
应用层HTTP, HTTPS, DNS, FTP, SMTP, WebSocket
应用层协议的作用边界：

应用层协议定义的是进程间的通信规则——两个应用程序如何交换和理解数据。HTTP 定义了请求-响应的格式（Request Line + Headers + Body），DNS 定义了域名查询的报文格式（查询/响应的 Header + Question + Answer 段）。它们都依赖下层 TCP/UDP 提供的传输能力，但关注点完全不同：应用层关心"数据是什么意思"，传输层关心"数据怎么可靠到达"。

WebSocket 比较特殊——它通过 HTTP Upgrade 机制从 HTTP 升级为全双工协议，之后不再遵循请求-响应模式，适合聊天、实时推送等场景。

传输层TCP, UDP

传输层是第一个端到端的层——只有通信两端的设备才会处理传输层协议，中间路由器只看 IP 层及以下。端口号是传输层的核心概念，它让一台主机上的多个进程可以同时使用网络。TCP 和 UDP 代表了两种完全不同的设计哲学：TCP 追求"完全可靠"（牺牲速度），UDP 追求"最快送达"（牺牲可靠性）。

网络层IP, ICMP, ARP

网络层的核心是 IP 协议——提供"尽力而为"的数据报传输。它不保证送达、不保证顺序、不保证不重复——这些可靠性工作全部交给传输层的 TCP。ICMP 是 IP 的辅助协议，用于报告网络错误（ping 命令就是 ICMP Echo Request/Reply）。ARP 负责 IP 地址到 MAC 地址的转换（局域网内通信需要 MAC 地址），是网络层和数据链路层之间的桥梁。

数据链路层MAC 地址, ARP

数据链路层在"相邻节点"之间传输帧（Frame），而网络层在"任意主机"之间传输包（Packet）。这一层使用 MAC 地址（48 位，固化在网卡 ROM 中）来标识设备。交换机（Switch）工作在数据链路层，通过 MAC 地址表转发帧。ARP 协议也在这里工作——当知道 IP 但不知道 MAC 时，广播 ARP 请求问"这个 IP 是谁的？"。这一层还负责帧的差错检测（CRC 校验）。

2. TCP vs UDP

TCPUDP
连接面向连接无连接

TCP 在数据传输前必须通过三次握手建立连接，两端维护连接状态（序列号、窗口大小、拥塞控制参数等）。UDP 不需要任何预先设置，应用程序直接把数据报扔到网络上，每个数据报独立路由。连接的开销是 TCP 延迟的来源之一（至少 1.5 个 RTT），但对于长时间的数据流传输，这个开销分摊后可以忽略。

可靠性可靠（确认+重传+排序+流量控制）不可靠

TCP 的可靠性是一组机制的组合：确认应答（ACK）让发送方知道哪些数据已到达；超时重传处理丢包；序列号确保接收方能按正确顺序组装数据；校验和检测传输错误。UDP 只有可选的校验和，没有确认、没有重传、不保序——如果应用需要这些，必须在应用层自己实现（如 QUIC 协议在 UDP 之上实现了类似 TCP 的可靠性）。

速度慢快

TCP 慢的根本原因是其设计目标——可靠传输。三次握手引入延迟，拥塞控制限制发送速率（不会一开始就满速发），流量控制受接收方窗口限制，超时重传会暂停发送。UDP 不做任何速率控制，应用程序可以以任意速度发送数据报（可能淹没接收方或引发网络拥塞）。这就是为什么视频直播和游戏用 UDP——它们宁可丢几个包也不愿等待重传造成卡顿。

头部开销20 字节8 字节

TCP 20 字节头部的构成：源端口(2) + 目的端口(2) + 序列号(4) + 确认号(4) + 数据偏移(0.5) + 保留(0.5) + 标志位(1) + 窗口大小(2) + 校验和(2) + 紧急指针(2)。UDP 8 字节只有：源端口(2) + 目的端口(2) + 长度(2) + 校验和(2)。对于小数据包（如 DNS 查询通常只有几十字节），TCP 头部占比可能超过 50%，非常不划算。

场景HTTP、文件传输、邮件视频直播、DNS、语音通话、游戏

场景选择的本质是延迟敏感性 vs 可靠性需求的权衡。HTTP/文件/邮件需要 100% 的数据完整性，丢一个字节就可能导致文件损坏或页面乱码，所以必须用 TCP。实时音视频的通话中偶尔丢几个包只会导致短暂的画面花屏或声音卡顿，但如果等重传会导致累积延迟越来越严重（TCP 的队头阻塞），用户体验更差。DNS 默认用 UDP 是因为查询-响应消息很短（一般几百字节），丢了直接重试比建立 TCP 连接更快。

广播/多播不支持支持

TCP 是端到端的连接，必须明确通信双方，无法向"所有设备"或"一组设备"广播。这是 TCP 连接模型决定的——需要双方确认序列号、维护各自的窗口。UDP 是无连接的，可以轻松发送广播（到局域网所有主机）或多播（到特定组）。局域网内的服务发现（如 DHCP 分配 IP、mDNS 发现打印机）大量依赖 UDP 广播/多播。

3. TCP 三次握手

▶ 展开详解
为什么两次握手不够——经典场景分析：

假设只有两次握手（客户端 SYN → 服务端 SYN+ACK → 连接建立）。关键问题出在网络中存在过期的 SYN 报文：客户端第一次发送 SYN 超时了，重发一个 SYN 成功建立连接、传输数据、关闭连接。如果此时第一个延迟的 SYN 终于到达服务端，服务端返回 SYN+ACK 后就会建立一个没人使用的死连接，白白占用服务器端口和内存资源。

三次握手解决了这个问题：客户端收到服务端的 SYN+ACK 后，如果是过期连接，客户端知道自己没发起过这个请求，会发送 RST 拒绝。哪怕客户端没有拒绝，至少第三次 ACK 让服务端确认"客户端真的想建立这个连接"。

为什么不是四次握手——效率最优解：

三次握手恰好是确认双方收发能力的最少次数：第一次握手后服务端知道"客户端发能力 OK"；第二次握手后客户端知道"服务端收发都 OK"；第三次握手后服务端知道"客户端收能力 OK"。四次意味着多一轮确认，但此时双方能力都已确认完毕，纯属多余。三次是最少的可靠握手次数。

SYN Flood 攻击原理：

攻击者伪造大量 IP 发送 SYN，服务端回复 SYN+ACK 后进入 SYN_RCVD 状态等待 ACK（这些伪造的 IP 永远不会回复），半连接队列很快被填满，正常请求无法到达。解决方案：SYN Cookie（不在服务端保存半连接状态，而是把信息编码在初始序列号中，第三次握手时解码验证）。

为什么是三次，不是两次？防止已失效的连接请求到达服务端导致错误建立连接。如果是两次握手，服务端收到过期的 SYN 就会建立无用的连接。 为什么不是四次？三次已经可以确认双方收发能力正常（客户端收/发正常，服务端收/发正常），四次没有必要。

4. TCP 四次挥手

▶ 展开详解
为什么挥手要比握手多一次？

TCP 是全双工协议——双方都可以独立地发送和接收数据。握手时双方都想建立双向通信，可以合并为三次。但关闭时，主动关闭方说"我不发了"只关闭了一方的发送通道，被动方可能还有数据要发——所以被动方的 ACK（确认收到 FIN）和 FIN（"我也不发了"）必须分开。这就形成了四次。

半关闭状态（Half-Close）的实际应用：

客户端发送 FIN 后进入 FIN_WAIT_2，此时的 TCP 连接处于半关闭状态：客户端不能发送数据了，但还能接收。服务端可能在 ACK 之后继续发送剩余数据，全部发完后才发送 FIN。HTTP/1.1 的长连接就是利用这个机制——服务端处理完请求后主动关闭。

大量 CLOSE_WAIT 的排查：

如果服务器上出现大量 CLOSE_WAIT 状态的连接，说明应用程序没有调用 close()。服务端收到了客户端的 FIN，内核自动回复了 ACK（进入 CLOSE_WAIT 状态），但应用程序没有处理这个通知（没有调用 close() 来发送 FIN）。通常是因为代码忘了关闭连接，或者是连接被其他逻辑持有（比如线程池没归还连接），导致文件描述符泄漏。

为什么是四次？TCP 全双工，双方都需要发送 FIN + ACK。服务端收到 FIN 后可能还有数据要发，所以 ACK 和 FIN 分开发送。

5. TIME_WAIT 为什么是 2MSL？

MSL = Maximum Segment Lifetime（报文最大存活时间，Linux 默认 30s）。 2MSL 的原因： ① 确保最后的 ACK 能到达：如果服务端没收到最后的 ACK，会重发 FIN，客户端需要在 TIME_WAIT 期间能重发 ACK ② 让旧连接的所有报文在网络中消失：防止旧连接的报文影响新连接 大量 TIME_WAIT 的问题：占用端口和系统资源。解决方案：net.ipv4.tcp_tw_reuse、SO_REUSEADDR。

确认应答与累计 ACK：

TCP 使用累计确认机制——ACK 号表示"该序号之前的所有数据都已收到"。例如 ACK=1001 表示字节 0~1000 全部到达。如果报文段乱序到达（先收到 1001-2000，后收到 1-1000），接收方会连续发送 ACK=1（期待第一个字节）来通知发送方。这就是重复 ACK，触发快重传的信号。

RTO 与 RTT 的关系：

RTO（Retransmission Timeout）必须大于 RTT（Round-Trip Time），否则会虚假重传。TCP 使用 Jacobson 算法动态计算 RTO：SRTT = α*SRTT + (1-α)*RTT_sample（平滑 RTT）+ RTTVAR = β*RTTVAR + (1-β)*|SRTT - RTT_sample|（RTT 偏差），最终 RTO = SRTT + 4*RTTVAR。这样 RTO 能适应网络波动。

滑动窗口的深层含义：

滑动窗口是一个流水线传输机制，解决"停止-等待"协议效率低的问题。发送方可以连续发送窗口内的多个报文而不等待 ACK（提高吞吐量），同时通过窗口大小限制发送速率（防止接收方溢出）。TCP 窗口有三个边界：已发送已确认 / 已发送未确认 / 可发送 / 不可发送，只有"可发送"区域的报文可以立即发出。

7. 流量控制 & 拥塞控制

流量控制（端到端）：接收方通过窗口大小（rwnd）告知发送方自己能接收多少，防止发送方太快把接收方"淹没"。 拥塞控制（网络层面）： 
① 慢启动：cwnd 从 1 开始，指数增长（每 RTT 翻倍）
② 拥塞避免：cwnd ≥ ssthresh 后，线性增长（每 RTT +1）
③ 快重传：连续收到 3 个重复 ACK 立即重传（不等超时）
④ 快恢复：收到 3 个重复 ACK，ssthresh = cwnd/2，cwnd = ssthresh，进入拥塞避免

二、HTTP

1. HTTP 各版本区别

HTTP/1.0HTTP/1.1HTTP/2HTTP/3
连接短连接长连接 (Keep-Alive)多路复用（单连接）基于 QUIC (UDP)

HTTP/1.0 每请求一个资源都要经历"TCP 三次握手 → HTTP 请求-响应 → TCP 四次挥手"的完整流程。一个页面如果有 10 张图片，就要建立 10 个 TCP 连接。HTTP/1.1 通过 Keep-Alive 头（默认开启）让一个 TCP 连接可以复用多次请求-响应，但请求和响应仍然要排队（队头阻塞）。HTTP/2 通过多路复用在一个 TCP 连接上并发发送多个 Stream（每个 Stream 有独立的帧 ID），互不阻塞。HTTP/3 直接抛弃 TCP 用 QUIC（基于 UDP），彻底消除了 TCP 层面的队头阻塞。

队头阻塞❌ 严重❌ 有（Pipeline）✅ 基本解决✅ 彻底解决
队头阻塞的实际影响：

HTTP/1.1 的 Pipeline 机制允许在一个连接上连续发送多个请求（不用等上一个响应），但响应必须按请求顺序返回。如果第一个请求的响应很慢（比如查询大报表），后面的响应即使已经准备好也必须排队等待。浏览器通常禁用 Pipeline，改用为同一域名开 6-8 个并发连接来绕开这个问题。

HTTP/2 在应用层解决了队头阻塞（通过 Stream 帧交错），但在 TCP 层仍有队头阻塞——TCP 是字节流，如果某个 TCP 包丢失，这之后的所有数据（可能包含其他 Stream 的帧）都要等待重传。HTTP/3 用 QUIC 替掉 TCP，每个 Stream 独立传输，丢包只影响该 Stream，其他 Stream 不受影响——这才是彻底的解决方案。

头部压缩无无HPACKQPACK
HPACK 压缩原理：

HTTP/2 的 HPACK 使用两种技术：1) 静态字典：预定义了 61 个常见请求/响应头（如 :method: GET, :status: 200），直接用索引号代替；2) 动态字典：双方维护一个动态表，把实际传输过的 Header 缓存起来，后续用索引引用。HPACK 还使用 Huffman 编码进一步压缩字符串。这样即使需要传输自定义 Header，也只需传输一次完整值，后续都用索引。

QPACK 的改进：HPACK 依赖 TCP 的严格有序传输来维护字典，这导致 HTTP/2 存在队头阻塞（字典更新被阻塞）。QPACK 设计为可以在乱序环境下工作，允许不依赖顺序地更新动态表，适应 QUIC 的数据流模型。

服务器推送无无✅✅

服务器推送（Server Push）让服务端在客户端请求 HTML 时主动推送 CSS/JS 等依赖资源，减少一次往返。但实践中推送效果不佳：浏览器可能已经缓存了这些资源（浪费带宽），且推送的资源可能占用浏览器更高优先级的请求。Chrome 106+ 已经废弃了 HTTP/2 Server Push，推荐使用 <link rel="preload"> 让浏览器自己决定加载优先级。

传输文本文本二进制帧二进制

HTTP/2 引入二进制分帧是重大架构变更。HTTP/1.x 的报文是纯文本，需要逐字节解析（大量字符串操作，容易出解析错误）。HTTP/2 将 HTTP 报文分解为一个个二进制帧（HEADERS 帧、DATA 帧、SETTINGS 帧等），每个帧有固定格式的头部（帧长度 + 帧类型 + 标志位 + Stream ID）。解析器只需读取固定位置的字段，速度快、实现简单，且在同一连接上可以交错发送不同 Stream 的帧——这是多路复用的基础。

2. HTTP vs HTTPS

HTTPHTTPS
端口80443

端口号是传输层的概念，让服务器能区分不同服务。80 和 443 是 IANA 分配的标准端口，浏览器在地址栏不输端口时默认使用这些端口。中间设备（防火墙、负载均衡器）也依赖端口号来做流量识别和转发。使用非标准端口虽然可行但不推荐——很多企业防火墙只放行 80/443。

加密明文传输SSL/TLS 加密
TLS 的混合加密架构：

HTTPS = HTTP over TLS。TLS 握手阶段使用非对称加密（RSA 或 ECDHE）安全交换"会话密钥"，之后的数据传输使用对称加密（如 AES-128-GCM）加密数据。这样设计的原因是：非对称加密计算量是 RSA-2048 比 AES-128 慢约 1000 倍，不适合加密大量数据；但非对称加密的密钥交换安全性高。各取所长。

除了加密，TLS 还提供：身份认证（通过证书链验证服务器身份，防止中间人攻击）和数据完整性（HMAC 或 AEAD 防止数据被篡改）。

证书不需要需要 CA 证书
证书链验证流程：

浏览器收到服务器的证书后，逐级验证：服务器证书 → 中间 CA 证书 → 根 CA 证书（内置在操作系统/浏览器中）。每一步验证签名的有效性、证书是否在有效期内、是否被吊销（通过 CRL 或 OCSP）。如果服务器证书的 SAN（Subject Alternative Name）字段包含访问的域名，验证通过。Let's Encrypt 提供了免费的 DV（域名验证）证书，自动续期，推动了 HTTPS 大规模普及。

连接TCP 三次握手TCP 三次握手 + TLS 握手

TLS 握手增加约 2 个 RTT（TLS 1.2）或 1 个 RTT（TLS 1.3）的延迟。TLS 1.3 通过减少握手轮次（移除了 Change Cipher Spec，合并了多个步骤）、只支持前向安全的密钥交换算法（ECDHE、DHE，不再支持 RSA 密钥交换）来提速。还有 TLS False Start（在收到 Finished 之前就开始发送应用数据）和 0-RTT（TLS 1.3 重连时可以在第一个包就携带数据）进一步降低延迟。

SEO无优势搜索排名更靠前

Google 从 2014 年起将 HTTPS 作为搜索排名信号（Ranking Signal），HTTPS 页面在同等条件下排名更靠前。Chrome 浏览器也会对 HTTP 页面显示"不安全"标记，影响用户信任。银行/金融类网站必须使用 HTTPS 才能通过安全合规审计。

3. HTTPS 握手过程（TLS 1.2）

sequenceDiagram participant C as 客户端 participant S as 服务端 Note over C,S: ① Client Hello C->>S: TLS 版本 + 随机数1 + 密码套件列表 Note over C,S: ② Server Hello S->>C: 选中的 TLS 版本 + 随机数2 + 选中的密码套件 S->>C: ③ Certificate: 服务器证书（含公钥） S->>C: ④ Server Hello Done Note over C: ⑤ 验证证书（CA 签名），生成随机数3 C->>S: ⑥ Client Key Exchange：用服务器公钥加密随机数3 C->>S: ⑦ Change Cipher Spec：后续通信加密 C->>S: ⑧ Finished（加密的） Note over S: 用私钥解密得到随机数3 Note over C,S: 双方用随机数1+2+3 生成对称密钥 S->>C: ⑨ Change Cipher Spec + Finished Note over C,S: ✅ 握手完成，对称加密通信
▶ 展开详解
三个随机数的作用与安全性：

最终会话密钥由"随机数1 + 随机数2 + 随机数3"共同生成（通过 PRF 伪随机函数）。只用客户端随机数不够——如果客户端随机数生成器很弱（如嵌入式设备），密钥可被预测；只用服务端随机数同理。双方随机数结合确保即使一方弱也猜不出密钥。

随机数3（Pre-Master Secret）是整个握手的安全核心——它用服务端公钥加密传输，只有拥有私钥的服务器能解密。任何人截获了 Client Hello 和 Server Hello（含随机数1、2），如果没有随机数3，无法推导出会话密钥。

TLS 1.2 vs TLS 1.3 的关键差异：
  • TLS 1.2 握手 = 2 RTT，TLS 1.3 = 1 RTT（Client Hello + 密钥交换参数一起发）
  • TLS 1.3 移除了 RSA 密钥交换（不支持前向安全性），只保留 (EC)DHE（支持前向安全）
  • TLS 1.3 移除了 CBC 模式、RC4、SHA-1 等不安全算法
  • TLS 1.3 支持 0-RTT 模式：重连时可以附带早期数据，减少延迟
前向安全性（Forward Secrecy）详解：

使用 RSA 密钥交换时，如果服务器私钥在未来泄露，攻击者可以解密所有历史会话（因为 Pre-Master Secret 是用公钥加密的，私钥能解出来）。ECDHE 每会话生成临时密钥对，会话结束后删除——即使服务器私钥泄露，历史会话也无法解密。这就是前向安全性：今天的密钥泄露不影响昨天的通信安全。

混合加密：非对称加密（RSA/ECDHE）交换密钥 + 对称加密（AES）传输数据。原因是非对称加密计算量大，不适合大量数据。

4. GET vs POST

GETPOST
语义获取资源创建/提交数据

REST 架构中，GET 对应"读取"操作，是安全（Safe）且幂等（Idempotent）的方法。安全意味着不修改服务器状态（你只是在读），幂等意味着多次执行效果相同。POST 用于创建新资源，即不幂等也不安全。虽然 HTTP 规范区分了这两者，但技术上 GET 完全可以写入数据库——这属于 API 设计规范，而非协议强制的。

参数位置URL 查询串请求体

参数放在 URL 还是 Body 不仅是位置差异，还影响缓存策略和请求语义。URL 可见让 GET 请求可以被书签保存、链接分享、浏览器历史记录。而 POST 的请求体（Body）对用户不可见，适用于敏感数据或大量数据。从 HTTP 协议角度，GET 也可以有 Body（虽然不推荐且可能被代理丢弃），POST 的 URL 也可以带参数。

安全性参数暴露在 URL/浏览器历史/日志相对安全（不在 URL）

安全性在这里指的是数据泄露风险而非是否被加密。HTTPS 加密了整个 HTTP 报文（包括 URL 路径和参数），传输过程中两者都安全。但 GET 的参数会出现在浏览器地址栏、历史记录、Referer 头、服务器访问日志、代理服务器日志中——这些地方是"传输过程之外"的数据泄露渠道。POST 的参数在请求体中，这些地方不会记录。

幂等性✅ 幂等❌ 不幂等
幂等性在实际系统中的作用：

幂等意味着"执行 1 次和执行 100 次的结果一样"。这对分布式系统非常重要——网络超时重试时，GET 可以安全重试（反正就是再读一次），但 POST 可能导致重复创建资源。实际业务中，银行转账接口虽然用 POST，但通过"幂等键"（Idempotency-Key 头或业务流水号）来实现幂等——相同流水号的请求只处理一次。

其他方法：PUT（更新/替换，幂等）、DELETE（删除，幂等——删第 100 次和第 1 次结果相同：资源没了）、PATCH（部分更新，不幂等）。

缓存可缓存不可缓存（通常）

HTTP 缓存机制（Cache-Control、ETag、Last-Modified）天然为 GET 设计——读取相同数据不应该每次都要服务器计算。GET 请求可以被浏览器缓存、CDN 缓存、代理缓存多层缓存。POST 请求不对浏览器缓存开放，但响应仍可被 CDN 缓存（如果 CDN 支持）。缓存能极大减少服务器负载和响应时间。

长度限制URL 有长度限制（浏览器/服务器）无长度限制

HTTP 协议本身没有定义 URL 长度上限，但各浏览器和服务器有实现限制：IE 限制约 2048 字符，Chrome 限制约 2MB（实际中很少遇到），Nginx 默认限制 8KB，Apache 默认限制 8KB，Tomcat 默认限制 8KB。POST 的请求体大小受服务器配置限制（如 Nginx 的 client_max_body_size 默认 1MB），可以通过配置调大。如果 GET 参数太长可能被截断，导致请求出错，所以大数据量只能用 POST。

5. HTTP 状态码

类别常见状态码
1xx 信息100 Continue

100 Continue 的典型流程：客户端发送带 Expect: 100-continue 请求头的 POST 请求，先只发请求头，服务器确认可以接收（返回 100）后，客户端再发送请求体。这样可以避免上传大文件后才发现被拒绝（如认证失败、权限不足）。其他 1xx 状态码：101 Switching Protocols（WebSocket 升级）、103 Early Hints（服务器推送资源提示）。

2xx 成功200 OK, 201 Created, 204 No Content

200 是最常见的成功响应，表示请求已被正确处理并返回了响应体。201 Created 在 REST API 中用于 POST 创建资源成功后返回，响应头通常带 Location 字段指向新资源的 URL。204 No Content 表示请求成功但没有返回体，常用于 DELETE 操作或 AJAX 请求只需要知道成功/失败不需要额外数据的场景。

3xx 重定向301 永久（搜索引擎换URL）, 302 临时, 304 Not Modified（缓存）
重定向状态码的精确区分：
  • 301 Moved Permanently：资源已永久迁移到新 URL。浏览器会缓存这个重定向，下次直接请求新地址。搜索引擎会将权重和链接转给新 URL。
  • 302 Found（临时重定向）：资源临时在新地址，浏览器不缓存，下次仍然请求原地址。但很多浏览器对 302 的实际行为有差异——有的会改变 POST 为 GET。
  • 307 Temporary Redirect / 308 Permanent Redirect：与 302/301 类似，但严格保持请求方法不变。POST 请求重定向后仍然是 POST。
  • 304 Not Modified：不发请求体，告诉浏览器"你缓存的版本还是最新的，直接用缓存"。配合 ETag 和 Last-Modified 使用。
4xx 客户端错误400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 405 Method Not Allowed
常见 4xx 的精确区分：
  • 400：请求格式错误（JSON 解析失败、参数类型不对），服务端无法理解请求。
  • 401："你需要先登录"。表示认证信息缺失或无效，通常会带 WWW-Authenticate 头引导客户端提供凭据。
  • 403："你已登录，但没有权限"。跟 401 容易被混淆——关键区别是 401 针对未认证用户，403 针对已认证但权限不足的用户。比如普通用户访问管理员页面应返回 403 而非 401。
  • 404：资源不存在。不仅是路由不匹配，也可能是资源被删除。
  • 405：请求方法不允许（如用 GET 访问只接受 POST 的接口）。
  • 429 Too Many Requests：触发限流，响应头通常包含 Retry-After 字段。
5xx 服务端错误500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout
5xx 背后对应的实际架构层次：
  • 500：应用层未捕获的异常（空指针、除零、数据库连接失败）。"Internal"意味着服务器自己也不知道具体是什么问题——好的 API 设计应该避免返回 500，全局异常处理应返回有意义的错误码。
  • 502：网关/反向代理（Nginx）从上游服务器收到无效响应。常见原因：应用服务器挂了（进程崩溃）、应用服务器返回了格式错误的响应、连接被重置。
  • 503：服务器暂时无法处理请求（过载、维护中）。应有 Retry-After 头告诉客户端多久后重试。Nginx 配置了限流时也会返回 503。
  • 504：网关等待上游服务器响应超时——说明请求被处理了但太慢。常见于：数据库查询超时、外部 API 调用超时、长计算。可能需要在 Nginx 增大 proxy_read_timeout，或者优化慢查询。
301 vs 302：301 永久重定向（浏览器缓存，下次直接走新地址）；302 临时重定向（每次都要问原地址）。

6. Cookie / Session / Token (JWT)

{header}.{payload}.{signature}"] G --> H["客户端存 localStorage/Cookie"] H --> I["每次请求手动带 Authorization: Bearer token"] I --> J["服务端验证签名，无需查存储"] end
▶ 展开详解
Session 模式的安全攻防：

Session 依赖于 Cookie 传输，Cookie 的关键安全属性：HttpOnly（禁止 JS 访问，防 XSS 窃取）、Secure（仅 HTTPS 传输，防中间人截获）、SameSite=Strict/Lax（防 CSRF 跨站请求伪造，禁止跨站请求带 Cookie）。SameSite=Lax 是较均衡的选择——用户从其他网站点击链接跳转时会带 Cookie（正常导航），但 img/form 等跨站请求不带（防 CSRF）。

Session 的扩展问题：单机用内存存储，分布式必须用 Redis 集中存储。所有请求都查 Redis 意味着 Redis 成为性能瓶颈和单点故障风险。Session 同步也是问题——修改 Session 后要更新 Redis。

JWT 的安全风险详解：
  • 无法主动失效：JWT 签发后在过期时间前一直有效。用户改密码后旧 token 仍可用——需要维护黑名单（JWT ID + Redis 存储失效 token 列表），这又引入了存储依赖。
  • payload 可见性：JWT 的 payload 只是 Base64 编码（不是加密！），任何拿到 token 的人都能解码看到里面的数据。绝不能在 JWT 中存放密码、身份证号等敏感信息。
  • 签名算法攻击：如果服务端配置错误接受了 alg: none 的 JWT（表示无签名），攻击者可以伪造任意 payload。必须明确指定和验证签名算法。
  • XSS 窃取：存 localStorage 的 JWT 易被 XSS 攻击读取（JS 能访问 localStorage，但读不到 HttpOnly Cookie）。
Session 优缺点：服务端存储 → 拓展性差（需 Redis 共享）但更安全（可主动失效）。
JWT 优缺点：无状态 → 好拓展，但签出后 token 仍有效（需黑名单），payload 过长影响性能。
银行场景：通常用 Session + Redis 集中存储，可主动踢人、设超时，更安全可控。

浏览器渲染的细化步骤（从 HTML 到像素）：

1. 构建 DOM 树：HTML 解析器逐字节解析 HTML → 生成 DOM 节点 → 构建 DOM 树。
2. 构建 CSSOM 树：遇到 CSS 链接或 style 标签 → 解析 CSS → 构建 CSSOM 树（CSS Object Model）。CSS 会阻塞渲染，但不会阻塞 DOM 构建。
3. 构建 Render 树：将 DOM 树和 CSSOM 树合并，只包含可见元素（display:none 不在其中）。
4. Layout（回流/重排）：计算每个节点在屏幕上的精确位置和大小。
5. Paint（绘制）：将计算好的布局像素化——把各节点绘制到不同的图层（如 Canvas、Video 在独立图层）。
6. Composite（合成）：GPU 将各图层合成为最终屏幕图像。

JavaScript 对解析阻塞的影响：

当 HTML 解析器遇到 <script> 标签时，会暂停 HTML 解析，下载并执行 JS（因为 JS 可能通过 document.write 修改 DOM）。这就是为什么推荐 <script async>（异步下载，下载完立即执行，不阻塞 HTML 解析但可能阻塞 DOMContentLoaded）或 <script defer>（异步下载，HTML 解析完再按顺序执行）。CSS 也会间接阻塞——JS 执行前需要等待所有 CSS 加载完（因为 JS 可能读取样式）。

Nginx → 应用 → DB 这一层的细化：

请求到达服务器后：Nginx 做反向代理和负载均衡，根据配置的 upstream 策略（轮询/加权/最少连接/IP hash）将请求转发给应用服务器实例 → Spring MVC 的 DispatcherServlet 接收 → 路由到对应 Controller → 调用 Service 层 → 查 MySQL（可能先查 Redis 缓存）→ 返回数据。

▶ 展开详解
递归查询 vs 迭代查询：

图中本地 DNS 服务器使用的是迭代查询——每一级 DNS 服务器告诉本地 DNS "我不知道答案，但你可以去问 XXX"，本地 DNS 像侦探一样层层追问。另一种是递归查询——客户端（浏览器）要求本地 DNS 服务器"你必须给我最终答案"，本地 DNS 自己去完成所有查询工作。实际网络中，客户端到本地 DNS 是递归查询，本地 DNS 到各级 DNS 是迭代查询。

根域名服务器的 Anycast 机制：

全球只有 13 组根域名服务器（a.root-servers.net 到 m.root-servers.net），但通过 Anycast（任播）技术在全球部署了超过 1500 个镜像节点。Anycast 让多个服务器共享同一个 IP，BGP 路由协议会自动将请求路由到最近的节点——这就是为什么 DNS 查询虽然要访问美国管理的根服务器，但查询速度并不慢。

DNS 的安全问题与改进：
  • DNS 劫持：中间人篡改 DNS 响应，将用户导向钓鱼网站。传统 DNS 基于 UDP 无加密无验证。
  • DNS over HTTPS (DoH)：使用 HTTPS 加密 DNS 查询，防止运营商/ISP 监控或篡改 DNS。Chrome/Firefox 已支持。
  • DNSSEC：为 DNS 响应添加数字签名（不加密），让客户端验证 DNS 记录是否被篡改。部署率仍然很低。
  • DNS 缓存污染：攻击者伪造 DNS 响应抢在真实响应之前到达本地 DNS 服务器，让缓存中存入错误的 IP。DNSSEC 可以防止（签名无法伪造）。
TTL (Time To Live) 的实际影响：

DNS 记录的 TTL 决定各级缓存的保留时间。TTL 太短 → DNS 查询频繁，延迟增大；TTL 太长 → IP 变更后用户长时间访问旧地址。运维中做域名迁移时通常提前把 TTL 降到 60 秒，迁移完成后改回 3600 秒。CDN 场景中 TTL 通常设置较短（1-5 分钟），因为节点 IP 可能频繁变化。
