课题一：办公领域智能中枢系统

1.1 课题概述

构建基于大语言模型（LLM）和智能体（Agent）技术的办公智能中枢。通过意图识别、任务拆分与编排、长期记忆、子智能体等能力，解决传统办公系统"任务理解低效、多任务协同困难、知识复用不足"三大痛点。对接OA、邮件等企业系统，实现跨平台自动化协同，最终形成可复用的办公知识库。

技术点详细说明面试可能追问
意图识别 LLM对用户自然语言输入进行语义理解，识别用户真实意图，区分"会议安排"、"文档查询"、"邮件起草"等不同场景 多意图冲突如何处理？
多意图识别与冲突处理：

当用户输入包含多个意图时（如"帮我查一下上周的会议纪要，然后发邮件给参会人"），需要：

1. 先用LLM做意图分类（classification），识别出"查纪要"和"发邮件"两个意图

2. 判断意图之间的依赖关系（DAG编排）——先查纪要，再发邮件

3. 如果两个意图互斥（如"取消会议"和"修改会议时间"同时出现），需要发起澄清式追问

4. 实现方式：在Prompt中要求LLM输出结构化JSON，包含intents数组和dependency图

任务拆分与编排 将复杂任务分解为原子性子任务，通过DAG（有向无环图）或状态机进行编排调度 如何保证子任务间的数据传递？
任务拆分与编排机制：

1. 使用ReAct（Reasoning + Acting）模式：LLM先推理，再调用工具执行

2. 任务拆分通过Few-shot Prompt让LLM输出子任务列表和依赖关系

3. 编排引擎实现：每个子任务有明确的输入/输出schema，通过上下文管理器传递数据

4. 失败重试：子任务失败时，编排器记录中间态，支持断点续传

5. 类似LangChain的Chain/Agent机制，或直接使用OpenAI的Function Calling + 多轮对话

长期记忆 基于向量数据库存储用户历史交互，实现"记住用户偏好、历史上下文"的能力 向量数据库选型？如何处理记忆衰减？
长期记忆实现方案：

1. 短期记忆：对话上下文窗口内的历史消息（受token限制）

2. 长期记忆：将历史对话embedding后存入向量数据库（如Milvus/Chroma/Qdrant），检索时按相似度召回最相关的历史交互

3. 记忆衰减策略：按时间加权（越近权重越高）、按重要性加权（用户明确标记的重要信息高分）

4. 记忆压缩：定期用LLM对历史对话做总结摘要，替代原始对话存储

子智能体调度 不同业务场景由专门的子Agent处理，主Agent负责任务路由和结果汇总 多Agent之间如何通信？
多Agent协作模式：

1. 主从模式：主Agent负责任务分发，子Agent专注特定领域（如会议Agent、邮件Agent）

2. 通信方式：通过标准化JSON消息传递，或共享上下文（SharedContext）

3. 与OA/邮件系统的对接：通过API接口调用企业系统，Agent生成调用参数

4. 结果汇总：主Agent收集各子Agent输出，做最终的自然语言整合

多源异构数据处理：

1. 数据来源：手机银行埋点日志（点击流）、网银交易记录（结构化）、柜面服务记录（半结构化）、客服对话（非结构化文本）

3. 行为序列建模：将客户行为按时间排序，形成行为链路（如：浏览理财产品 → 查看详情 → 对比 → 咨询客服 → 购买/放弃）

4. 时间窗口特征：滑动窗口（近7天/30天/90天）统计行为频次、时长、转化率等特征

Prompt Engineering 提示词工程 设计结构化的提示模板，引导LLM对客户行为进行分析和解读 如何保证Prompt输出的稳定性和一致性？
提示词工程设计思路：

1. 角色设定：你是银行客户行为分析专家

2. 结构化输入：将客户行为数据格式化为JSON输入LLM

3. 约束输出格式：要求输出结构化JSON（标签列表+置信度+推荐产品+理由）

4. Few-shot示例：在Prompt中提供2-3个标准示例，引导模型输出稳定

5. Chain-of-Thought：要求模型先推理再输出结论（"请先分析客户行为特征，再给出建议"）

6. 温度参数控制：分析类任务temperature设低（0.1-0.3），创意类可设高

模型微调 Fine-tuning 在银行领域数据上对基座模型进行微调（LoRA/QLoRA），提升模型对金融场景的理解能力 全量微调 vs LoRA的区别？何时选哪种？
模型微调方案对比：
方案参数量显存需求训练速度适用场景
全量微调全部参数极高（7B需~56GB）慢数据量大、追求极致效果
LoRA~0.1-1%低（7B需~16GB）快资源受限、多任务微调
QLoRA~0.1-1%极低（7B需~10GB）较快消费级GPU、快速实验

本课题使用QLoRA即可：将4-bit量化模型 + LoRA低秩适配，在消费级GPU上即可微调7B模型。

RAG知识检索增强 结合银行产品库、营销策略库，检索相关信息增强LLM的生成质量 RAG中的chunking策略如何选？
RAG在客户行为分析中的应用：

1. 知识库内容：银行产品信息（理财/基金/保险）、营销话术模板、合规要求

2. 检索时机：当需要推荐具体产品时，检索产品库；当需要给出营销建议时，检索话术库

3. Chunk策略：产品信息按产品维度切分（每个产品一个chunk），合规要求按条款切分

4. 重排序（Re-rank）：初检Top-K结果后用Cross-encoder做精排，提升相关性

5. 引用溯源：生成结果中标注信息来源，增强可解释性

端侧大模型部署 将大模型部署在自助终端设备本地（而非云端），实现低延迟、离线可用、数据不出设备 端侧部署的挑战和解决方案？
端侧部署核心技术：

1. 模型量化：使用INT4/INT8量化压缩模型（如GPTQ、AWQ、GGUF格式），将7B模型压缩到4GB以内

2. 小模型选型：Qwen-1.8B/4B、Phi-3-mini等轻量模型，专门针对端侧场景设计

3. 推理框架：使用llama.cpp、Ollama、vLLM等支持CPU/GPU混合推理的框架

4. 为什么端侧部署：银行自助终端涉及客户敏感数据，不能上传云端；离线环境；实时性要求（毫秒级响应）

5. 挑战：模型精度损失（量化带来）、硬件资源有限（嵌入式设备）、模型更新困难

多模态数据分析 同时分析"屏幕视觉信号"和"系统日志文本"，两种模态互补，提升异常检测准确性 视觉和日志两种信号如何融合？
多模态融合策略：

1. 视觉模态：截取终端当前屏幕截图，用视觉模型（或简化版用图像哈希对比）检测异常画面——白屏、卡顿界面、异常弹窗、功能按钮缺失等

2. 日志模态：解析系统日志，用NLP模型识别异常错误码、异常调用栈、超时记录等

3. 融合方式：

- 早期融合：将视觉特征+日志特征拼接后输入统一模型

- 晚期融合：视觉和日志各自独立判断，最后投票/加权综合

- 本课题更适用晚期融合：各自独立检测，交叉验证（如日志报错+画面卡顿→高置信度故障）

健康状态评估体系 建立多维度的设备健康评分模型，综合评估设备运行状态 健康评分的指标体系怎么设计？
设备健康评分设计（参考Google SRE黄金指标）：
指标维度具体指标权重数据来源
可用性服务正常响应率、交易成功率30%系统日志
性能界面响应延迟、交易处理时长25%日志+视觉（帧间时差）
错误率异常错误码频次、崩溃次数25%系统日志
饱和度CPU/内存/磁盘使用率20%系统监控

最终健康分 = Σ(维度得分 × 权重)，分数映射为：健康(80-100)/亚健康(60-79)/故障(0-59)

自动故障分析报告 基于历史故障数据和实时采集信息，自动生成故障分析报告 如何保证故障分析的准确性？
自动故障报告生成流程：

1. 异常检测 → 触发故障分析流程

2. 采集上下文数据：当前截图、近5分钟日志、系统资源快照

3. 向量检索历史相似故障案例（RAG）

4. LLM综合当前数据+历史案例，生成结构化故障报告：

- 故障时间、故障现象描述

- 可能根因（排序、置信度）

- 建议处理方案

- 相似历史案例参考

5. 报告推送给运维人员，人工确认后回收入故障知识库

AI智能解析 使用Qwen模型理解录制操作的业务语义，生成更可读、可维护的测试代码 AI如何让生成的代码更"可维护"？
AI增强测试代码生成：

Playwright原生录制生成的是底层操作序列（click、fill等），代码可读性差。AI的作用：

1. 语义理解：将"点击第3个按钮"理解为"点击'提交申请'按钮"，生成语义化的选择器

2. 断言自动生成：从操作上下文推断预期结果（如"提交后应跳转到成功页面"→自动加expect断言）

3. 代码重构：提取公共操作为函数/Page Object，遵循DRY原则

4. 数据驱动改造：识别硬编码的测试数据，改造为参数化

5. 异常处理补充：自动添加try-catch和重试逻辑

零代码测试方案 非技术人员通过录制即可生成测试用例，无需编写代码 零代码方案的局限性？
零代码的边界与局限：

适用场景：

- 回归测试（已有功能反复验证）

- 冒烟测试（核心链路快速验证）

- UI交互验证（页面跳转、表单提交）

不适用场景：

- 复杂逻辑验证（需要精确断言和数据处理）

- 接口级测试（适合直接用Postman/JMeter）

- 性能测试（大量并发需要专业工具）

- 动态数据的断言（录制时的数据可能已过期）

解决方案：零代码覆盖70%-80%的常规测试，复杂场景仍保留编码能力作为补充。

可视化回放与用例管理 GIF动图全程回放测试执行步骤，集中管理自动化用例 如何保证测试用例的可维护性？
用例管理与可维护性：

1. 版本管理：测试用例文件纳入Git管理，与业务代码同步版本

2. 分层组织：按模块/功能/场景三级目录组织用例

3. 标签分类：给用例打标签（冒烟/回归/全量/P0/P1），支持按标签筛选执行

4. 失败自动分类：AI分析失败原因——产品Bug / 环境问题 / 用例过期

5. 用例自愈：当页面元素变化导致用例失败时，AI尝试自动修复选择器

6. GIF回放价值：快速定位失败步骤，无需重现环境

问题类型可能的问题回答要点
技术选型 为什么选择Qwen而不是GPT？为什么用RAG而不是微调？为什么用Playwright而不是Selenium？ 需要有清晰的权衡分析
回答框架——每个技术选型都从四个维度分析：

1. 成本：Qwen开源/免费 vs GPT API收费；RAG无需训练成本 vs 微调需要GPU资源

2. 数据安全：银行数据不能出内网，必须私有化部署 → 开源模型优势

3. 性能：场景需求的具体指标（延迟、准确率、吞吐量）

4. 可维护性：团队技术栈、社区生态、文档完善度

架构设计 如果要你设计这个系统，你会怎么做？ 按数据层→模型层→应用层的三层架构回答
回答模板（以课题2客户行为分析为例）：

1. 数据层：从手机银行/网银/柜面等渠道采集行为数据，入Hive/数据湖，Spark做特征工程

2. 模型层：Qwen基座模型 + LoRA微调 + RAG检索增强，部署在GPU服务器上提供API服务

3. 应用层：Spring Boot构建后端服务，前端展示客户标签/行为路径/营销建议

4. 监控层：模型效果监控（标签准确率）、数据漂移检测、badcase收集

风险与挑战 这个项目最大的挑战是什么？你打算怎么解决？ 从技术、业务、合规三个角度分析
常见挑战及应对：

1. 数据质量问题：银行历史数据可能存在缺失/不一致 → 数据清洗Pipeline + 人工标注补充

2. 模型幻觉：LLM可能产生不存在的产品信息 → RAG约束 + 输出校验 + 人工审核

3. 合规风险：客户数据隐私保护 → 数据脱敏 + 端侧部署 + 最小权限原则

4. 效果评估：缺乏标注数据 → 先小规模试点，收集反馈后再迭代
