Linux 常用命令八股文

一、系统监控与资源查看

1. 系统整体状态

top / htop — 实时进程监控

关键指标解读：

  • load average：1min / 5min / 15min 平均负载。它的值等于处于可运行状态（R） + 不可中断睡眠状态（D）的线程数平均值。load > CPU 核心数 = 系统过载信号（可能 CPU 瓶颈或 IO 瓶颈）。单核 load=2 意味着平均有 1 个任务在排队等待 CPU。
  • CPU 行：us（用户态）、sy（内核态）、id（空闲）、wa（等待 IO）—— wa 持续 >10% 说明磁盘可能成为瓶颈。
  • MEM 行：total / free / used / buff/cache。Linux 会利用空闲内存做文件缓存（buff/cache），所以"free"值很小不一定说明内存不够。
  • htop 优势：彩色显示、鼠标操作、可直接对进程发送信号（F9 kill），支持树状进程视图。
free -h — 内存使用概览 
$ free -h
              total        used        free      shared  buff/cache   available
Mem:           15Gi       3.2Gi       2.1Gi       256Mi        10Gi        12Gi
Swap:         2.0Gi       0.0Gi       2.0Gi

关键理解：free 列是"完全未被使用的内存"，available 列才是"应用程序可用内存"（包括可回收的 buff/cache）。面试常问：free 很小但 available 很大——内存充裕，只是被缓存占用了；free 和 available 都很小——真正内存不足。

buff vs cache 区别：buff（buffer）是块设备（磁盘）的元数据缓存；cache 是文件内容的页面缓存（page cache）。应用程序读文件时会先查 page cache，命中则避免磁盘 IO。

df -h / du -sh — 磁盘使用 
$ df -h          # 查看文件系统级别磁盘使用
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        50G   42G  5.1G  90% /
$ du -sh /var/log/*   # 查看目录级别磁盘使用
2.3G    /var/log/app
512M    /var/log/nginx

经典面试题：df 显示磁盘满了（100%），但 du 统计各目录加起来远小于总使用量？

答案：已删除的文件仍然被某个进程打开着（持有文件描述符）。文件被 unlink 后目录条目消失（du 看不到），但进程没关闭 fd 的话，内核不会释放数据块（df 仍然算占用）。用 lsof | grep deleted 找到这些文件，重启或 kill 对应进程即可释放。
vmstat 1 — 虚拟内存统计（每秒输出一次） 
$ vmstat 1
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 3  0      0  2.1G   512M   9.8G    0    0    12    45  120  280  5  2 93  0  0
 2  1      0  2.1G   512M   9.8G    0    0     0   120  150  320  8  3 88  1  0

关键列解读：

  • r（runnable）：运行队列中的进程数。如果 r > CPU 核心数，说明 CPU 是瓶颈。
  • b（blocked）：处于不可中断睡眠（等待 IO）的进程数。b > 0 持续说明 IO 瓶颈。
  • si / so（swap in / out）：如果 si/so > 0，系统正在换页——内存压力大的标志。频繁 swap 会导致性能急剧下降。
  • us / sy / id / wa：用户态 / 内核态 / 空闲 / IO等待 的 CPU 时间百分比。
  • cs（context switch）：上下文切换次数，过高可能线程数太多或锁竞争激烈。
iostat -x 1 — 磁盘 IO 统计 
$ iostat -x 1
Device   r/s   w/s   rkB/s   wkB/s   await  r_await  w_await  %util
sda     15.0  80.0   600.0  3200.0   12.50     5.00     14.00  85.0

关键列解读：

  • await：IO 请求的平均等待时间（毫秒）。机械盘正常 <20ms，SSD 正常 <5ms。高 await 是磁盘瓶颈的直接信号。
  • %util：设备带宽利用率。接近 100% 表示磁盘已饱和。注意：SSD 能并行处理请求，%util 100% 不一定意味着性能极限，而机械盘 100% 就是真的瓶颈。
  • r_await / w_await：分别看读/写延迟，帮助定位是读多还是写多的场景。

2. 进程管理

命令用途常用组合 / 关键参数
ps aux列出所有进程详情（BSD 风格）ps aux | grep java

ps aux 各列含义：USER（进程所有者）、PID（进程ID）、%CPU、%MEM、VSZ（虚拟内存 KB）、RSS（物理内存 KB）、TTY（终端）、STAT（状态：R运行 S睡眠 D不可中断睡眠 Z僵尸 T停止）、START（启动时间）、TIME（累计CPU时间）、COMMAND（命令）。

ps -ef vs ps aux：ps -ef 是 Unix 风格，输出 PPID（父进程ID）和 STIME（启动时间），适合查看父子关系。ps aux 是 BSD 风格，输出 CPU 和内存百分比，适合性能分析。很多面试官会问"如何查看进程的父进程"——用 ps -ef | grep PID 看 PPID 列。

lsof -p PID列出进程打开的所有文件（包括 socket、pipe）lsof -p 1234 | wc -l

"too many open files" 错误的排查：lsof -p PID 可以查看一个进程打开了多少个文件描述符。Linux 默认每个进程打开文件数限制（ulimit -n）通常是 1024，对高并发服务（如 Nginx、Kafka）明显不够。生产环境通常设置为 65535 或更高。修改方式：ulimit -n 65535（临时），/etc/security/limits.conf（永久）。

lsof 其他用途：lsof -i :8080（查看端口占用）、lsof /var/log/app.log（查看哪些进程在写这个日志）、lsof | grep deleted（找到被删除但仍占磁盘的文件）。

netstat / ss网络连接统计ss -tlnp | grep :8080

ss 为什么比 netstat 快？netstat 读取 /proc/net/tcp（文本文件），ss 通过 netlink socket 直接向内核查询数据（二进制协议），尤其在大量连接时差异显著（几千个连接时 netstat 可能需要几秒，ss 毫秒级返回）。

TCP 状态统计（面试常考）：netstat -an | awk '/^tcp/ {print $6}' | sort | uniq -c | sort -rn —— 输出所有 TCP 状态的连接数分布。TIME_WAIT 连接过多（数千个）说明短连接频繁创建和关闭，可以调整 net.ipv4.tcp_tw_reuse 或改用长连接。

kill向进程发送信号kill -9 PID（强制）/ kill -15 PID（优雅）

kill 命令不是"杀死进程"，而是"发送信号"——默认信号是 SIGTERM（15）。常见信号：1(SIGHUP 重新加载配置)、3(SIGQUIT Java 线程转储)、9(SIGKILL 内核强制终止，不可捕获)、15(SIGTERM 通知进程自行退出，可捕获做善后清理)。

SIGKILL vs SIGTERM 的关键区别：SIGKILL 不经过进程，由内核直接终止——进程没机会清理资源、刷新缓冲区、关闭连接。优雅停止应该先用 SIGTERM，等待 N 秒后如果进程还没退出再用 SIGKILL。很多中间件（Nginx、Kafka）的 stop 脚本都是这个套路。

信号类型速查表
编号信号名含义能否捕获
1SIGHUP终端挂起 / 重新加载配置✅
2SIGINTCtrl+C 中断✅
3SIGQUITCtrl+\ 退出 + core dump（Java 线程转储）✅
9SIGKILL强制终止（内核直接 kill）❌
15SIGTERM终止（默认信号，通知进程自行退出）✅
17SIGCHLD子进程状态变化（结束/暂停）✅

二、日志排查与文本处理

3. 日志查看与搜索

tail -f / -F — 实时跟踪日志 
$ tail -f app.log       # 跟随文件增长，文件被重命名/删除后停止
$ tail -F app.log       # 跟随文件名，文件轮转后自动重新打开（更健壮）
$ tail -n 200 app.log   # 显示最后 200 行
面试重点：-f 和 -F 的区别。日志轮转（logrotate）时，app.log 被重命名为 app.log.1，然后创建新的 app.log。tail -f 仍然握着旧文件的 inode，不会再输出新日志；tail -F 会检测文件被替换，自动关闭旧文件并打开新文件。生产环境看日志一定用 -F。
grep — 文本搜索之王
参数作用示例
-i忽略大小写grep -i "error" app.log

在日志排查中非常实用——Java 异常 "Exception" vs "exception" vs "EXCEPTION" 大小写可能不一致。grep -i 统一处理。

-v反向匹配（排除）grep -v "DEBUG" app.log

反向过滤常用于排除噪音——比如看 ERROR 和 WARN 但不看 DEBUG。也可以多次管道：grep "ERROR\|WARN" app.log | grep -v "heartbeat"（排除心跳日志的干扰）。

-c计数匹配行数grep -c "ERROR" app.log

快速统计错误数量：grep -c "ERROR" app.log 输出数字。也可以结合 -v 统计非匹配行：grep -vc "DEBUG" app.log。

-n显示行号grep -n "NullPointer" app.log

显示行号后用 vim/less 可以快速跳转到对应行，如 vim +行号 app.log。结合 -H 在多文件搜索时显示文件名。

-A / -B / -C显示上下文grep -B 2 -A 10 "Exception" app.log

-A N：匹配行之后 N 行（After）。-B N：匹配行之前 N 行（Before）。-C N：前后各 N 行（Context）。排查异常时最常用的组合：grep -B 2 -A 10 "Exception" app.log —— 看异常发生前 2 行（请求参数）和异常后 10 行（完整堆栈）。

-E扩展正则（支持 | + ? 等）grep -E "ERROR|WARN|FATAL" app.log

不用 -E 的话管道符 | 会被当作普通字符匹配。也可以用 egrep 命令（等同于 grep -E）。

-o只输出匹配部分grep -o "userId=\d+" app.log

配合 -P（Perl 正则）可以提取日志中的特定字段，如 grep -oP 'elapsed=\K\d+' app.log 提取耗时数字用于统计。

less — 分页查看大文件 
$ less -N app.log     # -N 显示行号
/ERROR                # 搜索 "ERROR"，n 下一个匹配，N 上一个
& 2025-05             # 只显示包含 "2025-05" 的行（过滤模式）
G                     # 跳到文件末尾，g 跳到开头
100g                  # 跳到第 100 行
F                     # 进入 tail -f 模式，Ctrl+C 退出

less 的优势：不把整个文件加载到内存（按需读取），支持前后翻页、搜索、过滤。grep 找到行号后，用 less +行号 文件名 直接定位。

awk — 列处理利器 
$ cat access.log | awk '{print $1}'                    # 打印第 1 列（IP）
$ cat access.log | awk '{sum+=$10} END {print sum}'    # 第 10 列求和（响应时间）
$ cat access.log | awk '{print $1, $NF}'               # 第 1 列和最后一列
$ cat access.log | awk -F '"' '{print $6}'             # 以双引号为分隔符，打印第 6 个字段
经典日志分析组合：cat access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -10 —— 统计访问量最多的前 10 个 IP 地址。这是银行面试必考的管道组合题。

awk 工作原理：对每一行执行 {操作}（主处理块），BEGIN 块在读取第一行之前执行（可定义变量、打印表头），END 块在所有行处理完后执行（打印汇总结果）。

sed — 流编辑器 
$ sed 's/old/new/g' file.txt          # 全文替换
$ sed -i 's/old/new/g' file.txt       # -i 直接修改文件（无需重定向）
$ sed -n '5,10p' file.txt             # 打印第 5-10 行
$ sed '/pattern/d' file.txt           # 删除匹配行
$ sed -n '/START/,/END/p' file.txt    # 打印从 START 到 END 之间的所有行

sed 在日志分析中常用于提取特定时间段的日志或清理格式。

sort / uniq — 排序和去重 
$ sort -n file.txt              # 按数值排序（默认字典序）
$ sort -r file.txt              # 逆序
$ sort -k 2 file.txt            # 按第 2 列排序
$ sort file.txt | uniq          # 去除相邻重复行
$ sort file.txt | uniq -c       # 统计每行的出现次数
$ sort file.txt | uniq -d       # 只显示重复的行
关键点：uniq 只能去除相邻重复行，所以通常先 sort 再 uniq。sort -n 和字典序的区别是面试常考点——"10" 在字典序中小于 "2"，因为按字符比较 '1' < '2'，所以必须用 -n 做数值排序。

4. 文件查找与操作

find — 文件查找（运维最常用命令之一） 
# 按名称查找
$ find /app/logs -name "*.log" -mtime +7 -exec rm {} \;
# 解析：-name 通配符匹配 | -mtime +7 修改时间超过7天 | -exec 对每个结果执行 rm

# 按大小查找
$ find . -type f -size +100M -exec ls -lh {} \;

# 按类型查找
$ find . -type f -name "*.java" | xargs grep "main"

# 按权限查找
$ find / -perm 777 -type f 2>/dev/null

# 查找并排除某些目录
$ find . -name "*.log" -not -path "./node_modules/*"
find -exec vs xargs 的区别（面试必问）：-exec 每次找到文件就启动一次后面的命令（效率低）；xargs 把前面的输出收集起来，分批传给后面的命令（效率高，但文件名含空格时会出错——用 find -print0 | xargs -0 解决）。经典清理脚本：find /app/logs -name "*.log" -mtime +30 -delete（删除 30 天前的日志）。

为什么要把 TID 转成 16 进制？

因为 jstack 输出的线程栈信息中，线程 ID 是 16 进制格式（nid=0x...）。top -Hp 和 ps -eLf 显示的是 10 进制线程 ID，必须转成 16 进制才能在 jstack 中匹配到对应的线程栈。

银行生产环境 CPU 飙升的常见原因：
  • 死循环：某个边界条件没处理好导致循环无法退出
  • 频繁 Full GC：堆空间不足，GC 线程持续运行（jstat 看 FGC 列快速增长）
  • 正则表达式回溯：用户输入匹配了有缺陷的正则导致灾难性回溯
  • JSON/XML 大报文解析：超大报文反复解析和序列化
  • hash 冲突：HashMap 的 key 设计不当导致大量元素落到同一个 bucket

三、JVM 排查命令（银行面试核心）

5. JDK 自带工具

工具核心用途常用命令
jps -l列出 Java 进程jps -l | grep -i server

jps（Java Process Status）输出 PID 和类名（或 jar 包名）。jps -l 显示完整包名，jps -v 显示 JVM 启动参数（非常有用——可以看到 -Xms、-Xmx、-XX:+UseG1GC 等关键配置）。jps 原理是读取 /tmp/hsperfdata_用户名/ 下的文件，如果该目录被清理则 jps 无法正常工作。

jstack PID线程转储（thread dump）jstack PID | grep -A 10 "BLOCKED\|WAITING\|deadlock"

jstack 是分析死锁的首选工具：输出末尾如果有 "Found 1 deadlock" 段落，说明发现了死锁。它直接列出死锁涉及的线程、各自持有的锁和等待的锁。也可以用 jstack -l PID（-l 显示锁的额外信息）。

解读 jstack 输出：每个线程一行标题（线程名、是否 daemon、优先级、nid=16 进制线程 ID），然后是线程状态（RUNNABLE/BLOCKED/WAITING/TIMED_WAITING）和调用栈。BLOCKED 状态大量出现说明锁竞争严重。

常见线程状态含义：RUNNABLE（正在运行或等待 CPU）、BLOCKED（等待获取对象锁）、WAITING（无限期等待 notify/join）、TIMED_WAITING（限时等待比如 Thread.sleep）、TERMINATED（已结束）。

jmap内存映射jmap -heap PID / jmap -histo PID / jmap -dump:format=b,file=heap.bin PID

三个常用子命令：

  • jmap -heap PID：堆配置和使用情况概览（各代的容量、使用量、GC 算法）
  • jmap -histo PID：类实例数量和大小的直方图（按内存占用排序），快速定位哪些类的实例占了最多内存：jmap -histo PID | head -20
  • jmap -histo:live PID：触发一次 Full GC 后再统计（只统计存活对象，更准确但会暂停应用）
  • jmap -dump:format=b,file=heap.bin PID：导出堆快照用于 MAT（Eclipse Memory Analyzer）离线分析，这个命令会触发 STW

生产环境注意事项：jmap -dump 会触发 Full GC（STW），在生产环境谨慎使用。优先使用 jmap -histo 做快速分析，或者应用启动时加 -XX:+HeapDumpOnOutOfMemoryError 让应用在 OOM 时自动生成 dump。

jstat -gc PIDGC 统计监控jstat -gc PID 1000 10（每秒输出，共 10 次）

jstat -gc 关键列解读：

列含义关注点
S0C / S1CSurvivor 0/1 容量两个应交替使用
EC / EUEden 容量 / 使用量EU/EC 比例高说明很快要 Minor GC
OC / OUOld 容量 / 使用量OU 持续增长不放 = 内存泄漏的关键信号
MC / MUMetaspace 容量 / 使用量MU 持续增长 = 动态类加载过多
YGC / YGCTYoung GC 次数 / 耗时YGC 增长快说明新对象创建频繁
FGC / FGCTFull GC 次数 / 耗时FGC 频繁增长 = 严重问题！

银行监控关键指标：FGC（Full GC）是银行系统最关注的指标之一。核心交易系统中 Full GC 频繁会导致长时间 STW（几秒甚至几十秒），直接造成交易超时、影响 SLA。监控平台通常对 FGC 设置告警阈值。

jinfo PIDJVM 配置信息jinfo PID | grep -i "Xms\|Xmx\|GC"

jinfo 可以查看 JVM 的所有系统属性和 VM flags。jinfo -flag <flag名> PID 查看某个 flag 的值（如 jinfo -flag UseG1GC PID）。jinfo -flag +PrintGCDetails PID 甚至可以动态开启某些诊断 flag（但不是所有 flag 都可动态修改）。

arthas阿里开源在线诊断工具dashboard / thread / watch / trace / jad

arthas 是生产环境诊断的终极利器：不需要重启应用，动态 attach 到 JVM 上。核心命令：dashboard（实时面板，类似 top 但只显示 Java 线程）、thread -b（自动检测死锁阻塞线程）、watch（观测方法入参和返回值）、trace（跟踪方法调用链路和耗时）、jad（在线反编译 class 文件）、redefine（热替换 class 文件）。

arthas 在银行场景的应用：线上交易变慢，用 trace 跟踪调用链路找到瓶颈方法；某笔交易返回数据异常，用 watch 观测该方法的入参和返回值；怀疑第三方 jar 包有 bug，用 jad 反编译查看实际逻辑。

6. 常见问题排查流程

场景一：CPU 飙升排查 
# 步骤 1：找到高 CPU 进程
top                        # 看到 java 进程 CPU 200%

场景二：内存泄漏排查 
# 步骤 1：观察 GC 趋势
jstat -gc 12345 2000       # 每 2 秒输出 GC 统计
# 观察到 OU（老年代使用）持续增长，FGC 越来越频繁

# 步骤 2：查看对象直方图
jmap -histo:live 12345 | head -30
# 看到某个自定义类有大量实例（如 100 万个 CacheEntry）

# 步骤 3：生成 heap dump 离线分析
jmap -dump:format=b,file=heap.bin 12345
# 用 MAT / JProfiler 打开，分析泄漏引用链
常见内存泄漏原因：静态集合（static Map/List）不断添加不释放；ThreadLocal 使用后未 remove() 导致线程池复用时残留；未关闭的流和连接；监听器注册后未注销；HashMap 的 key 没有实现 hashCode/equals 导致同一对象被重复放入。

为什么内存泄漏在银行系统特别危险？

银行核心交易系统（如支付、账务）要求 7x24 运行，不能轻易重启。内存泄漏会导致 Full GC 越来越频繁，每次 GC 暂停时间越来越长——从几毫秒发展到几秒甚至几十秒。当 GC 暂停超过交易超时时间，就会出现大面积交易失败。如果 Full GC 后内存仍然不足，最终 OOM 导致 JVM 崩溃。

OOM 自动 dump 配置：
  • -XX:+HeapDumpOnOutOfMemoryError（OOM 时自动生成堆转储）
  • -XX:HeapDumpPath=/app/dump/（指定 dump 文件路径，需确保磁盘空间充足）
  • -XX:OnOutOfMemoryError="/app/scripts/restart.sh"（OOM 时执行脚本，自动重启）

四、文件权限与 Shell 基础

7. 权限管理

chmod — 修改文件权限 
$ ls -l script.sh
-rw-r--r-- 1 user group 1024 May 25 10:00 script.sh
# 类型  owner  group  other
# -     rw-    r--    r--

$ chmod 755 script.sh     # 数字法：owner=rwx group=r-x other=r-x
$ chmod +x script.sh      # 符号法：给所有用户加执行权限
$ chmod u=rwx,g=rx,o=r file   # 精确设置

权限数字记忆法：r=4, w=2, x=1。三位数字分别代表 owner、group、other。755 = rwx(7) + r-x(5) + r-x(5)。常用组合：644（文件）、755（目录/脚本）、600（私密文件如密钥）。

chown / umask 
$ chown appuser:appgroup app.log      # 修改文件所有者和组
$ chown -R appuser:appgroup /app/     # -R 递归修改目录下所有文件

$ umask                                # 查看当前权限掩码
0022                                   # 默认值
# 新文件默认权限 = 666 - umask = 644 (rw-r--r--)
# 新目录默认权限 = 777 - umask = 755 (rwxr-xr-x)

8. Shell 基础

重定向与管道
符号含义示例
>覆盖写入文件echo "hello" > file.txt

覆盖重定向——如果文件已存在，内容会被清空后替换。在银行脚本中日志文件通常用 >>（追加）而非 >（覆盖），防止误清空历史日志。

>>追加写入文件echo "hello" >> file.txt

追加到文件末尾，不覆盖已有内容。最适合日志写入场景。

2>&1stderr 重定向到 stdoutjava -jar app.jar > app.log 2>&1

顺序很重要！2>&1 必须放在 > 的后面，因为重定向是从左到右处理的。如果写成 2>&1 > app.log，stderr 会继承旧 stdout（终端），而新 stdout（app.log）不会被 stderr 使用。

&> /dev/null丢弃所有输出（stdout + stderr）nohup java -jar app.jar &> /dev/null &

/dev/null 是 Linux 的"黑洞"设备文件，写入其中的数据会被内核直接丢弃。常用于静默执行或不需要输出的场景。等价于 > /dev/null 2>&1。

|管道——将左边 stdout 传给右边 stdinps aux | grep java | wc -l

管道只传递 stdout，不传递 stderr：如果要把 stderr 也传递给下一个命令，需要先重定向：command 2>&1 | grep ERROR。管道是 UNIX 哲学"小而美"的核心——每个命令只做一件事，通过管道组合完成复杂任务。

后台运行与变量 
# 后台运行
$ nohup java -jar app.jar > app.log 2>&1 &   # nohup: 忽略 HUP 信号，终端关闭不退出
$ java -jar app.jar &                         # 后台运行，但终端关闭后进程退出

# Shell 变量（注意：等号两边不能有空格！）
$ APP_HOME=/opt/app
$ echo $APP_HOME         # 输出 /opt/app
$ echo ${APP_HOME}/logs  # 输出 /opt/app/logs（花括号防止歧义）

# 常用组合
$ ps aux | grep java | wc -l           # 统计 Java 进程数
$ ss -tlnp | grep :8080               # 查端口占用
$ find / -type f -size +100M 2>/dev/null  # 查大文件（忽略权限错误）
$ ps aux | grep java | awk '{print $2}' | xargs kill -15   # 优雅批量杀 Java 进程

▶ 展开详解
D 状态为什么设计为"不可中断"？

进程在读写磁盘时，内核把数据从磁盘读到内核缓冲区，这个过程必须完整执行完。如果允许被信号中断，数据可能处于不一致的状态（读到一半）。D 状态通常是短暂的（磁盘 IO 完成后恢复），但如果磁盘有问题（NFS 服务器挂死、坏块等），D 状态可能持续很久——这就是为什么 D 状态的进程无法被 kill -9 杀掉（内核在等待 IO 驱动的回调）。D 状态的线程数被计入 load average，所以磁盘 IO 问题也会导致 load 飙升。

Zombie 进程的危害：

僵尸进程本身不占用 CPU 和内存（只有进程表中残留一条记录），但它占用一个 PID。Linux 的 PID 数量有限（默认 32768），大量僵尸进程可能耗尽 PID 导致无法创建新进程。解决方法：使用 wait()/waitpid() 回收子进程，或把父进程的 SIGCHLD 信号处理设为 SIG_IGN（自动回收），或直接 kill 僵尸的父进程（孤儿进程由 init 回收）。

六、银行运维面试高频考点

考点 1：df 和 du 结果不一致的原因？
已删除文件仍被进程持有（fd未释放）。用 lsof | grep deleted 定位，重启该进程释放磁盘空间。
考点 2：load average 是什么？什么值算高？
R状态 + D状态线程数的平均值（1min/5min/15min）。load > CPU 核心数 = 过载信号。但要注意 D 状态是由 IO 瓶颈导致的（非 CPU 瓶颈），所以 load high + CPU idle high = IO 瓶颈。
考点 3：kill -9 和 kill -15 的区别？
kill -9 (SIGKILL) 由内核直接终止进程，不可捕获，进程无机会清理资源。kill -15 (SIGTERM) 通知进程自行退出，进程可捕获做善后（关闭连接、刷新缓冲、释放锁）。优雅停止：先 SIGTERM，等 N 秒后仍存活再 SIGKILL。
考点 4：如何查看 Java 应用的 GC 情况？
jstat -gc PID 1000 10，关注 FGC（Full GC 次数）和 FGCT（Full GC 累计耗时）。FGC 频繁增长说明堆配置不合理或有内存泄漏。
考点 5：管道和重定向的区别？
管道 (|) 连接两个命令，把前者的 stdout 传给后者的 stdin。重定向 (>/>>) 把命令的输出写到文件。管道只传递 stdout（不传递 stderr），如果想在管道中传递 stderr，需要先 2>&1。
考点 6：服务器连接数过多怎么排查？
ss -tan | awk '{print $1}' | sort | uniq -c | sort -rn 查看各状态的连接数分布。ESTABLISHED 过多检查连接池配置；TIME_WAIT 过多检查是否短连接频繁创建；CLOSE_WAIT 过多说明应用没有正确 close() 连接（代码 bug）。
