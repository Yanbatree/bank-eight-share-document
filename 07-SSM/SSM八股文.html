一、Spring IoC

1. IoC 和 DI

graph TD subgraph "传统方式 - 控制正转" A["类 A"] -->|"主动 new"| B["类 B"] end subgraph "IoC - 控制反转" C["IoC 容器"] -->|"注入"| D["类 A"] C -->|"注入"| E["类 B"] D -.->|"使用"| E Note["创建和组装由容器负责
应用程序被动接收"] end
IoC (Inversion of Control)：对象的创建、管理、装配权从程序员反转给 Spring 容器。
DI (Dependency Injection)：IoC 的具体实现方式——容器在运行时动态注入依赖（构造器注入、Setter 注入、字段注入）。
推荐构造器注入：不依赖反射，字段可设为 final，方便测试。

2. AOP 实现原理

graph TD subgraph "JDK 动态代理" A["目标类实现了接口"] --> B["Proxy.newProxyInstance()"] B --> C["生成 $Proxy 类
实现相同接口"] end subgraph "CGLIB 代理" D["目标类无接口"] --> E["Enhancer 生成子类"] E --> F["生成子类代理
继承目标类"] end Note["Spring Boot 2.x 默认 CGLIB
Spring Boot 1.x 默认 JDK 动态代理"]
JDK 动态代理 vs CGLIB： • JDK 代理：基于接口，只能代理有接口的类，生成速度快 • CGLIB：基于继承，不能代理 final 类/方法，运行时生成字节码 • 注意：同一个类内部方法相互调用 AOP 不生效（this.xxx() 不走代理对象）

三、Spring 事务

1. 事务传播行为

传播行为说明场景
REQUIRED有事务则加入，没有则新建（默认）最常用
REQUIRES_NEW总是新建事务，挂起当前事务日志记录（独立提交）
NESTED嵌套事务，内层回滚不影响外层Savepoint 机制
SUPPORTS有事务则加入，没有则非事务执行查询
MANDATORY必须在事务中，否则抛异常强制事务
NEVER不能在事务中，否则抛异常—
NOT_SUPPORTED非事务执行，有则挂起—

2. 事务失效的 8 大场景

场景原因解决
① 方法非 publicAOP 代理只能拦截 public改为 public
② 同类内部调用this.method() 不走代理注入自己 / 拆分到另一个类
③ 异常被 try-catch 吞了Spring 靠异常回滚catch 中抛异常 / 手动回滚
④ 异常类型不匹配默认只回滚 RuntimeException@Transactional(rollbackFor = Exception.class)
⑤ 数据库引擎不支持MyISAM 不支持事务用 InnoDB
⑥ 多线程不同线程不同事务—
⑦ 传播行为设置错误如设为 NOT_SUPPORTED检查 propagation 设置
⑧ 类本身没被 Spring 管理不是 Bean，没有代理加 @Service/@Component

五、Spring MVC

Spring MVC 请求处理流程

sequenceDiagram participant C as 客户端 participant DS as DispatcherServlet participant HM as HandlerMapping participant HA as HandlerAdapter participant H as Controller participant VR as ViewResolver participant V as View C->>DS: HTTP 请求 DS->>HM: 查找 Handler HM-->>DS: HandlerExecutionChain
(Handler + Interceptors) DS->>HA: 执行 Handler HA->>H: 调用 Controller 方法 H-->>HA: ModelAndView / @ResponseBody JSON HA-->>DS: ModelAndView DS->>VR: 解析视图 VR-->>DS: View 对象 DS->>V: 渲染 V-->>DS: 渲染结果 DS-->>C: HTTP 响应
核心组件： • DispatcherServlet：前端总控制器 • HandlerMapping：根据 URL 找到 Handler • HandlerAdapter：执行 Handler 的适配器 • HandlerInterceptor：拦截器（pre → Controller → post → afterCompletion） • ViewResolver：解析视图名到具体 View（前后端分离后少用了） • HttpMessageConverter：JSON/XML 序列化（@ResponseBody 用这个）

六、MyBatis

1. MyBatis 核心原理

flowchart TD A["SqlSessionFactory"] --> B["SqlSession"] B --> C["Executor 执行器"] C --> D["MappedStatement
(SQL 封装)"] D --> E["TypeHandler
参数/结果映射"] E --> F["JDBC 执行"] F --> G["数据库"]
#{} vs ${}： • #{}：预编译占位符 → ?，防 SQL 注入。用于参数值 • ${}：直接字符串拼接，有 SQL 注入风险。用于表名/列名等动态标识 缓存： • 一级缓存：SqlSession 级别，默认开启 • 二级缓存：Mapper 级别，需手动开启（<cache/>），跨 SqlSession 插件机制：通过拦截器（Interceptor）拦截 Executor、StatementHandler、ParameterHandler、ResultSetHandler。分页插件（PageHelper）就是拦截 Executor 的 query 方法。
