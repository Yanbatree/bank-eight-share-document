二、缓存三大问题

1. 缓存穿透 / 击穿 / 雪崩

graph TD subgraph "🔴 缓存穿透" P1["请求查询不存在的数据"] --> P2["缓存没有，DB也没有"] P2 --> P3["大量请求直接打到 DB"] P3 --> P4["解决：布隆过滤器 / 缓存空值"] end subgraph "🟡 缓存击穿" H1["热点 key 过期"] --> H2["大量并发请求同一 key"] H2 --> H3["瞬时流量全部打到 DB"] H3 --> H4["解决：互斥锁 / 逻辑过期"] end subgraph "🔵 缓存雪崩" X1["大量 key 同时过期 / Redis 宕机"] --> X2["请求全部打到 DB"] X2 --> X3["DB 扛不住，服务崩溃"] X3 --> X4["解决：过期加随机值 / 高可用集群"] end
问题原因解决方案
穿透查询不存在的数据① 布隆过滤器 ② 缓存空值（短过期） ③ 参数校验
击穿热点 key 过期瞬间① 互斥锁（setnx） ② 不设过期 + 异步更新 ③ 逻辑过期

2. 缓存和数据库一致性

sequenceDiagram participant App as 应用 participant Cache as Redis participant DB as MySQL Note over App,DB: ✅ 推荐的更新策略：先更新 DB，再删缓存 App->>DB: ① UPDATE 数据 DB-->>App: 成功 App->>Cache: ② DELETE 缓存 Note over App,Cache: 下次读时 Cache Miss，从 DB 加载最新数据 Note over App,DB: ❌ 为什么不是"先删缓存，再更新 DB"？ Note over App,DB: 先删 → 另一线程读到旧数据写入缓存 → 缓存是旧数据
最终一致性方案： ① 先更新 DB，再删缓存（最常用） ② 删缓存失败 → 重试 / 发 MQ 消息异步重试 ③ 延时双删：删缓存 → 更新 DB → 延时再删一次 ④ Canal：监听 binlog，异步更新缓存（适合强一致场景）

3. 过期删除策略

惰性删除 + 定期删除：
• 惰性删除：访问 key 时检查是否过期，过期则删除。省 CPU，占内存
• 定期删除：每隔一段时间随机抽一批 key 检查并删除过期 key。折中方案
Redis 用的是两种结合：定期随机抽取 + 访问时惰性检查

4. 内存淘汰策略（8种）

策略行为
noeviction不淘汰，写请求直接报错（默认）
allkeys-lru所有 key 中，淘汰最近最少使用的（推荐）
allkeys-lfu所有 key 中，淘汰使用频率最低的
volatile-lru有过期时间的 key 中，淘汰最近最少使用的
volatile-lfu有过期时间的 key 中，淘汰使用频率最低的
volatile-ttl有过期时间的 key 中，淘汰 TTL 最短的
allkeys-random所有 key 中随机淘汰
volatile-random有过期时间的 key 中随机淘汰
LRU vs LFU：LRU 看"最近" → 热点数据不容易淘汰；LFU 看"频率" → 防止偶尔访问一次就留下来

最佳实践：RDB + AOF 混合使用。Redis 4.0+ 支持混合持久化——RDB 快照 + 增量 AOF。

哨兵三个任务：监控（心跳检测）、自动故障转移（选新 Master）、通知客户端。
哨兵集群：至少 3 个实例，避免单点故障。用 Raft 类似算法选出 Leader 执行故障转移。

3. Cluster 集群（3.0+）

16384 个哈希槽，每个节点负责一部分槽。 
slot = CRC16(key) % 16384
特点： • 去中心化：节点间通过 Gossip 协议通信 • 主从结构：每个分片有 Master + Slave • 客户端重定向：key 不在当前节点 → 返回 MOVED（永久）或 ASK（临时迁移中） • 至少 3 主 3 从 保证高可用 为什么是 16384？ 心跳包大小 + 节点数限制的折中。16k 槽足够 1000 个节点用。

四、分布式锁

1. 用 Redis 实现分布式锁

sequenceDiagram participant A as 客户端 A participant R as Redis participant B as 客户端 B A->>R: SET lock_key unique_value NX EX 10 R-->>A: OK ✅ 获取锁成功 Note over A: 执行业务逻辑... B->>R: SET lock_key unique_value NX EX 10 R-->>B: nil ❌ 锁被占用 A->>R: Lua 脚本：先判断 value 是否自己的，是则 DEL Note over A: GET value == unique_value → DEL key R-->>A: OK 释放锁
核心要素：
① SET key value NX EX timeout：原子加锁 + 设过期
② value 用唯一标识（UUID / 线程ID）：释放时验证，防止误删
③ Lua 脚本释放：先 GET 判断 value，相等再 DEL——保证原子性
④ 过期时间要足够：防止业务没执行完锁就过期了

2. 分布式锁的常见问题

问题方案
锁过期，业务还没结束看门狗机制（Redisson）：定时续期
释放了别人的锁value 存唯一标识 + Lua 脚本验证后删除
主从切换锁丢失RedLock：向多个独立实例加锁，过半成功才算获取
单点故障使用 RedLock / Zookeeper 方案

五、线程模型

1. 单线程模型详解

graph TD subgraph "Redis 单线程模型" A["多个客户端连接"] --> B["IO 多路复用
(epoll/select/kqueue)"] B --> C["事件队列"] C --> D["单线程
逐个处理事件"] D --> E["读事件 → 解析命令 → 执行 → 写响应"] end
Redis 6.0 之前为什么单线程？
• 瓶颈在内存和网络，不在 CPU
• 单线程避免了上下文切换和锁竞争
• 数据结构简单高效，命令执行很快

缺点：单个慢命令（如 KEYS、FLUSHALL）会阻塞所有请求。大 key 的 DEL 也会阻塞。

2. IO 多路复用

核心思想：一个线程同时监听多个 socket，哪个有事件就处理哪个。

系统调用机制特点
select遍历所有 fdO(n)，有 1024 限制
poll遍历，链表存 fdO(n)，无数量限制
epoll（Linux）回调机制，内核通知就绪 fdO(1)，无数量限制，Redis 首选

3. Redis 6.0+ 的多线程

注意：Redis 6.0 引入的多线程只是网络 IO 多线程，命令执行仍然是单线程。 
读请求解析（多线程）→ 命令执行（单线程）→ 写响应（多线程）
为什么不全多线程？保证原子性——单线程执行命令天然线程安全，无需加锁。
