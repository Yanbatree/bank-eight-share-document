▶ 展开详解
MVCC（Multi-Version Concurrency Control）核心思想：

允许多个事务同时读取同一行数据而不会彼此阻塞——每个事务看到的是它开始时的数据版本。通过"版本链"来控制并发事务访问同一记录的行为。

关键数据结构：

1. 版本链：由行记录的 DB_ROLL_PTR 串联形成的链表，每个节点是数据的历史版本（存储在 undo log 中）。

2. Read View：包含活跃事务 ID 信息的结构体，用于判断版本可见性。「读提交」在每个 SELECT 前重新生成 Read View；「可重复读」在第一条 SELECT 时生成一个 Read View 并贯穿整个事务。

3. Undo Log：存储旧版本数据的日志文件，支持回滚和 MVCC。

4. 隐藏列：InnoDB 每条聚簇索引记录有两个隐藏列——trx_id（最近修改该行的事务ID）和 roll_pointer（指向 undo log 中上一版本的指针）。

面试加分点：

MVCC 只在 读提交 和 可重复读 隔离级别下有效。读未提交直接读最新版本（不需要 MVCC），串行化通过加锁实现（不需要 MVCC）。

从字节码到操作系统：

synchronized 编译后在同步代码块前后加上 monitorenter 和 monitorexit 字节码指令，依赖操作系统底层互斥锁（Mutex Lock）实现。执行 monitorenter 时尝试获取对象锁，计数器 +1；执行 monitorexit 时计数器 -1，归零时释放锁。

Monitor 内部的两个队列：

1. EntryList：竞争锁的线程先进入此队列，等待获取 monitor 锁。

2. WaitSet：线程调用 wait() 后释放锁并从 Owner 移入此队列，等待 notify/notifyAll 唤醒后重新进入 EntryList 竞争锁。

锁升级过程（JDK 1.6 后）：

偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁（Monitor），这是 synchronized 性能优化的关键。银行面试通常只问到 Monitor 层面，但如果你能说出锁升级过程，是明显的加分项。

内核态切换开销：

synchronized 是排他锁，线程阻塞/唤醒涉及用户态到内核态的切换，这是性能消耗的主要来源。这也是为什么 JDK 引入了锁升级机制——尽量避免进入重量级锁。

Q3: 两个线程交替打印 1-10 如何实现？

核心思路：synchronized + wait()/notify() 实现线程间通信。 代码实现： 
public class AlternatePrint {
    private static int count = 1;
    private static final Object lock = new Object();

    public static void main(String[] args) {
        // 线程A：打印奇数
        new Thread(() -> {
            while (count <= 10) {
                synchronized (lock) {
                    if (count % 2 == 1) {
                        System.out.println("线程A: " + count++);
                        lock.notify();
                    } else {
                        try { lock.wait(); } catch (InterruptedException e) {}
                    }
                }
            }
        }, "线程A").start();

        // 线程B：打印偶数
        new Thread(() -> {
            while (count <= 10) {
                synchronized (lock) {
                    if (count % 2 == 0) {
                        System.out.println("线程B: " + count++);
                        lock.notify();
                    } else {
                        try { lock.wait(); } catch (InterruptedException e) {}
                    }
                }
            }
        }, "线程B").start();
    }
}
输出结果： 
线程A: 1  线程B: 2  线程A: 3  线程B: 4  ...  线程A: 9  线程B: 10
▶ 展开详解
除了 wait/notify，还有哪些实现方式？

1. Lock + Condition：用 ReentrantLock 的 newCondition() 创建两个 Condition，分别控制奇数和偶数线程的等待唤醒，比 synchronized 更灵活。

2. Semaphore：创建两个信号量，奇数线程持有 permit，偶数线程初始为 0，打印后互相释放对方的 permit。

3. BlockingQueue：用 SynchronousQueue 或 TransferQueue 实现线程间的一对一传递。

4. AtomicInteger + 自旋：用 CAS 循环判断当前应打印奇数还是偶数（但会浪费 CPU）。

面试加分点：如果面试官追问"多个线程交替打印"，可以提到用 Semaphore 的 acquire/release 链实现 N 个线程的循环打印（如 3 线程交替打印 ABC）。

Q4: 你项目中 Redis 数据一致性问题怎么解决的？

graph TD A["写操作: 先更新 DB"] --> B["再删除 Cache"] B --> C{"删除成功?"} C -->|"成功"| D["完成"] C -->|"失败"| E["方案1: 消息队列重试"] C -->|"失败"| F["方案2: Canal 订阅 Binlog"] E --> G["MQ 消费者重试删除"] F --> H["Canal 伪装 MySQL 从库"] H --> I["解析 Binlog → MQ → 删除缓存"]
▶ 展开详解
为什么是"更新DB + 删除缓存"而不是"更新DB + 更新缓存"？

1. 删除缓存是幂等操作，更新缓存不是（并发写可能导致缓存覆盖成旧值）。

2. 删缓存更轻量——如果缓存中数据计算复杂（涉及多表join），更新缓存代价大且可能浪费（写完后从未被读就又被更新了）。

3. 懒加载思路：删了缓存后，下次读请求 miss 时从 DB 加载最新数据，保证数据新鲜度。

删除缓存失败的补救方案：

方案1 — 消息队列重试：删除失败 → 将删除操作写入MQ → 消费者重试删除。优点：实现简单。缺点：业务代码侵入。

方案2 — Canal + Binlog 订阅：Canal 伪装为 MySQL 从库，订阅 binlog 变更 → 解析后发到 MQ → 缓存删除消费者处理。优点：完全解耦，业务代码零侵入。缺点：架构复杂度增加，Canal 本身需要维护。

一定要记住的面试关键句："缓存是通过牺牲强一致性来换取性能的，属于 CAP 中的 AP。可以保证最终一致性，但做不到强一致性。缓存一定要设置过期时间作为兜底。"

Q5: Linux 如何查看一个 Java 进程？

方法1 — ps 命令（最常用）： 
ps -ef | grep java          # 查看所有 Java 进程详细信息
ps -aux | grep java         # 简化输出（PID + 命令）
方法2 — jps 命令（JDK 自带，专门用于 Java 进程）： 
jps                         # 显示 PID 和主类名/JAR名
jps -lvm                    # 显示详细启动参数
方法3 — top 命令（实时监控）： 
top -p <PID>                # 监控指定进程的 CPU/内存
top -c                       # 显示完整命令行
▶ 展开详解
各命令的输出关键字段：

ps -ef 输出：UID（用户）、PID（进程ID）、PPID（父进程ID）、C（CPU使用率）、STIME（启动时间）、CMD（完整命令行）。其中 PID 用于后续的堆栈分析，CMD 中可以识别出是哪个 Java 服务。

银行场景的常见追问：

"找到进程后怎么排查问题？" → jstack PID（线程堆栈，排查死锁/CPU 高）、jmap -heap PID（堆内存概况）、jstat -gc PID 1000（GC 实时监控）、jinfo PID（JVM 参数）。这几个命令在银行运维场景中非常实用，建议记住。

五、无领导小组讨论

常见形式：8-12人一组，给定一个话题（可能是金融科技相关、社会责任相关、或通用管理类），25-40分钟讨论后汇报结论。 常见角色：
角色职责风险
Leader把握讨论节奏、分配时间控制欲太强会扣分
Timer提醒时间节点只报时不行，要推动进度
Reporter总结汇报讨论结论逻辑不清、遗漏要点
Reporter 的汇报技巧：用"我们小组经过讨论认为..."开头（强调团队而非个人）。先说结论再讲理由。如果组内有分歧，在汇报中客观呈现（"部分同学认为A方案更好，理由是..."）。时间控制在2-3分钟内。如果作为 Reporter 能补充一句"以上总结可能不够全面，欢迎其他同学补充"，体现谦逊和团队意识，很加分。
Contributor提供建设性观点发言太少没存在感
Contributor 如何出彩：不是发言越多越好，而是每次发言要有质量。高质量发言类型：打破僵局（"我们先明确一下讨论框架..."）→ 总结归纳（"目前我们讨论了A和B两点，是否可以先确定A，再聚焦B？"）→ 推动共识（"大家对这个方案似乎没有太多异议，我们是否可以进入下一步？"）→ 补充视角（"从技术可行性的角度，这个方案需要考虑XX因素"）。银行面试官更看重团队协作而非个人表现。
银行面试的特殊性：银行的无领导 不鼓励太 aggressive，更看重团队协作意识、逻辑清晰度、沟通表达能力。千万别抢话、否定别人时要有建设性。

Q1: 数据库的索引用的什么数据结构？

graph TD subgraph "InnoDB 索引结构" A["主键索引（聚簇索引）"] --> B["B+ 树"] C["二级索引（非聚簇索引）"] --> B D["自适应哈希索引（AHI）"] --> E["Hash 表"] end subgraph "B+ 树核心优势" F["非叶子只存键 → 树矮 → IO 少"] G["叶子双向链表 → 范围查询快"] H["叶子存完整数据 → 主键查询一次到位"] end
核心答案：InnoDB 主键索引使用 B+ 树，哈希索引使用 Hash 表。B+ 树叶子节点按照主键顺序存放整行数据，叶子之间通过双向指针连接形成有序链表——这使得等值查询和范围查询都能高效完成，且磁盘 IO 次数少（3-4 层即可覆盖千万级数据）。
▶ 展开详解
为什么 InnoDB 选择 B+ 树而不是其他数据结构？
  • vs 哈希索引：哈希 O(1) 等值查询极快，但完全不支持范围查询（WHERE id > 100）、排序（ORDER BY）、模糊匹配（LIKE）。InnoDB 的自适应哈希索引（AHI）仅作为 B+ 树的补充——当某个数据页被频繁等值访问时，InnoDB 自动在内存中为它建立哈希索引，加速热点查询。
  • vs 二叉搜索树/红黑树：二叉树每个节点只有两个子节点，数据量大时树高极高（1000 万行 → 树高约 23 层），每次查询需要 ~23 次磁盘 IO。B+ 树每个节点是一个 16KB 的页，可以存放成百上千个索引键（扇出约 1170），高度通常仅 3-4 层。
  • vs B 树：B 树的非叶子节点也存储完整数据行，导致每个页能存的键更少、树更高。B+ 树非叶子节点只存键不存数据，单个页能放更多键，树更矮。且 B+ 树叶子节点的双向链表天然支持范围扫描，而 B 树需要中序遍历来回溯父节点。
B+ 树在 InnoDB 中的物理存储细节：

InnoDB 以 16KB 页为最小存储单位，每个页包含若干条记录。B+ 树的根页、分支页（内节点）、叶子页均以页的形式存储在表空间（.ibd 文件）中。叶子页之间通过 FIL 页头中的前后指针（previous page / next page）连接成双向链表。页内记录通过单向指针连接。当页满时触发页分裂（page split），当页太空时触发页合并（page merge）。

面试加分点：提一下 InnoDB 的自适应哈希索引（AHI）——它是自动的，不需要手动创建，当 InnoDB 检测到某个索引被频繁等值查询时，会在 buffer pool 中为热点页建哈希索引。可以通过 SHOW ENGINE INNODB STATUS 查看 AHI 命中率。

Q2: 索引失效的几个场景？

#失效场景原因示例
1索引列使用函数或运算破坏了索引值，优化器无法匹配WHERE YEAR(create_time)=2024
深入解析：索引存的是 create_time 列的原始值（如 '2024-03-15 10:30:00'），而 WHERE 条件是对 YEAR() 函数的结果（2024）进行判断。优化器看到的不是"create_time 在某个区间"，而是"某个函数计算后的值等于 2024"——完全无法利用 B+ 树的有序性。
解决方案：WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'——等价的范围条件，可以走索引。
2左模糊 LIKE无法二分定位起点WHERE name LIKE '%张三'
为什么 LIKE '张%' 可以走索引？因为 B+ 树按字典序排序，知道以"张"开头，可以直接在 B+ 树中定位到"张"的起始位置，然后向后扫描。而 '%张' 表示"以张结尾的任意字符串"——在字典序中"张"可能在任意位置（如"a张"、"b张"、"c张"），无法二分查找定位。
面试加分：如果确实需要后缀模糊查询，可以建反向索引（把字符串反转后建索引），或者用全文索引（FULLTEXT）。
3隐式类型转换MySQL 对索引列做了隐式函数转换WHERE phone=13800138000（phone 是 varchar）
为什么是索引列被转换？MySQL 的隐式转换规则是：字符串和数字比较时，把字符串转为数字。所以 phone（varchar）被隐式 CAST 为数字——相当于 WHERE CAST(phone AS SIGNED)=13800138000，索引列上有了函数，索引失效。
反过来不失效：WHERE id='100'（id 是 int 型）——MySQL 把 '100' 转为数字 100，不会对 id 列做转换，索引正常。
字符集不一致也会导致隐式转换：两个表的 JOIN 列字符集不同（如 utf8mb4 vs utf8），也会触发隐式转换导致索引失效。
4!= / NOT IN / IS NOT NULL非等值无法精确定位区间WHERE status != 1
分情况讨论：
  • != 和 <>：大部分情况不走索引（因为"不等于 1"意味着大部分行都满足，优化器认为回表代价太高）。但如果结合其他等值条件，优化器可能走其他列的索引。
  • NOT IN：通常不走索引，且注意 NULL 陷阱——如果 NOT IN 的子查询结果中有 NULL，整个查询返回空。
  • IS NOT NULL：在某些情况下可以走索引（MySQL 5.6+ 对 IS NOT NULL 做了一定的索引优化），但通常不如 IS NULL 高效。
5OR 条件中一侧无索引优化器认为全表扫描更快WHERE a=1 OR b=2（b 没索引）
优化策略：如果 OR 两侧各自有索引，MySQL 5.6+ 可以用 index_merge（索引合并）——在内存中用 bitmap 做并集运算。如果 OR 一侧无索引，改写成 UNION ALL：SELECT * FROM t WHERE a=1 UNION ALL SELECT * FROM t WHERE b=2 AND a!=1（让 a=1 走索引，b=2 全表扫描但只扫一次不重复 a=1 的部分）。
6联合索引不满足最左前缀跳过最左列，后续列无法二分定位索引 (a,b,c)，只查 WHERE b=2
最左前缀的核心原因：联合索引按 (a, b, c) 的顺序排序——先按 a 排，a 相同再按 b 排，b 相同再按 c 排。如果跳过 a 直接查 b，索引就不是有序的了，无法用二分查找定位起始位置。
特殊情况——跳跃扫描（Index Skip Scan）：MySQL 8.0.13+ 引入了跳跃扫描优化——当联合索引的前导列基数很低（比如 a 只有 2-3 个不同值）时，优化器会自动拆成多个 a 等值条件分别扫描。但这不是通用的，不能依赖。
面试口诀："函数 LIKE% 转类型，!= OR 没前缀"——六种场景一个字不落说出来，面试官会认为你基础扎实。

Q3: HAVING 关键字作用？

graph LR A["WHERE age > 18"] --> B["过滤后 600 行"] B --> C["GROUP BY dept_id 分组"] C --> D["聚合: COUNT/SUM/AVG"] D --> E["HAVING COUNT > 10 过滤"]
对比维度WHEREHAVING
过滤对象原始数据行分组后的聚合结果
执行阶段GROUP BY 之前GROUP BY 之后
能否用聚合函数不能可以（COUNT/SUM/AVG 等）
能否用列别名不能（WHERE 先于 SELECT 执行）可以（MySQL 扩展，非标准 SQL）
▶ 展开详解
完整 SQL 的执行顺序（必背）：

FROM → ON → JOIN → WHERE → GROUP BY → 聚合函数计算 → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT

这个顺序可以解释很多现象：为什么 WHERE 不能用别名（SELECT 在后面执行，别名还没定义）→ 为什么 HAVING 可以用聚合函数（聚合函数在它前面计算）→ 为什么 ORDER BY 可以用别名（SELECT 在它前面执行）。

实战案例： 
-- 查询平均薪资 > 15000 的部门中，25 岁以上员工的平均薪资
SELECT dept_id, AVG(salary) AS avg_sal
FROM employee
WHERE age > 25          -- ① 先过滤掉 ≤25 岁的
GROUP BY dept_id        -- ② 按部门分组
HAVING AVG(salary) > 15000  -- ③ 只保留均薪 > 15000 的部门
ORDER BY avg_sal DESC;  -- ④ 按平均薪资降序排列
HAVING 可以不用 GROUP BY 单独使用吗？

语法上可以（MySQL 允许），但这就退化为相当于 WHERE 了。标准 SQL 中 HAVING 应该和 GROUP BY 配对使用。面试中不建议提这种"语法糖"用法。

Q4: 数据库优化了解过吗？

graph TD A["数据库优化"] --> B["SQL 层面"] A --> C["架构层面"] A --> D["参数层面"] B --> B1["合理建索引（覆盖索引 + 联合索引）"] B --> B2["避免索引失效（函数/LIKE%/OR）"] B --> B3["杜绝 SELECT *（覆盖索引失效）"] B --> B4["深分页优化（延迟关联）"] B --> B5["小表驱动大表 JOIN"] C --> C1["读写分离（主写从读）"] C --> C2["分库分表（水平+垂直）"] C --> C3["冷热数据分离"] C --> C4["引入缓存（Redis）"] D --> D1["buffer_pool 调大"] D --> D2["连接数 / 超时时间"] D --> D3["慢查询阈值 long_query_time"]
▶ 展开详解
五个优化层面逐一展开： ① 建合适索引：
  • 为 WHERE / JOIN / ORDER BY / GROUP BY 的列建索引
  • 高频查询考虑覆盖索引（把 SELECT 的列都包含在联合索引中，避免回表）
  • 区分度低的列（如性别、状态）不建议单独建索引，但可以放在联合索引的最左列（利用其过滤后的数据再排序）
  • 避免过多索引——索引占磁盘空间、拖慢写入速度（INSERT/UPDATE/DELETE 需要维护所有索引）
② 优化 SQL 写法：
  • 杜绝 SELECT * → 一方面浪费网络带宽，另一方面破坏覆盖索引的可能性
  • 避免索引失效场景（函数、前置 LIKE、隐式转换、OR 无索引）
  • 多用 IN 代替多个 OR（MySQL 会对 IN 做优化）
  • 大偏移量 LIMIT 用延迟关联改写
  • JOIN 时小表驱动大表，被驱动表关联列必须有索引
③ 分库分表：
  • 垂直分库：按业务拆库（用户库、订单库、商品库），缓解单库并发压力
  • 垂直分表：冷热字段分离（user 拆为 user_base + user_detail），减少单行大小，提高 buffer pool 利用率
  • 水平分库分表：按分片键（如 user_id）hash 取模，解决单表数据量过大的问题（>500 万行考虑拆分）
④ 读写分离：
  • 一主多从架构，写操作走主库、读操作走从库
  • 需要处理主从延迟问题——对实时性要求高的读（如刚写入后立即读）仍需走主库
  • 中间件方案：ShardingSphere-Proxy、MyCat、ProxySQL
⑤ 参数调优：
  • innodb_buffer_pool_size：调大缓冲池（建议物理内存的 60-80%），减少磁盘 IO
  • slow_query_log + long_query_time：开启慢查询日志，持续监控
  • max_connections：根据业务并发量合理设置
  • innodb_flush_log_at_trx_commit：强一致性设为 1，高并发可设为 2（但可能丢失 1 秒数据）

Q5: 怎么看 SQL 执行计划？

关键字段含义判断标准
type访问类型（最重要）system > const > eq_ref > ref > range > index > ALL
各类型详解：
  • const：主键或唯一索引等值查询，最多返回一行。最优。
  • eq_ref：JOIN 时被驱动表走主键/唯一索引。每个驱动行匹配至多一行，非常好。
  • ref：非唯一索引等值查询。可能返回多行，正常水平。
  • range：索引范围扫描（>、<、BETWEEN、IN）。可接受。
  • index：全索引扫描（扫整个索引树）。虽然用了索引但没有减少数据量，需优化。
  • ALL：全表扫描。需要优化的信号。
key实际使用的索引名有值说明走了索引；NULL 说明没走
rows优化器预估扫描行数越小越好。远大于实际返回行数 → 索引选择性差
rows 的局限性：rows 是基于统计信息的预估值，不是实际扫描行数。当统计信息过期时（表中数据大量变更后未执行 ANALYZE TABLE），rows 可能严重失准。通过 SHOW INDEX FROM table 查看索引基数（Cardinality），Cardinality 越接近表总行数，索引区分度越好。
Extra额外重要信息Using index（好） | Using filesort（差） | Using temporary（差）
Extra 关键值解读：
  • Using index（好）：覆盖索引，不需要回表。
  • Using index condition（中等）：索引下推（ICP），在索引层过滤后再回表。
  • Using where（一般）：Server 层额外做了 WHERE 过滤。
  • Using filesort（坏）：需要额外排序——排序列没有索引，或排序方式无法利用索引。优化重点。
  • Using temporary（坏）：需要临时表（常见于 GROUP BY 和 ORDER BY 不同列、DISTINCT + ORDER BY 等场景）。严重影响性能。
  • Using join buffer（需关注）：JOIN 时被驱动表没走索引，数据被放入 join buffer。给被驱动表的 JOIN 列加索引。
key_len索引使用的字节数越大说明联合索引中用到的列越多
key_len 实战用法：联合索引 (name VARCHAR(50), age INT, dept_id INT)，UTF-8MB4 编码下：
- key_len=202 → 只用了 name 列（50×4+2=202）
- key_len=206 → 用了 name + age（202+4=206）
- key_len=210 → 用了完整三列（206+4=210）
用这个可以验证最左前缀是否按预期生效。

EXPLAIN 的三种用法：
  • EXPLAIN SELECT ...：查看执行计划（最常用）
  • EXPLAIN FORMAT=JSON SELECT ...：JSON 格式输出，包含优化器的成本估算（cost_info）和候选方案（considered_execution_plans）——面试提这个能体现深度
  • EXPLAIN ANALYZE SELECT ...（MySQL 8.0.18+）：实际执行 SQL 并输出每一步的真实耗时和实际返回行数，与估算的 rows 对比可以发现统计信息问题
面试实战话术："拿到一条慢 SQL，我先 EXPLAIN 看 type 是不是 ALL/index，然后看 key 有没有走索引，再看 rows 是否远大于实际返回行数，最后关注 Extra 有没有 filesort 和 temporary。如果没走索引，排查六大索引失效场景；如果走了索引但 rows 很大，考虑索引区分度是否够；如果出现 filesort，考虑给排序列加索引或改写 SQL。"

Q6: 怎么保持数据一致？

graph TD A["数据一致性保证"] --> B["事务层面"] A --> C["并发层面"] A --> D["主从层面"] B --> B1["ACID 四大特性"] B --> B2["undo log（原子性/回滚）"] B --> B3["redo log（持久性/崩溃恢复）"] C --> C1["四种隔离级别"] C --> C2["MVCC 多版本并发控制"] C --> C3["行锁/间隙锁/表锁"] D --> D1["binlog + redo log 两阶段提交"] D --> D2["半同步复制"] D --> D3["主从一致性校验（pt-table-checksum）"]
▶ 展开详解
三个层面的一致性保障机制： 1. 事务层面——单机数据一致性：
  • 原子性（Atomicity）：通过 undo log 保证事务要么全部执行、要么全部回滚。undo log 记录修改前的旧数据，回滚时逆向操作恢复数据。
  • 持久性（Durability）：通过 redo log（WAL 机制）保证事务提交后数据不丢失。即使系统崩溃，重启后通过 redo log 重放恢复已提交但未刷盘的数据。
  • 一致性（Consistency）：由原子性+隔离性+持久性共同保证，加上数据库层面的约束（主键、外键、NOT NULL、CHECK）。
2. 并发层面——多事务数据一致性：
  • 隔离性（Isolation）：通过 MVCC + 锁实现。MVCC 负责快照读（普通 SELECT 读历史版本），Next-Key Lock 负责当前读（SELECT FOR UPDATE/UPDATE/DELETE 锁住范围防幻读）。
  • 四种隔离级别：READ UNCOMMITTED（脏读）→ READ COMMITTED（不可重复读）→ REPEATABLE READ（InnoDB 默认，MVCC 基本解决幻读）→ SERIALIZABLE（完全串行）。
  • 银行场景特别提示：金融核心系统（如账务、交易）通常选择 RR 或更高隔离级别，配合乐观锁（version 字段）或悲观锁（SELECT FOR UPDATE）保证并发安全。
3. 主从层面——分布式数据一致性：
  • 两阶段提交：保证 binlog 和 redo log 的一致性——redo prepare → 写 binlog → redo commit。如果 binlog 写完但 redo 未 commit 时崩溃，恢复时会检查 binlog 完整性并重新提交。
  • 半同步复制：主库提交事务时等待至少一个从库确认收到 binlog 后才返回客户端成功。相比异步复制（主库不等待从库），减少了主从切换时的数据丢失风险。
  • 主从一致性校验：使用 pt-table-checksum（Percona Toolkit）定期校验主从数据是否一致，发现不一致用 pt-table-sync 修复。
面试加分句："数据一致性是一个多层面的保证——事务 ACID 保证单机一致、MVCC+锁保证并发一致、两阶段提交+半同步复制保证主从一致。不存在万能的一致性方案，需要根据业务场景在一致性和性能之间权衡。"

Q7: Session 共享问题怎么实现？

▶ 展开详解
为什么需要 Session 共享？

单体应用部署多台机器 + Nginx 负载均衡后，用户的两次请求可能被分发到不同服务器。如果 Session 存储在服务器本地（Tomcat 内存），第二次请求到了第二台服务器上拿不到第一次登录时存的 Session，用户就被判定为未登录。这就是 Session 不共享的问题。

四种方案详细对比： 方案① — Redis 集中存储（最推荐）：
  • 原理：所有服务器共享同一个 Redis，Session 统一读写 Redis
  • 实现：Spring Session + spring-session-data-redis，引入依赖后自动接管 HttpSession，对业务代码透明
  • 优点：高性能（Redis 内存操作）、支持分布式扩展、Session 独立于应用服务器（服务器重启不丢 Session）
  • 缺点：Redis 变成单点（需要 Redis 哨兵/集群保证高可用）、网络延迟（同机房-1ms 可忽略）
  • 配置关键：spring.session.store-type=redis + Session 过期时间合理设置
方案② — Session 复制（Tomcat 内置）：
  • 原理：Tomcat 的 Session 复制机制，一台服务器创建/修改 Session 后广播给集群中所有其他服务器
  • 优点：内置支持，不需要额外中间件
  • 缺点：只适合小集群（3-5 台）——随着服务器数量增加，广播量呈 N² 增长，网络开销巨大；所有服务器都存全量 Session，内存浪费
方案③ — Cookie 存储：
  • 原理：Session 数据加密后存在客户端的 Cookie 中，每次请求自动携带
  • 优点：服务端完全无状态，架构最简单
  • 缺点：Cookie 有 4KB 大小限制（只适合存少量信息）；数据暴露在客户端有安全风险（需要加密+签名防篡改）；每次请求都携带增大网络开销
方案④ — 统一 Session 服务：
  • 原理：独立部署一个 Session 微服务，所有应用的登录/鉴权都走这个服务
  • 适合：大型微服务体系，多个独立应用需要单点登录（SSO）
  • 实现：基于 OAuth2.0 / JWT Token + 认证中心
面试话术模板："我们项目用的是 Spring Session + Redis 方案。引入 spring-session-data-redis 后配置 session store type 为 redis 即可，对业务代码完全透明。Redis 我们做了哨兵模式保证高可用，Session 过期时间设为 30 分钟，同时在 Nginx 配置了 ip_hash 策略作为降级方案。"

Q8: 代理模式好处？

graph LR subgraph "静态代理" A["客户端"] --> B["代理类
(编译时生成)"] B --> C["目标类"] B -.->|"增强"| D["日志/权限/事务"] end subgraph "动态代理" E["客户端"] --> F["JDK 动态代理

(基于继承)"] F --> H["目标类(接口实现)"] G --> I["目标类(可无接口)"] end
▶ 展开详解
代理模式的核心价值——"增强而不侵入"：

代理模式在不修改原始类代码的前提下，为其增加额外的功能（日志记录、权限校验、事务管理、性能监控等）。这符合开闭原则（对扩展开放，对修改关闭），是 AOP（面向切面编程）的思想基础。

三种代理方式详解： 1. 静态代理：
  • 编译时手动创建代理类，代理类和目标类实现同一接口
  • 优点：实现简单直观，编译期确定代理关系，运行效率高
  • 缺点：每个需要被代理的类都要手动写一个代理类 → 代码冗余、难以维护。如果 100 个 Service 都需要加日志，就要写 100 个代理类
2. JDK 动态代理：
  • 运行时通过 Proxy.newProxyInstance() + InvocationHandler 动态生成代理类
  • 核心限制：目标类必须实现接口——代理类实现与目标类相同的接口
  • 原理：在 JVM 运行时动态生成一个实现了指定接口的代理类字节码，所有方法调用通过 InvocationHandler.invoke() 转发
  • Spring AOP 默认使用 JDK 动态代理（当目标类实现了接口时）
3. CGLIB 动态代理：
  • 通过字节码技术（ASM）在运行时为目标类动态生成一个子类（继承目标类）
  • 优点：目标类不需要实现接口，可以对普通类进行代理
  • 限制：final 类和 final 方法不能被代理（因为无法继承/重写）
  • Spring AOP 中当目标类没有实现接口时，自动切换为 CGLIB
面试关键总结：
  • JDK 代理：基于接口，通过反射调用，创建代理对象快但执行略慢（反射有开销）
  • CGLIB 代理：基于继承，通过字节码生成子类，创建代理对象慢（需要生成字节码）但执行快（直接方法调用）
  • Spring Boot 2.x 开始默认使用 CGLIB（因为大多数 Service 类不需要定义接口）
  • 代理模式最大的应用场景是 Spring AOP——@Transactional 事务管理就是通过 AOP 代理实现的
代理模式一句话：在不修改原代码的前提下增强功能（日志/事务/权限），实现解耦。静态代理编译生成、JDK 动态代理基于接口、CGLIB 基于继承。Spring AOP 的 @Transactional 就是通过代理实现的。

Q1: Java 创建多线程的方式？

方式原理特点
① 继承 Thread 类extends Thread，重写 run()简单但单继承限制，不推荐
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("线程运行中");
    }
}
new MyThread().start();
不推荐原因：Java 是单继承，继承了 Thread 就不能继承其他类了，扩展性差。且任务和线程耦合在一起，不符合"任务与执行分离"的设计原则。
② 实现 Runnable 接口implements Runnable，重写 run()推荐。任务与线程分离，可复用
class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("任务执行中");
    }
}
new Thread(new MyTask()).start();
// 或 Lambda
new Thread(() -> System.out.println("执行")).start();
为什么推荐：Runnable 只是任务定义，可以交给多个 Thread、线程池、甚至其他执行器来执行。任务和线程解耦，符合单一职责原则。
③ Callable + FutureTaskimplements Callable，有返回值 + 可抛异常需要返回值时使用
class MyCallable implements Callable<String> {
    @Override
    public String call() throws Exception {
        Thread.sleep(1000);
        return "执行结果";
    }
}
FutureTask<String> ft = new FutureTask<>(new MyCallable());
new Thread(ft).start();
String result = ft.get();  // 阻塞等待获取结果
Callable vs Runnable 的关键区别：
  • Runnable.run() 无返回值，不抛受检异常
  • Callable.call() 有返回值（泛型），可以抛 Exception
  • FutureTask 同时实现了 Runnable 和 Future 接口，起到桥接作用——既能被 Thread 执行，又能通过 get() 获取返回值
④ 线程池创建通过 Executors 或 ThreadPoolExecutor最佳实践，生产环境必须用
// 不推荐：Executors（阿里规约禁止）
ExecutorService pool = Executors.newFixedThreadPool(10);

// 推荐：手动 new ThreadPoolExecutor
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    5,                    // corePoolSize
    10,                   // maximumPoolSize
    60L, TimeUnit.SECONDS,// keepAliveTime
    new LinkedBlockingQueue<>(100),  // workQueue
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
pool.execute(() -> System.out.println("任务执行"));
为什么不推荐 Executors？
  • newFixedThreadPool / newSingleThreadExecutor 使用无界 LinkedBlockingQueue → 任务堆积可能导致 OOM
  • newCachedThreadPool 允许创建无限线程 → 高并发下大量线程导致 OOM
  • 阿里规约强制要求用 ThreadPoolExecutor 显式设置参数
面试黄金回答："实际开发中只用线程池（Runnable + ThreadPoolExecutor），不推荐直接 new Thread。继承 Thread 和实现 Runnable 都要手动创建销毁线程，而线程池复用线程、控制并发数、管理任务队列，是生产环境的最佳实践。"

Q2: 线程池原理、核心配置赋值？

线程池 7 大参数（ThreadPoolExecutor 构造器）：

IO 密集型：2×CPU 核数

▶ 展开详解
线程池运行流程（四个阶段）：

① 提交任务 → ② 当前线程数 < corePoolSize → 创建新线程执行（即使有空闲线程）
③ 当前线程数 >= corePoolSize → 任务放入 workQueue 等待
④ workQueue 满了 + 当前线程数 < maximumPoolSize → 创建非核心线程执行
⑤ workQueue 满了 + 当前线程数 = maximumPoolSize → 执行拒绝策略

核心线程数怎么设置（面试高频）：
  • CPU 密集型任务（计算为主，极少 IO）：coreSize = CPU 核心数 + 1。因为 CPU 密集型任务几乎不阻塞，线程数超过 CPU 核心数会导致频繁上下文切换，反而降低性能。+1 是为了在某线程因缺页中断等原因暂停时，有一个备用线程顶上。
  • IO 密集型任务（网络/磁盘 IO 多，线程大部分时间在等待）：coreSize = 2 × CPU 核心数（或更高，取决于 IO 等待时间占比）。因为线程大部分时间在阻塞等待 IO，CPU 是空闲的，可以多开几个线程在 CPU 空闲时处理其他任务。
  • 更精确的公式：核心数 = CPU 核心数 × (1 + IO 等待时间 / CPU 计算时间)。例如线程 50% 时间在等 IO → 核心数 = CPU 核数 × (1 + 0.5/0.5) = 2 × CPU 核数。
线程池与银行场景的结合：

银行系统中，线程池常用于批量任务处理（如日终跑批、批量短信通知）、异步日志写入、定时任务调度等。合理配置线程池可以避免任务堆积导致的服务不可用——例如支付接口调用超时后，如果线程池无界可能所有线程都被卡住。因此银行系统尤其需要注意：① 线程池配合超时熔断；② workQueue 必须设上限；③ 监控线程池状态（activeCount、queueSize、completedTaskCount）。

阿里规约中的线程池规范：

① 不允许使用 Executors 创建线程池（OOM 风险）→ 用 ThreadPoolExecutor
② 线程池不允许使用无界队列（LinkedBlockingQueue 不设容量 = Integer.MAX_VALUE ≈ 无界）
③ 线程池需要提供 ThreadFactory 给线程命名（方便排查问题）
④ 任务执行需要加 try-catch，防止异常导致线程退出

Q3: 线程池拒绝策略？

策略行为适用场景
AbortPolicy
（默认）		直接抛 RejectedExecutionException需要感知任务被拒绝的场景
特点：最直接的拒绝方式——任务被拒后立即抛异常，调用方可以捕获并处理（如记录日志、告警、降级）。
缺点：如果调用方没捕获异常，会导致上层业务异常（如 HTTP 请求返回 500）。
适用：任务必须执行、不能丢失的场景。异常信息明确，便于排查问题。
DiscardPolicy直接丢弃任务，不抛异常允许丢失的次要任务（如非关键日志）
最大风险：任务被静默丢弃——调用方不知道任务没被执行，也没有任何日志或异常提示。在银行场景中非常危险——如果支付通知被丢弃，用户和系统都不知情。
使用建议：如果非要用，至少加日志记录（自定义一个包装策略，丢弃前 log.warn）。
DiscardOldestPolicy丢弃队列中最旧的任务，重试当前任务只关心最新任务，可丢弃历史
特点："喜新厌旧"——抛弃队首（最早等待的任务），然后把当前任务加入队尾。
风险：被丢弃的旧任务可能是关键任务（如先提交的支付请求被丢了），所以适用场景非常有限。一般只用于"最新数据最重要"的场景（如实时行情推送，旧行情没意义了）。
CallerRunsPolicy
（推荐）		由提交任务的线程自己执行通用场景，银行系统推荐
工作原理：线程池满了 → 不拒绝、不丢弃 → 由调用方线程（如 main 线程或 Tomcat 线程）自己执行这个任务。相当于"老板（调用方）自己干活"——牺牲了调用方的处理能力，但保证了任务不丢失。
优点：① 任务不会丢失；② 调用方线程执行任务时不会接收新任务，相当于一种天然的流量控制和背压机制；③ 系统不会因为任务堆积而完全崩溃。
缺点：调用方线程被占用期间无法处理其他请求，可能影响整体吞吐量。
银行场景推荐理由：金融系统中数据的完整性高于吞吐量——"慢一点可以，但丢了不行"。CallerRunsPolicy 在系统过载时提供了一种优雅降级方式。
▶ 展开详解
如何选择拒绝策略——银行系统视角：

① 核心交易类（支付、转账）：AbortPolicy + 异常监控告警。任务拒绝就是系统容量问题了，必须人工介入。

② 普通异步任务（短信通知、日志写入）：CallerRunsPolicy。允许调用方线程自己执行，牺牲一点响应时间但保证任务不丢。

③ 非关键实时数据（监控指标采集、非核心统计）：DiscardOldestPolicy。如果系统已经过载，丢弃旧数据采集任务，保留最新的。

④ DiscardPolicy：银行系统中几乎不使用——任何静默丢弃的行为都有合规风险。

自定义拒绝策略：

可以实现 RejectedExecutionHandler 接口，自定义拒绝逻辑。例如：

  • 拒绝时写入数据库持久化，后续定时任务补偿
  • 拒绝时写入 MQ，由独立消费者处理
  • 拒绝时触发告警（企业微信/钉钉/邮件通知运维）
面试记忆口诀："Abort 抛异常，Discard 静默丢，DiscardOldest 丢旧取新，CallerRuns 自己干。"
银行推荐：CallerRunsPolicy >> AbortPolicy > DiscardOldestPolicy > DiscardPolicy（基本不用）

▶ 展开详解
五大内存区域逐一详解： ① 程序计数器（Program Counter Register）：
  • 存储内容：当前线程正在执行的字节码指令的行号
  • 特点：JVM 规范中唯一没有规定 OOM 的区域
  • 作用：多线程切换时，每个线程需要记住自己执行到哪了——PC 寄存器就是记录这个位置的。线程恢复执行时从中断位置继续
② 虚拟机栈（JVM Stack）：
  • 存储内容：栈帧（Stack Frame），每个方法调用对应一个栈帧
  • 栈帧包含：局部变量表（基本类型 + 对象引用）、操作数栈、动态链接、方法返回地址
  • 常见异常：StackOverflowError（递归过深导致栈深度超过限制）、OutOfMemoryError（无法申请足够内存扩展栈）
③ 本地方法栈（Native Method Stack）：
  • 与虚拟机栈类似，但服务于 native 方法（用 C/C++ 写的本地方法）
  • HotSpot 将本地方法栈和虚拟机栈合并，不单独区分
④ Java 堆（Heap）——最重要：
  • 存储内容：所有对象实例和数组（基本类型在栈上，引用类型对象在堆上）
  • 分区：新生代（Eden + Survivor S0 + Survivor S1） + 老年代（Old Gen）
  • GC 主战场：Minor GC 回收新生代，Major/Full GC 回收老年代和整个堆
  • JDK 8+ 的改进：字符串常量池从方法区移到了堆中
⑤ 元空间（MetaSpace，JDK 8+）：
  • 替代了 JDK 7 及之前的方法区（永久代 PermGen）
  • 存储内容：类的元数据（类名、方法信息、字段信息）、运行时常量池、静态变量
  • 与永久代的关键区别：元空间使用本地内存（Native Memory），不再是 JVM 堆的一部分——不再受 -XX:MaxPermSize 限制，而是受系统可用内存限制。可以通过 -XX:MaxMetaspaceSize 限制
  • Why 移出堆？永久代大小难以预测（不同项目的类数量和字符串常量差异大），经常导致 OOM: PermGen space。移出后用本地内存更灵活
银行面试加分点——OOM 与内存区域的关系：

Heap space OOM → 堆内存不足（对象太多/内存泄漏）→ 排查大对象
MetaSpace OOM → 动态生成类太多（CGLIB 代理类、反射）→ 排查类加载
StackOverflow → 递归太深或栈太小 → 排查递归逻辑或调大 -Xss
Unable to create new native thread → 线程数超系统限制 → 减少线程或调大系统 ulimit

Q2: 自动装箱拆箱？

操作方向底层方法示例
装箱基本类型 → 包装类Integer.valueOf(int)Integer i = 100;
valueOf 的缓存机制（-128 ~ 127）： 
public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];
    return new Integer(i);
}

Integer 内部维护了一个 IntegerCache，默认缓存 -128 到 127 的所有 Integer 对象。在这个范围内的装箱操作不会创建新对象，而是直接从缓存中取。这就是为什么：

Integer a = 100, b = 100;
System.out.println(a == b);  // true（同一个缓存对象）
Integer c = 200, d = 200;
System.out.println(c == d);  // false（超出缓存范围，new 了两个不同对象）
拆箱包装类 → 基本类型Integer.intValue()int j = i;
拆箱的 NPE 陷阱： 
Integer i = null;
int j = i;  // NullPointerException！拆箱时调用 i.intValue()，但 i 是 null

银行开发中常见场景：数据库查询返回的字段可能为 NULL，如果用 Integer 接收后在逻辑中做拆箱操作（如 > < 比较、加减运算），必须判空。阿里规约要求所有 POJO 属性用包装类型、RPC 返回值用包装类型，但使用前要判空。

▶ 展开详解
常见包装类的缓存范围：
  • Byte / Short / Integer / Long：缓存 -128 ~ 127
  • Character：缓存 0 ~ 127（ASCII 字符）
  • Boolean：缓存 TRUE 和 FALSE 两个常量
  • Float / Double：没有缓存（浮点数取值无限，缓存意义不大）
面试陷阱题——三目运算符中的拆箱： 
Integer i = null;
Object o = true ? i : 0;  // 三目运算符要求两个分支类型一致
// 这里 0 是 int，i 被自动拆箱为 int → NPE！

三目运算符要求返回类型一致——如果一个是基本类型（0 是 int），另一个的包装类型会被自动拆箱。如果包装类型恰为 null → NPE。解决方案是让两个分支类型一致：
Object o = true ? i : Integer.valueOf(0);

包装类的实际使用场景：
  • 泛型：List<Integer> 不能用 List<int>（泛型不支持基本类型）
  • 数据库 NULL 值：POJO 中的字段用 Integer/Long 等包装类型，因为数据库字段可能为 NULL
  • RPC 返回值：远程调用后字段可能为 NULL
  • 反射调用：反射中获取和设置字段值只能用 Object，无法用基本类型

Q3: Spring 容器启动流程？

graph TD A["加载配置 XML/注解"] --> B["扫描 BeanDefinition"] B --> C["实例化 Bean 反射创建"] C --> D["依赖注入 Autowired"] D --> E["初始化回调 PostConstruct"] E --> F["容器就绪"]
▶ 展开详解
Spring 容器启动的六大步骤详解： ① 加载配置：

读取配置文件（XML、注解 @Configuration、或 properties/yaml）。对于 Spring Boot，从 application.yml 和 @SpringBootApplication 注解开始解析。这个阶段确定了要扫描哪些包、加载哪些配置类。

② 扫描 BeanDefinition：

Spring 通过 @ComponentScan 扫描指定包路径下的所有类，找到带 @Component、@Service、@Repository、@Controller 等注解的类，以及 @Bean 方法返回的对象。每个 Bean 被包装成一个 BeanDefinition 对象，包含：Bean 的类名、作用域（singleton/prototype）、是否懒加载、依赖关系、初始化/销毁方法等元信息。所有 BeanDefinition 注册到 BeanDefinitionRegistry（一个 ConcurrentHashMap）中。

③ 实例化 Bean（通过反射）：

遍历 BeanDefinition 列表，通过反射（Class.newInstance() 或 Constructor.newInstance()）创建 Bean 实例对象。此时对象属性还未赋值，只是一个"空壳"对象。

这个阶段只是创建对象，不做依赖注入。Spring 会处理循环依赖——通过三级缓存（singletonFactories → earlySingletonObjects → singletonObjects）来解决构造器循环依赖以外的情况。

④ 依赖注入（DI）：

给实例化后的 Bean 填充属性——解析 @Autowired、@Value、@Resource 等注解，从容器中找到对应的依赖 Bean 并注入。注入方式优先级：构造器注入 > setter 注入 > 字段注入（反射直接设置字段）。Spring 官方推荐构造器注入（不可变、不依赖反射、方便测试）。

⑤ 初始化回调：

Bean 属性填充完成后，按以下顺序执行初始化回调：
① @PostConstruct 注解的方法
② InitializingBean 接口的 afterPropertiesSet() 方法
③ @Bean(initMethod = "xxx") 指定的方法

这个阶段可以做：数据预热（把数据库数据加载到缓存）、资源初始化（连接池预热）、启动后自检等。

⑥ 容器就绪：

Spring 容器完全初始化完成，发布 ContextRefreshedEvent 事件。可以通过监听这个事件（ApplicationListener 或 @EventListener）来执行一些"所有 Bean 都就绪后"的操作。

面试加分——三级缓存解决循环依赖：

一级缓存 singletonObjects（成品池）→ 二级缓存 earlySingletonObjects（半成品池）→ 三级缓存 singletonFactories（半成品工厂）。A 依赖 B、B 依赖 A 时：创建 A → 发现需要 B → 把 A 的半成品工厂放入三级缓存 → 创建 B → 发现需要 A → 从三级缓存获取 A 的半成品 → B 注入 A 半成品 → B 初始化完毕 → A 注入成品 B → A 初始化完毕。三级缓存是 Spring 解决循环依赖的核心设计。

Q1: 查看系统资源使用？

资源类型命令关键信息
整体负载top / htopCPU 使用率、内存使用率、负载均值（load average）、进程列表
top 输出关键指标解读：
  • load average：三个值分别是 1 分钟、5 分钟、15 分钟的 CPU 平均负载（正在运行 + 等待运行的进程数）。如果 1 分钟值突然 > CPU 核心数，说明系统正在过载。
  • us（user）：用户态 CPU 占比——应用代码消耗的 CPU。如果 us 持续很高，说明应用计算密集。
  • sy（system）：内核态 CPU 占比——系统调用消耗。如果 sy 很高，可能是频繁的系统调用（如频繁 IO、大量线程切换）。
  • wa（iowait）：等待 IO 的 CPU 占比——如果 wa 很高，说明磁盘 IO 是瓶颈。
  • id（idle）：CPU 空闲率。
  • RES（resident memory）：进程实际占用的物理内存（不包含 swap）。
内存free -h总内存（total）、已用（used）、空闲（free）、缓存（buff/cache）、可用（available）
free -h 输出的关键理解：
  • free 和 available 的区别：free 是完全没有使用的内存；available 是 free + 可以被回收的缓冲/缓存（buff/cache）。判断内存是否够用，看 available 而非 free。Linux 会积极用空闲内存做文件缓存（buff/cache），这部分在内存紧张时可以释放。
  • swap：如果 swap 的 used 值持续增长，说明物理内存不够了，系统开始用磁盘当内存——性能会急剧下降。
磁盘df -h文件系统、总大小、已用、可用、挂载点

  • iostat -x 1：查看磁盘 IO 详情——await（平均响应时间）、%util（磁盘繁忙度）。util 持续 100% 说明磁盘 IO 瓶颈

Q2: 进程和线程的区别？

维度进程（Process）线程（Thread）
定义资源分配的最小单位CPU 调度的最小单位
为什么是这样定义的？操作系统给每个进程分配独立的虚拟地址空间、文件描述符表、信号处理等资源——进程是"持有资源的实体"。而 CPU 是按线程粒度来调度的——一个进程可以有多个线程，CPU 轮流执行这些线程。因此"资源分配看进程，CPU 调度看线程"。
内存每个进程拥有独立的虚拟地址空间同一进程内的线程共享进程的地址空间（堆、全局变量、文件句柄）
线程共享 vs 独享：
  • 线程共享：堆、全局变量、静态变量、文件描述符、信号处理器、代码段
  • 线程独享：栈（每个线程有自己的调用栈）、程序计数器、寄存器、线程局部存储（TLS）
通信方式IPC（管道、消息队列、共享内存、Socket）直接读写共享内存（堆中对象），但需注意线程安全
进程间通信（IPC）的几种方式：
  • 管道（Pipe）：半双工，父子进程间使用（如 Linux 命令的 | 操作符）
  • 消息队列：消息的链表，存放在内核中，独立于进程生命周期
  • 共享内存：最快的 IPC 方式——直接映射同一块物理内存到多个进程的虚拟地址空间。但需要配合信号量等同步机制
  • Socket：不仅可以用于网络通信，也可以用于本机进程间通信（Unix Domain Socket）
  • 信号（Signal）：异步通知机制，如 kill -9 发送 SIGKILL
创建/销毁开销大（需分配独立地址空间 + PCB + 页表）小（只需分配栈 + TCB + 寄存器）
开销对比分析：创建进程需要复制父进程的地址空间（fork 时的写时拷贝优化可以减少复制，但仍需要建立页表映射），分配进程控制块（PCB），开销通常是毫秒级。创建线程只需要分配栈空间（通常 1MB）+ 线程控制块（TCB），开销是微秒级。在 Linux 中，线程实际上是轻量级进程（LWP），通过 clone() 系统调用创建，可以指定共享哪些资源。
隔离性强（一个进程崩溃不影响其他进程）弱（一个线程崩溃可能导致整个进程退出）
银行系统的选择：对于银行核心系统，隔离性非常重要。所以微服务通常部署为独立的 JVM 进程——一个服务 OOM 不会拖垮另一个。但在单个服务内部，使用线程池来处理并发请求以提高吞吐量。这是"进程做隔离，线程做并发"的经典实践。
一句话总结：进程是资源分配的最小单位（独享内存），线程是 CPU 调度的最小单位（共享进程资源、创建开销小）。
面试金句："进程之间就像不同家庭的房子——各住各的，互不影响；线程就像同一家里的不同成员——共享客厅厨房但有自己独立的卧室。"

Q3: 服务注册 & 发现？

为什么需要服务注册与发现？

单体应用时代，服务地址是固定的（写死在配置文件里）。微服务时代，服务实例数量是动态变化的——弹性伸缩会增加实例、容器重启会换 IP、灰度发布会有不同版本。如果还是硬编码服务地址，每次变更都要改配置重启，不可维护。

服务注册与发现的核心思想：服务提供者把自己的地址告诉注册中心（注册），服务消费者从注册中心获取可用地址列表（发现），不再硬编码地址。

健康检查的两种模式：
  • 客户端心跳（Eureka 模式）：服务提供者定时向注册中心发送心跳（如每 30s 一次）。注册中心 N 秒未收到心跳就剔除该实例。这种模式对注册中心压力小，但可能误判（心跳断了但服务还能用）。
  • 服务端探测（Consul/Nacos 模式）：注册中心主动探测服务提供者的健康端点（如 GET /actuator/health）。更准确（能判断服务是否真的健康），但对注册中心压力大。
服务发现的两种模式：
  • 客户端发现（Client-Side Discovery）：消费者从注册中心拉取服务列表，自己决定调用哪个实例（通过 Ribbon/LoadBalancer 做客户端负载均衡）。Spring Cloud + Nacos 就是这种模式。
  • 服务端发现（Server-Side Discovery）：消费者通过中间代理（如 Nginx/K8s Service）调用服务提供者，代理层负责从注册中心获取列表并转发。K8s 的 Service + kube-proxy 就是这种模式。
银行场景的特殊性——灰度发布与注册中心：

银行系统经常需要灰度发布（比如先在 10% 的实例上部署新版本，观察没问题再全量）。Nacos 支持通过元数据标签（如 version=v2）将服务实例分组，消费者根据标签选择调用哪些实例，配合网关实现流量染色和灰度路由。

Q1: TCP 位于哪一层？

核心答案：TCP 位于 OSI 七层模型的传输层（第 4 层），也对应 TCP/IP 四层模型的传输层。传输层负责端到端的可靠数据传输，TCP 通过序列号/确认号、滑动窗口、拥塞控制等机制提供可靠的、面向连接的数据传输服务。

为什么需要分层？

分层设计的核心思想——每一层只关心自己的职责，上层不需要知道下层怎么实现。例如浏览器（应用层）只负责 HTTP 协议的打包和解包，不需要关心数据怎么可靠地传过去（那是 TCP 的事），更不需要关心数据走哪条路由（那是 IP 的事）。这样每层独立演进：HTTP/1.1 升级到 HTTP/2，TCP 层完全不用改。

银行系统对 500 的处理：① 必须记录错误堆栈到日志；② 返回给前端的信息不能包含堆栈或敏感信息（如 SQL 语句、IP 地址）→ 统一返回"系统繁忙，请稍后再试"；③ 触发告警通知运维；④ 做好全局异常处理（@ControllerAdvice + @ExceptionHandler）。

Q3: TCP 和 UDP 区别？

维度TCPUDP
连接面向连接（三次握手建立、四次挥手释放）无连接（直接发送，无需建立连接）
为什么 TCP 需要三次握手？"三次"是因为要确认双方的收发能力都正常——一次握手客户端确认服务器可达，二次握手服务器确认客户端可收到自己消息，三次握手客户端告诉服务器"我知道你能收到我的消息了"。两次握手无法防止失效的连接请求到达服务器导致服务器空等（历史连接问题）。
可靠性可靠（序列号+确认应答+超时重传+校验和）不可靠（发了就不管，丢包不重传）
TCP 如何保证可靠？四个机制：
① 序列号与确认应答（Seq + Ack）——每个字节都有编号，接收方确认收到的字节
② 超时重传——发送方在一定时间内没收到 ACK 就重传
③ 流量控制（滑动窗口）——接收方告诉发送方自己还能收多少，防止发送过快撑爆接收方
④ 拥塞控制（慢启动、拥塞避免、快重传、快恢复）——感知网络状况，防止过多数据注入网络
速度较慢（握手开销+确认机制+拥塞控制限速）较快（无连接建立、无确认等待、无拥塞控制）
UDP 为什么快？UDP 的包头只有 8 字节（TCP 20 字节），协议栈处理开销小。UDP 不需要维护连接状态（不需要 TCB），不需要保存发送窗口和接收窗口，不需要计时器管理。应用程序有数据直接往 socket 写，内核直接打包发送。但代价是没有任何可靠性保证。
首部开销20 字节（最少）8 字节
适用场景文件传输、网页浏览、邮件、远程登录——需要数据完整性的场景直播、视频通话、在线游戏、DNS 查询——实时性优先、允许少量丢包的场景
银行系统中 TCP/UDP 的使用：
  • 银行几乎所有的业务通信都用 TCP（HTTP/HTTPS、数据库连接、MQ 消息投递、RPC 调用），因为金融数据不能丢。
  • UDP 在银行系统中的使用非常有限——一般是 DNS 查询（域名解析，默认用 UDP）、NTP 时间同步（银行对时间戳要求极高）等基础协议。
  • 偶尔有极低延迟交易场景（如量化交易行情推送）会用 UDP 组播，但要配合应用层做丢包补偿。
▶ 展开详解
TCP 三次握手详解： 
Client → Server: SYN=1, seq=x                    （"在吗？我想建立连接"）
Server → Client: SYN=1, ACK=1, seq=y, ack=x+1    （"在的，我准备好了"）
Client → Server: ACK=1, seq=x+1, ack=y+1          （"好的，开始传数据吧"）
  • 第一次握手：只是 Client 证明自己发送能力正常
  • 第二次握手：Server 证明自己收发正常 + Client 收到后证明自己收发都正常
  • 第三次握手：Server 收到后证明 Client 能收到自己的消息（双方收发能力都验证完毕）
TCP 四次挥手详解： 
Client → Server: FIN=1, seq=u                （"我说完了，准备关"）
Server → Client: ACK=1, seq=v, ack=u+1       （"知道了，等我收完"）
Server → Client: FIN=1, ACK=1, seq=w, ack=u+1（"我也说完了，可以关了"）
Client → Server: ACK=1, seq=u+1, ack=w+1     （"好的，再见"）
Client 进入 TIME_WAIT 状态，等 2MSL 后关闭

为什么是四次而不是三次？因为 TCP 是全双工的——双方都可以独立地发送和接收数据。Client 说"我说完了"只关闭了 Client→Server 的方向，Server 可能还有数据要发。所以 Server 先 ACK 表示"知道了"，等自己也发完了再发 FIN。这就是"两次控制消息、四次交互"。

TCP 拥塞控制的四个算法：
  • 慢启动（Slow Start）：连接建立后，拥塞窗口 cwnd 从 1 开始，每收到一个 ACK 翻倍（指数增长），直到达到慢启动阈值 ssthresh
  • 拥塞避免（Congestion Avoidance）：cwnd >= ssthresh 后，线性增长（每个 RTT +1），慢慢探测网络容量
  • 快重传（Fast Retransmit）：收到 3 个重复 ACK 时（说明中间有一个包丢了），不等超时立即重传
  • 快恢复（Fast Recovery）：快重传后不进入慢启动（不把 cwnd 降到 1），而是把 ssthresh 和 cwnd 都设为原来的一半，进入拥塞避免
面试加分——TCP 粘包/拆包问题：

TCP 是流式协议，没有消息边界。发送方两次 write("Hello") 和 write("World") 可能被合并为一个 TCP 段发送（粘包），也可能被拆成多个段发送（拆包）。接收方需要自己定义消息边界——常用方案：① 固定长度消息；② 特殊分隔符（如换行符）；③ 消息头 + 消息体（头中定义体长度，如 Netty 的 LengthFieldBasedFrameDecoder）。

Q1: 乐观锁、悲观锁、分布式锁三者区别？

先锁住再操作		"大概率没人跟我抢"

三种锁的设计哲学对比：
  • 悲观锁——预防为主：认为冲突是常态，每次读写都加锁。适合写多读少、冲突概率高的场景（如银行账户扣款）。
  • 乐观锁——事后检查：认为冲突是偶然，不加锁直接操作，提交时检查是否有人改过。适合读多写少、冲突概率低的场景（如商品信息编辑）。
  • 分布式锁——跨进程协调：悲观/乐观锁只能解决单机（单个数据库/单个 JVM）的并发问题，分布式锁解决多个独立服务实例之间的互斥问题。
实现方式数据库：SELECT ... FOR UPDATE
Java：synchronized / ReentrantLock		数据库：version 字段 + UPDATE WHERE version=old
Java：AtomicInteger(CAS)		Redis：SET NX EX + Lua
ZooKeeper：临时顺序节点 + Watch
三种实现的具体写法： 
-- 悲观锁：SELECT ... FOR UPDATE
BEGIN;
SELECT stock FROM product WHERE id=1 FOR UPDATE;  -- 锁住这一行
UPDATE product SET stock=stock-1 WHERE id=1;
COMMIT;

-- 乐观锁：版本号
UPDATE product SET stock=stock-1, version=version+1
WHERE id=1 AND version=#{oldVersion};  -- 版本号对不上就失败

-- 分布式锁：Redis
SET lock:product:1 uuid NX EX 30  -- 获取锁
-- 执行业务...
-- Lua 脚本释放锁
适用场景单机/单库，写多读少
冲突概率高（账户余额、库存扣减）		单机/单库，读多写少
冲突概率低（用户信息编辑）		分布式微服务环境
跨 JVM/跨机器的互斥
银行场景中的选择策略：
  • 账户扣款（扣余额）：用悲观锁（SELECT FOR UPDATE），因为并发扣款概率高且数据绝对不能错。
  • 用户修改手机号：用乐观锁（version），因为很少有两个操作同时修改同一用户的手机号。
  • 定时任务（多实例只执行一次）：用分布式锁（Redis SET NX），保证多个微服务实例中只有一个执行。
  • 分布式场景下的库存扣减：悲观锁 + 分布式锁结合——先用 Redis 分布式锁拿到操作权，再在 DB 中用悲观锁完成扣减。
性能低（阻塞等待，锁持有期间其他事务等待）高（无锁操作，提交时只需一次版本校验）中（网络开销 + Lua执行，但比 DB 锁快）
一句话总结：悲观锁"先锁再用"（防冲突）、乐观锁"用了再查"（事后校验）、分布式锁"跨机器协调"（外部中间件统一管控）。三种锁解决的是不同层面的并发问题，实际项目中经常组合使用。
面试金句："悲观乐观是思想，分布锁是工具。悲观锁靠数据库，乐观锁靠版本号，分布式锁靠 Redis/ZK。"

Q3: 什么是多级缓存？

graph LR A["用户请求"] --> B["L1 本地缓存 Caffeine"] B --> C["L2 分布式缓存 Redis"] C --> D["L3 数据库 MySQL"] B -.-> E["微秒级响应"] C -.-> F["多实例共享"] D -.-> G["持久化兜底"]
▶ 展开详解
为什么要多级缓存？单级缓存不好吗？

单靠 Redis：每个请求都要走网络（即使是内网也约 1ms），高并发下 Redis 成为瓶颈。单靠本地缓存：多实例数据不一致，且 JVM 堆容量有限（存太多会导致 GC 频繁）。多级缓存结合了两者的优势——大部分请求在本地命中了，没命中的走 Redis，两个都 miss 才打到 DB。

L1 本地缓存的选型：
  • Caffeine（推荐）：Google Guava Cache 的继任者，使用 W-TinyLFU 算法（Window TinyLFU），结合了 LFU 的高命中率和 LRU 的时效性。性能远超 Guava Cache。
  • 配置示例：maximumSize=10000, expireAfterWrite=60s, refreshAfterWrite=30s——最多 1 万条、60 秒过期、30 秒后异步刷新。
多级缓存的一致性处理：

更新数据时：① 更新 DB → ② 删除 Redis → ③ 通知所有实例删除本地缓存（通过 MQ/Redis Pub/Sub 广播）。Caffeine 收到通知后 invalidate 对应 key，下次读时重新从 Redis 加载。

银行场景的实际价值：

如银行的热门活动页面配置（利率、活动规则等）——访问量极大、数据变化不频繁。用本地缓存（Caffeine 5 分钟）+ Redis（30 分钟），大部分流量在本地消化，Redis 压力降低 90%+，即使 Redis 挂了本地缓存还能撑一段时间。

多级缓存 = 本地缓存(Caffeine) + 分布式缓存(Redis) + 数据库(MySQL)
L1（本地）命中率最高、RT 最低；L2（Redis）多实例共享；L3（DB）兜底。更新时依次删除各级缓存保证最终一致。

（二）算法手撕

Q1: 判断回文字符串（两种思路）

思路一：反转对比 
public boolean isPalindrome(String s) {
    // 预处理：只保留字母数字，转小写
    String filtered = s.replaceAll("[^a-zA-Z0-9]", "").toLowerCase();
    String reversed = new StringBuilder(filtered).reverse().toString();
    return filtered.equals(reversed);
}
思路二：双指针（推荐，O(1) 额外空间） 
public boolean isPalindrome(String s) {
    int left = 0, right = s.length() - 1;
    while (left < right) {
        // 跳过非字母数字字符
        while (left < right && !Character.isLetterOrDigit(s.charAt(left)))
            left++;
        while (left < right && !Character.isLetterOrDigit(s.charAt(right)))
            right--;
        // 比较字符（忽略大小写）
        if (Character.toLowerCase(s.charAt(left))
                != Character.toLowerCase(s.charAt(right)))
            return false;
        left++;
        right--;
    }
    return true;
}
时间复杂度：O(n)  |  空间复杂度：反转法 O(n)，双指针法 O(1)
▶ 展开详解
面试中如何展示你的思考过程：

"首先我想到最直观的方法——把字符串反转，如果和原串一样就是回文。这是 O(n) 时间 + O(n) 空间。但我马上意识到可以优化空间——用两个指针一个从左、一个从右，同时向中间移动并比较字符，不等就直接返回 false。这是 O(n) 时间 + O(1) 空间。"

加分表现：提到如何处理非字母数字字符（Character.isLetterOrDigit）、大小写忽略（Character.toLowerCase），以及 left < right 的边界条件。面试官通常会接着问"如果字符串很长（1GB）怎么办"——双指针一次只比较两个字符，不需要加载整个串到内存，更适合大文件。反转法需要完整加载 + 反转 + 比较，内存占用大。

Q2: 二叉树层次遍历用什么数据结构实现？

graph TD A["层序遍历 = BFS"] --> B["用队列 Queue 实现"] B --> C["① 根入队"] C --> D["② 出队并访问"] D --> E["③ 左右子节点入队"] E --> F["④ 重复 ②③ 直到队空"]
核心答案：用队列（Queue）实现。

为什么是队列不是栈？栈是 LIFO（后进先出），会导致深度优先（DFS）；队列是 FIFO（先进先出），保证同一层的节点按顺序依次被处理——这正是层序遍历的要求。 代码实现： 
public List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> result = new ArrayList<>();
    if (root == null) return result;

    Queue<TreeNode> queue = new LinkedList<>();
    queue.offer(root);

    while (!queue.isEmpty()) {
        int levelSize = queue.size();  // 当前层的节点数
        List<Integer> level = new ArrayList<>();

        for (int i = 0; i < levelSize; i++) {
            TreeNode node = queue.poll();
            level.add(node.val);
            if (node.left  != null) queue.offer(node.left);
            if (node.right != null) queue.offer(node.right);
        }
        result.add(level);  // 逐层收集结果
    }
    return result;
}
▶ 展开详解
BFS vs DFS 的数据结构选择：
  • BFS（层序/广度优先）→ 队列 Queue：FIFO 保证了"先进入的先处理"。想象一层层扫描——第 1 层全处理完才开始第 2 层，这正是队列的顺序特性。
  • DFS（先序/中序/后序）→ 栈 Stack（或递归，本质是调用栈）：LIFO 保证了"一条路走到黑再回溯"。每次取最近入栈的节点深入探索。
层序遍历的时间/空间复杂度：
  • 时间复杂度：O(n)，每个节点入队出队一次
  • 空间复杂度：O(w)，w 为树的最大宽度（队列中最多存放一整层的节点）。平衡二叉树最底层节点数 ≈ n/2，所以最坏 O(n)。但实际应用中队列大小远小于 n。
扩展——层序遍历的变种：
  • 自底向上遍历（107 题）：结果列表反转即可
  • 锯齿形遍历（103 题）：加一个 flag 控制每层是正序还是逆序加入结果
  • 求二叉树的最大宽度（662 题）：给每个节点编号（i → 左子 2i，右子 2i+1），每层宽度 = 最右编号 - 最左编号 + 1
面试回答模板："二叉树层次遍历用队列（Queue）实现，本质是 BFS。根节点入队，然后循环：出队 → 访问 → 左右子节点入队。关键是 BFS 用队列（FIFO），DFS 用栈（LIFO）或递归——这是数据结构决定算法行为的最经典案例。"

TTL（Time To Live）= 消息存活时间。

消息在队列中超过 TTL 仍未消费 → 自动变为死信消息（Dead Letter Message），进入死信队列（DLQ）。 两种设置方式：
方式粒度示例
队列 TTL队列中所有消息的默认 TTL创建队列时声明 x-message-ttl=60000
消息 TTL单条消息级别的 TTL发送时 setExpiration 或 setTTL

▶ 展开详解
TTL + 死信队列的典型应用场景：
  1. 延时队列：发送消息到一个没有消费者的队列，设 TTL=30分钟。TTL 到期后消息进入死信队列，死信队列的消费者 30 分钟后才收到消息——实现了延时投递。常用于：订单 30 分钟未支付自动取消、定时提醒。
  2. 超时订单处理：下单时发一条 TTL=30分钟的消息到延时队列，消费者收到后检查订单状态，未支付则关闭订单。
  3. 失败重试上限：消息消费失败后重新投递，设置 TTL 限制最大存活时间。超时后进入死信队列，人工介入处理，避免无限重试。
不同 MQ 的 TTL 实现差异：
  • RabbitMQ：原生支持 TTL + DLX（死信交换机）。消息到期后可自动路由到指定交换机。
  • RocketMQ：通过 delayLevel 实现延时消息（1s/5s/10s/30s/1m/2m/...等 18 个级别）。不是任意 TTL 都支持，需要使用预设级别。
  • Kafka：没有原生的消息 TTL。通过 retention.ms 控制消息留存时间——超过时间的消息被删除而非进入死信队列。延时队列需靠业务层实现。
银行场景中的 TTL 应用：

转账超时处理——用户发起跨行转账后，如果 2 小时内未收到对方银行的确认回执，消息进入死信队列触发人工对账流程。TTL 设为 2 小时，死信消费端自动生成异常工单并通知运营人员。

Q1: 如何保证 i++ 操作的原子性？

graph TD A["i++ 为什么不是原子操作？"] --> B["① 读：从内存读 i 的值"] B --> C["② 改：i + 1 计算"] C --> D["③ 写：把新值写回内存"] D --> E["这三步之间可能被其他线程打断！"]
三种保证 i++ 原子性的方案：
方案实现适用场景
① synchronizedsynchronized(lock) { i++; }简单、可靠，适合低竞争
原理：Java 内置锁（Monitor），编译后在同步块前后加上 monitorenter/monitorexit 字节码指令。JVM 保证同一时刻只有一个线程进入同步块。JDK 1.6 后经过锁升级优化（偏向锁 → 轻量级锁 → 重量级锁），性能已大幅提升。
缺点：重量级锁涉及线程阻塞/唤醒的系统调用，低竞争时还好，高竞争下性能下降明显。
② AtomicInteger（CAS）atomicInt.incrementAndGet();推荐，高性能，适合高并发计数
CAS 原理（Compare And Swap）：从内存读当前值 E → 计算新值 V → CPU 原子指令比较"内存值是否仍是 E"，是则更新为 V，否则重新读取重试。整个过程只有一次 CPU 原子指令，不需要加锁。
核心代码（Unsafe 类）：compareAndSwapInt(this, valueOffset, expect, update)
ABA 问题：CAS 只检查值是否相同，不检查期间是否被改过又改回来。解决方案：AtomicStampedReference（加版本号）。但在 i++ 计数场景中 ABA 不影响正确性。
③ Lock 锁lock.lock(); try { i++; } finally { lock.unlock(); }需要更灵活锁控制时（tryLock超时、公平锁）
ReentrantLock vs synchronized：
  • ReentrantLock 支持 tryLock(超时)、lockInterruptibly（可中断）、公平锁模式
  • synchronized 自动释放（代码块退出或异常）、语法更简洁、无需手动 unlock
  • 一般场景推荐 synchronized（简单可靠），需要高级特性的用 ReentrantLock
面试标准答案："i++ 不是原子操作（读→改→写三步），多线程下有竞态条件。保证原子性有三招——synchronized 加锁（简单可靠）、AtomicInteger 的 CAS 自旋（高性能，推荐用于计数器）、ReentrantLock（灵活控制）。一般计数场景首选 AtomicInteger。"

  • Spring IOC：读取 XML/注解配置 → Class.forName 加载类 → Constructor.newInstance 创建 Bean → Field.set 注入依赖。这就是 @Autowired 能工作的底层原因。
  • Spring AOP：JDK 动态代理通过反射获取接口方法 → Proxy.newProxyInstance 生成代理对象 → invoke 反射调用目标方法。
  • MyBatis：ResultSet 映射到 Java 对象时，通过反射调用 setter 方法或直接设置字段值。
  • JUnit：扫描带 @Test 注解的方法，通过 Method.invoke 执行测试方法。
  • 序列化框架（Jackson/Gson）：反射读取对象的 getter 获取字段值，或读取 JSON key 后用反射 setter 设置值。
反射的性能和缺点：
  • 反射调用比直接调用慢（需要安全检查、类型检查、方法查找），约慢 10-50 倍。但现代 JVM 通过反射调用的内联优化，热点代码差距会缩小。
  • 破坏封装性——setAccessible(true) 可以访问 private 成员，可能绕过安全限制。
  • 编译期无法检查——如果类名/方法名写错了，编译时不报错，运行时抛异常。
一句话：反射让 Java 在运行时能够"看透"一个类的内部结构并动态操作它。Spring IOC/DI、AOP、MyBatis 的底层全都依赖反射。
面试金句："如果没有反射，我们可能还停留在配置文件 + new 对象的时代，是反射成就了 Java 框架生态。"

Q4: ArrayList 扩容机制？初始化能否自定义扩容参数？

扩容机制：
  1. 默认构造：new ArrayList() → 初始为空数组 DEFAULTCAPACITY_EMPTY_ELEMENTDATA → 首次 add 时扩容到 10
  2. 扩容算法：容量不够时，新容量 = 旧容量 + 旧容量 >> 1 = 旧容量的 1.5 倍
  3. 底层实现：Arrays.copyOf() → System.arraycopy()（将旧数组元素复制到新数组）
可以自定义初始化参数： 
// 方式1：指定初始容量（推荐，避免扩容开销）
new ArrayList<>(100);  // 直接创建容量为 100 的数组

// 方式2：传入已有集合
new ArrayList<>(existingList);  // 根据集合大小初始化
为什么 1.5 倍而不是 2 倍？
  • 1.5 倍比 2 倍更节省空间（特别是大容量时，2 倍扩容可能浪费近 50% 空间）
  • 1.5 倍仍然保证了均摊 O(1)的插入时间（扩容次数是 O(log₁.₅ n) ≈ 比 2 倍多约 70% 的扩容次数，但每次扩容数据复制量更小）
▶ 展开详解
扩容的 JDK 源码关键逻辑（简化版）： 
private void grow(int minCapacity) {
    int oldCapacity = elementData.length;
    // 新容量 = 旧容量 + 旧容量的一半（1.5 倍）
    int newCapacity = oldCapacity + (oldCapacity >> 1);

    if (newCapacity - minCapacity < 0)
        newCapacity = minCapacity;  // 1.5 倍还不够 → 直接用所需容量

    if (newCapacity - MAX_ARRAY_SIZE > 0)
        newCapacity = hugeCapacity(minCapacity);  // 超大数组处理

    elementData = Arrays.copyOf(elementData, newCapacity);
}
生产环境最佳实践：
  • 如果能预估容量，一定要在构造时指定：new ArrayList<>(expectedSize)。避免反复扩容的数据拷贝开销。
  • 大数据量时调用 ensureCapacity()：在循环 add 之前调用 list.ensureCapacity(10000) 一次性扩容到位。
  • 注意 1.5 倍扩容的内存峰值：扩容时需要同时存在旧数组和新数组，内存峰值是 1 + 1.5 = 2.5 倍原数组大小。10GB → 扩容峰值 25GB，可能导致 OOM。

Q5: list.add(1) 底层执行流程？

graph TD A["list.add(1)"] --> B["ensureCapacityInternal(size + 1)"] B --> C{"elementData 够用？"} C -->|"够"| D["elementData[size++] = 1"] C -->|"不够"| E["grow() 扩容"] E --> F["容量 = 1.5 倍 (或 minCapacity)"] F --> G["Arrays.copyOf 复制到新数组"] G --> D D --> H["返回 true"]
完整执行步骤：
  1. 判断容量：ensureCapacityInternal 检查当前 size+1 是否超出数组长度
  2. 容量不够 → 扩容：调用 grow()，新容量 = 旧容量 × 1.5，通过 Arrays.copyOf 创建新数组并复制旧数据
  3. 容量够 → 直接赋值：elementData[size] = 1;
  4. size++：更新当前元素数量 size = size + 1;
  5. 返回 true：add 方法总返回 true（List 接口定义）
▶ 展开详解
面试中的细节——"默认构造首次 add 发生了什么？"

new ArrayList() 时 elementData = {}（空数组，长度 0），size = 0。第一次 add 时：ensureCapacityInternal(1) → calculateCapacity 检测到是空数组 → 取 Math.max(DEFAULT_CAPACITY=10, size+1=1) = 10 → grow(10) → 扩容到 10。这就是为什么 ArrayList 默认构造首次 add 后容量是 10，不是因为默认容量是 10，而是"首次 add 时如果容量不足，至少扩到 10"。

add(int index, Object element) 和 add(Object) 的区别：
  • 尾部 add(e)：size++ 处直接赋值 O(1)（不考虑扩容）
  • 中间位置 add(i, e)：先将 index 后面所有元素后移一位（System.arraycopy），再赋值，O(n) System.arraycopy(elementData, index, elementData, index+1, size-index); elementData[index] = e;

  • Redis 连接池：Lettuce（Spring Boot 默认，Netty 异步）、JedisPool（BIO）

连接验证机制——为什么需要？

TCP 连接可能因为网络中断、服务器重启、防火墙踢掉空闲连接等原因"假活"——看起来还在，但实际发数据时会失败。连接池需要验证机制：① borrow 时验证（testOnBorrow）→ 借出前发 ping，保证拿到的连接一定可用；② idle 时验证（testWhileIdle）→ 空闲检测线程定时检查，剔除死连接。

Q2: SpringBoot 中 starter 是什么？

Starter = 一站式依赖启动器。

本质：一个 Maven 依赖包，内部打包了"某个功能需要的所有 jar 包 + 自动配置类"。引入一个 starter 就引入了所有需要的依赖，不需要一个个手动添加和解决依赖冲突。

举例：引入 spring-boot-starter-web 后自动包含了： 
spring-boot-starter-web
  ├── spring-boot-starter（核心 starter：自动配置 + 日志）
  ├── spring-boot-starter-json（Jackson）
  ├── spring-boot-starter-tomcat（内嵌 Tomcat）
  ├── spring-webmvc（Spring MVC）
  └── spring-web（Spring Web）
一个 starter，自动传递引入 5 个核心依赖——这就是"一站式"的含义。 新建 SpringBoot 项目最小需要的基础 starter： 
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>  <!-- Web 项目标配 -->
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>   <!-- 测试 -->
    <scope>test</scope>
</dependency>
▶ 展开详解
starter 的自动配置原理：

每个 starter jar 的 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports（Spring Boot 3.x）文件中，列出了需要自动加载的配置类。Spring Boot 启动时通过 @EnableAutoConfiguration → AutoConfigurationImportSelector 扫描并加载这些配置类。配置类上用 @ConditionalOnClass、@ConditionalOnBean 等条件注解控制是否生效。

自定义 Starter 的命名规范：
  • 官方 Starter：spring-boot-starter-xxx（如 spring-boot-starter-web）
  • 第三方 Starter：xxx-spring-boot-starter（如 mybatis-spring-boot-starter、redisson-spring-boot-starter）。命名反过来的原因：Maven 中央仓库按 groupId 组织，方便搜索。

Q4: @Transactional 注解作用，为什么用注解实现事务？

sequenceDiagram participant C as Controller participant P as AOP Proxy participant S as Service(目标) participant DB as 数据库 C->>P: service.transfer() P->>DB: ① 开启事务（BEGIN） P->>S: ② 调用目标方法 S-->>P: ③ 返回结果 / 抛异常 alt 正常返回 P->>DB: ④ COMMIT else 抛 RuntimeException P->>DB: ④ ROLLBACK end
@Transactional 的核心作用：声明式事务管理。在方法上加 @Transactional 注解后，Spring AOP 自动在方法前后加上事务管理逻辑——方法成功则提交（commit），出现 RuntimeException 则回滚（rollback），开发者不需要手写 begin/commit/rollback 代码。 为什么用注解而不是手写事务？
  1. 低侵入、高解耦：业务代码只关心业务逻辑，不需要夹杂事务控制代码。一个 @Transactional 注解替代了重复的 try-catch-commit-rollback 模板代码
  2. AOP 动态代理实现：Spring 为标注了 @Transactional 的 Bean 生成代理对象，在代理中织入事务管理逻辑——这是 AOP 的经典应用
  3. 灵活配置：可以配置传播行为（propagation）、隔离级别（isolation）、超时时间（timeout）、只读（readOnly）等
  4. 声明即文档：看注解就知道这个方法需要事务，意图清晰
▶ 展开详解
@Transactional 的关键属性：
属性含义常用值
propagation事务传播行为REQUIRED(默认): 有则加入，无则新建
REQUIRES_NEW: 总是新建独立事务
isolation隔离级别DEFAULT(跟随数据库，通常是 RR)
timeout超时秒数默认 -1（不超时），建议设合理值防长事务
readOnly只读事务（优化）true: 纯查询，提示数据库无需加锁
rollbackFor哪些异常回滚默认只回滚 RuntimeException 和 Error
@Transactional 常见失效场景（面试高频）：
  • 同一个类中方法调用：this.methodB() 不经过代理，事务不生效。因为 AOP 是通过代理对象工作的，this 是目标对象而非代理对象。
  • 非 public 方法：Spring AOP 默认只代理 public 方法，private/protected 方法上加 @Transactional 无效。
  • 异常被 catch 吞掉：方法内 try-catch 了 RuntimeException 且没往外抛，Spring 感知不到异常，不会回滚。
  • 数据库引擎不支持事务：MyISAM 不支持事务，@Transactional 加了也没用。
  • rollbackFor 未指定受检异常：默认只回滚 RuntimeException，如果抛的是受检异常（Exception），需要指定 rollbackFor = Exception.class。
底层实现原理：Spring 通过 AOP 切入 @Transactional 方法——在代理的 invoke 方法中，TransactionInterceptor 调用 PlatformTransactionManager（如 DataSourceTransactionManager）获取事务，然后执行目标方法，根据结果提交或回滚。本质上是把你手写的 JDBC 事务模板代码（conn.setAutoCommit(false) → execute → commit/rollback → conn.setAutoCommit(true)）通过动态代理自动完成了。
@Transactional 一句话：声明式事务注解，AOP 动态代理自动管理事务（开/提/回）。业务代码零侵入，可灵活配置传播/隔离/超时。注意 5 个常见失效场景——同类调用、非 public、异常被吞、引擎不支持、rollbackFor 未指定。

Q1: 聚簇索引与非聚簇索引区别？

graph TD subgraph C1["聚簇索引 | 主键索引"] A["叶子节点 = 完整行数据"] --> B["一张表只有一个"] end subgraph C2["非聚簇索引 | 二级索引"] D["叶子节点 = 主键值"] --> E["一张表可有多个"] end D -->|"回表"| A
维度聚簇索引非聚簇索引（二级索引）
叶子节点内容完整行数据主键值（不是整行数据）
一张表的数量只能有 1 个可以有 多个

查询过程查到叶子 = 拿到所有数据查到叶子 → 拿到主键 → 回表查聚簇索引

▶ 展开详解
深入理解"聚簇"的含义：

"聚簇"的意思是"数据和索引聚集（存在一起）"。InnoDB 的聚簇索引叶子节点就是数据页本身——数据和索引是同一个 B+ 树。MySQL 官方文档称之为 clustered index，Oracle 称之为 Index-Organized Table（索引组织表）。

MyISAM 没有聚簇索引的概念——所有索引的叶子节点存的是数据文件的物理地址（行号），数据和索引分离存放。所以 MyISAM 的主键和二级索引在查询效率上没有本质区别。

为什么二级索引不直接存地址而存主键？

因为 InnoDB 的数据是按主键顺序存放的，如果发生页分裂（插入导致数据页分裂），行的物理位置会变化。如果二级索引存的是物理地址，每次页分裂都要更新所有二级索引的引用——代价太大。存主键值则不受物理位置变化的影响，主键值永远不变。

Q2: 数据库性能调优怎么做？

graph TD A["数据库性能调优"] --> B["① SQL 层面（见效最快）"] A --> C["② 索引层面（性价比最高）"] A --> D["③ 表结构层面"] A --> E["④ 参数层面"] A --> F["⑤ 架构层面"] B --> B1["慢查询日志→EXPLAIN→优化 SQL"] C --> C1["合理建索引 + 覆盖索引"] C --> C2["避免索引失效"] D --> D1["冷热分离（垂直分表）"] D --> D2["合适的数据类型（int > varchar）"] E --> E1["buffer_pool_size 调大"] E --> E2["连接数/超时合理设置"] F --> F1["读写分离（主写从读）"] F --> F2["分库分表（水平拆分）"] F --> F3["引入 Redis 缓存"]
▶ 展开详解
五个层面的调优详解： ① SQL 层面（最先做）：
  • 开启慢查询日志（slow_query_log + long_query_time），用 mysqldumpslow/pt-query-digest 找出耗时 SQL
  • EXPLAIN 分析 type/key/rows/Extra，定位全表扫描（ALL）和临时表（Using temporary）
  • 改写 SQL：避免 SELECT *、大偏移量 LIMIT 用延迟关联、多表 JOIN 用子查询拆分、OR 改 UNION
② 索引层面（最重要）：
  • WHERE / JOIN / ORDER BY / GROUP BY 列建索引
  • 高频查询建覆盖索引（联合索引包含所有 SELECT 列）
  • 避免冗余索引和重复索引（如单独索引 + 联合索引前缀重复）
  • 定期 ANALYZE TABLE 更新索引统计信息
③ 表结构层面：
  • 字段选最小够用的数据类型（如 age 用 TINYINT 而不用 INT）
  • 冷热字段分离——大字段（TEXT/BLOB）拆到扩展表
  • 适当冗余减少 JOIN（反范式化）——如订单表冗余存用户名，避免每次都 JOIN 用户表
④ 参数层面：
  • innodb_buffer_pool_size：设为物理内存的 70-80%
  • innodb_flush_log_at_trx_commit：强一致用 1，性能优先用 2
  • max_connections：根据应用连接池配置合理设置
  • query_cache（8.0 已移除，不用管）
⑤ 架构层面（最后手段）：
  • 读写分离：一主多从，写走主、读走从，中间件（ShardingSphere/ProxySQL）
  • 垂直分库：按业务拆（用户库、订单库），缓解并发写入压力
  • 水平分库分表：按 user_id hash 取模，解决单表过大问题（>1000 万行）
  • 引入 Redis：热点数据缓存，减少 DB 读压力
调优的优先顺序（面试话术模板）："数据库性能调优我一般按这个顺序来——先看慢查询日志找到瓶颈 SQL → EXPLAIN 分析执行计划 → 建索引或优化 SQL 写法（90% 的问题到这一步就解决了）→ 如果还不够，调参数（主要是 buffer_pool）→ 升级硬件（SSD、加内存）→ 最后才考虑架构改造（读写分离、分库分表），因为架构改造成本最高、风险最大。"
调优优先级：SQL 优化 > 索引优化 > 参数调优 > 表结构优化 > 架构改造
90% 的性能问题，加个索引、改条 SQL 就能解决。不要动不动就分库分表。
