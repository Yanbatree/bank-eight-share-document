1. 索引底层数据结构？为什么用 B+ 树？

为什么不用二叉树/红黑树？

二叉树在数据量大时高度会非常高（例如 1000 万行 → 树高约 23 层），每次查询需要 ~23 次 IO。B+ 树每个节点是一个页（16KB），可以存放成百上千个索引键，高度通常只有 3-4 层，IO 次数大幅降低。

为什么不用哈希索引？

哈希索引 O(1) 只能做等值查询，无法处理范围查询（WHERE id > 100）、排序（ORDER BY）、模糊匹配（LIKE 'abc%'）。而实际业务中范围查询非常常见。

为什么不用 B 树？

B 树的非叶子节点也存完整数据，导致每个页能存的键更少，树更高。B+ 树非叶子节点只存键，页内键密度大，树更矮。且 B+ 树叶子节点的双向链表天然支持范围查询。

InnoDB 中 B+ 树的实际结构

InnoDB 中 B+ 树就是聚簇索引的物理结构。叶子页之间通过双向指针连接，页内记录通过单向指针连接，形成有序链表。每次插入/删除可能触发页分裂/合并。

对比哈希索引B 树B+ 树
等值查询O(1)O(logN)O(logN)
等值查询详解：

哈希索引通过哈希函数直接定位到桶，再在桶内遍历链表（如果哈希冲突多会退化），等值查询理论 O(1)。但 InnoDB 的自适应哈希索引（AHI）只在热点页上建立哈希索引。

B/B+ 树等值查询从根节点开始，每次通过二分查找确定下一层节点，直到叶子节点。B+ 树因为非叶子只存键，每次 IO 能覆盖更多数据范围，实际 IO 次数更少。

范围查询不支持需中序遍历叶子链表直接扫
范围查询是关键区分点：

实际业务中超过 70% 的查询包含范围条件（时间范围、ID 范围、分页等）。B+ 树只需找到第一个匹配的叶子节点，然后沿着双向链表向后扫描即可，非常高效。

B 树需要在中序遍历中不断回溯父节点，路径更长、IO 更多。哈希索引完全无法处理范围查询（因为哈希值不保序）。

排序不支持复杂天然有序
排序与 ORDER BY 的关系：

B+ 树叶子节点本身就是有序链表，执行 ORDER BY 时如果走索引，可以直接利用这个有序性，不需要额外的 filesort 操作。

哈希索引的数据是无序存放的，任何排序都需要额外排序。B 树虽然数据总体有序，但叶子节点不互联，排序遍历需要中序遍历 + 栈辅助。

树高度—较高（节点存数据）更矮（节点只存键）
树高度直接影响 IO 次数：

InnoDB 默认页大小 16KB。假设一行数据约 1KB，索引键 8 字节 + 指针 6 字节：

  • B 树：每个节点存完整行（1KB），一页约 16 行 → 扇出约 16 → 1000 万行需约 6 层
  • B+ 树：非叶子节点只存键（14 字节），一页约 1170 个键 → 扇出约 1170 → 1000 万行仅需 3 层

每次 IO 可能是一次磁盘随机读（~10ms），3 层 vs 6 层意味着 30ms vs 60ms 的差距。

核心：B+ 树非叶子节点不存 data，一个页能放更多索引键 → 树更矮 → IO 次数更少。叶子节点双向链表 → 范围查询直接遍历即可。

▶ 展开详解
聚簇索引的物理含义：

聚簇索引的叶子节点就是数据页本身——数据和索引存在一起。InnoDB 按主键顺序存放数据，所以主键最好是自增的（避免页分裂）。如果没有显式定义主键，InnoDB 会选择第一个非空唯一索引，或者自动生成隐藏的 row_id。

非聚簇索引为什么需要回表：

二级索引的叶子存的是主键值 + 索引列，不包含完整行数据。当查询列超出索引覆盖范围时，需要用拿到的主键值再去聚簇索引查一次——这就是 回表。回表是一次额外的 B+ 树搜索，可能触发随机 IO。

MyISAM 的区别：

MyISAM 没有聚簇索引，所有索引的叶子都存的是数据文件的物理地址（行号），数据和索引分离存放。所以 MyISAM 的主键和二级索引查询效率一样，但范围查询没有 B+ 树链表优势。

聚簇索引：叶子节点存放整行数据，一张表只有一个（一般是主键）。
非聚簇索引（二级索引）：叶子节点存放主键值，查完还需要回表。

3. 回表 & 覆盖索引

▶ 展开详解
回表的代价：

回表意味着多一次完整的 B+ 树搜索，从根到叶子多一轮 IO。如果查询返回很多行（比如 LIMIT 1000），每行都要回表一次，变成 N+1 问题。优化器可能会因此放弃索引走全表扫描（当回表代价 > 全表扫描代价时）。

覆盖索引的极致优化：

EXPLAIN 的 Extra 列显示 Using index 即代表覆盖索引——查询完全在索引树内完成，不需要回表。这是最好的查询方式。常见做法：为高频查询建立"宽"联合索引，把所有需要的列都包含进去。

ICP 与覆盖索引的关系：

索引下推（ICP）是在回表之前的过滤优化——先在索引层过滤掉不满足条件的行，只对符合条件的行回表。但它仍然需要回表，只是减少回表次数。覆盖索引则完全消除了回表。两者可以配合使用。

覆盖索引 = 查询的列都在索引中，不需要回表。常用手段是建立合适的联合索引。

4. 最左前缀原则

graph LR subgraph "联合索引 (a,b,c) 等效于" X["索引 (a)"] Y["索引 (a,b)"] Z["索引 (a,b,c)"] end
▶ 展开详解
为什么非要最左列？

联合索引按 (a, b, c) 的顺序排序——先按 a 排，a 相同再按 b 排，b 相同再按 c 排。类似于字典先按拼音首字母排序。如果你跳过 a 直接查 b，索引就相当于未排序的乱序数据，无法用二分查找定位。

实际案例分析：
  • WHERE a=1 AND c=3 → 只用到了 a 列索引（找到 a=1 的所有行，然后逐行过滤 c=3）
  • WHERE a=1 AND b>2 AND c=3 → 用到了 a 和 b（范围查询会让后续列失效，c=3 需要回表过滤）
  • WHERE a=1 AND b=2 ORDER BY c → 完美——a,b 定位，c 直接利用索引排序
如何利用这个原则优化 SQL？

把等值查询列放前面、范围查询列放后面、排序/分组列按顺序放。例如 (status, user_id, create_time) 适合 WHERE status=1 AND user_id>100 ORDER BY create_time。

命中：WHERE a=1 / WHERE a=1 AND b=2 / WHERE a=1 AND b=2 AND c=3
不命中：WHERE b=2（跳过 a）、WHERE a=1 AND c=3（b 断了，只用到了 a）
原因：联合索引按 (a,b,c) 排序，跳过第一列就无法用二分查找定位。

5. 索引失效场景

场景示例原因
!= / <>WHERE status != 1无法精确定位范围

!= 意味着"除了 1 以外的所有值"，这不是一个连续区间，B+ 树无法定位一个起点然后向后扫描。优化器计算后发现大部分行都满足条件（如果只有 0/1 两个值还好，但如果有 0-100，!=1 返回 99% 数据），走全表扫描比索引+回表更快。

例外：如果结合其他等值条件（WHERE status != 1 AND user_id = 100），且 user_id 上有索引，优化器可能会走 user_id 索引。

LIKE 前导模糊WHERE name LIKE '%张'无法二分定位起点

B+ 树按字符串字典序排序，'%张' 表示任意字符开头的字符串，优化器不知道从哪个位置开始扫描（可能是 'a张'、'b张'、'c张'...），所以放弃索引。

LIKE '张%' 可以走索引：因为知道以 '张' 开头，可以定位到 '张' 在 B+ 树中的位置，然后顺序扫描。

列上有函数/运算WHERE YEAR(time)=2024破坏了索引值

索引存的是 time 列的原始值（如 '2024-03-15 10:30:00'），但 WHERE 条件是对 YEAR(time) 的结果（2024）进行匹配。优化器无法把索引值和函数结果对应起来。

解决方案：改为 WHERE time >= '2024-01-01' AND time < '2025-01-01'，这是等价的纯范围条件，可以走索引。

隐式类型转换WHERE phone=13800138000 (phone是varchar)MySQL 对索引列做了隐式转换

当字符串列和数字比较时，MySQL 会把字符串转为数字——相当于对 phone 列执行了 CAST(phone AS SIGNED)。这在优化器看来就是"列上有函数"，索引失效。

反过来不失效：WHERE id = '100'（id 是 int），MySQL 会把 '100' 转为数字，不会对 id 列做转换，索引正常。

OR 不全有索引WHERE a=1 OR b=2（b 没索引）优化器认为全表更快

OR 意味着两个条件的结果取并集。如果 b 没有索引，b=2 只能全表扫描，a 的索引也无法限制扫描范围，优化器直接选择全表扫描。

解决方案：把 OR 拆成 UNION（如果两个条件各自都有索引），或者给 b 也加索引。

违反最左前缀联合索引 (a,b) 只查 b

详见上面"最左前缀原则"——跳过第一列就相当于没有这个索引。但如果查询是 WHERE a=1 OR b=2 且 b 有单独索引，可以通过 index_merge（索引合并）优化来利用两个索引的 bit map 做交集/并集。

6. 索引下推（ICP）

flowchart LR subgraph "无 ICP" A["存储引擎: 按 name 过滤"] --> B["返回所有匹配行给 Server"] B --> C["Server 层: 再按 age 过滤"] end subgraph "有 ICP (5.6+)" D["存储引擎: 按 name AND age 一起过滤"] --> E["只返回符合全部条件的行"] end
▶ 展开详解
ICP 解决什么问题？

MySQL 架构中，存储引擎（InnoDB）负责从索引中取数据，Server 层负责在 WHERE 条件中做最终过滤（尤其是涉及非索引列的条件）。在没有 ICP 时，存储引擎把所有满足索引条件的行都返回给 Server 层，由 Server 层再过滤非索引条件——这意味着很多行被无谓地回表、传输、然后丢弃。

ICP 把部分 WHERE 条件下推到存储引擎层，让引擎在索引扫描时就利用这些条件过滤。虽然非索引列不能在索引中定位，但可以在扫描过程中判断——命中了才回表。

ICP 触发条件：
  • 查询使用二级索引（非聚簇索引）
  • WHERE 条件中包含索引列 + 非索引列
  • EXPLAIN 的 Extra 列显示 Using index condition
ICP 的局限性：

ICP 只能下推"可以在索引扫描过程中判断"的条件（比如 WHERE name LIKE '张%' AND age > 20，name 走索引，age 可以在索引叶子节点上判断）。它不能替代覆盖索引，因为回表仍然存在，只是次数减少。

效果：在索引层就过滤掉不满足全部 WHERE 条件的行，减少回表和网络开销。

原子性 — 为什么用 undo log 而不是直接回滚？

事务可能在执行到一半时崩溃。undo log 在修改数据前先记录"旧值"，崩溃恢复时通过 undo log 把已修改的页恢复到事务开始前的状态。undo log 也是 MVCC 的基础（存储历史版本）。

一致性 — 为什么说是"由 A+I+D 共同保证"？

一致性是目的，原子性、隔离性、持久性是手段。数据库本身还通过约束（主键、外键、NOT NULL、CHECK）来保证数据完整性。如果业务写错了 SQL（比如把所有人的年龄设成 -1），数据库无法阻止——这属于业务一致性。

隔离性 — MVCC + 锁如何配合？

MVCC 负责"快照读"（普通 SELECT）——读历史版本，不加锁，高并发。锁（行锁、间隙锁）负责"当前读"（SELECT ... FOR UPDATE / UPDATE / DELETE）——读最新版本，加锁，防止并发冲突。两者互补实现 RR 级别的隔离性。

持久性 — redo log 的 WAL 机制

如果每次事务提交都直接刷数据页到磁盘，会产生大量随机 IO。redo log 是顺序写入（追加到日志文件末尾），速度快。崩溃后通过 redo log 重放（redo）来恢复已提交但未刷盘的数据。

特性说明实现机制
原子性事务不可分割，全做或全不做undo log 回滚

undo log 记录的是"修改前的数据"——INSERT 对应写一条 DELETE 的 undo；UPDATE 对应写一条反向 UPDATE 的 undo。回滚时按 undo log 依次逆向执行。undo log 本身也是通过 redo log 保证持久性的——undo 页的修改也写 redo。

一致性事务前后数据完整性约束不变其他三个特性共同保证

数据库层面的约束包括：实体完整性（主键唯一）、参照完整性（外键约束）、用户定义完整性（NOT NULL、CHECK、触发器）。InnoDB 通过 doublewrite buffer（防止页部分写）和 crash recovery 机制保证数据页的一致性。

隔离性并发事务互不干扰MVCC + 锁机制

MVCC 的核心是 ReadView——一个事务开始时生成的数据快照。不同隔离级别的 ReadView 生成时机不同：RC 每次查询生成新快照，RR 事务开始时生成一次。锁（共享锁 S / 排他锁 X / 意向锁 IS/IX）用于写操作的并发控制。

持久性提交后数据永久保存redo log + 刷盘

事务提交时，redo log 必须落盘（fsync），但数据页可以不立即刷盘（由后台线程异步刷）。这就是 WAL 的核心思想——只需保证日志持久化，数据最终会持久化。MySQL 通过 innodb_flush_log_at_trx_commit=1 保证每次提交都 fsync redo log。

2. 事务并发三大问题

sequenceDiagram participant T1 as 事务 T1 participant DB as 数据库 participant T2 as 事务 T2 Note over T1,T2: 脏读 T1->>DB: 修改 age=20（未提交） T2->>DB: 读取 age → 20 T1->>DB: ROLLBACK（age 恢复原值） Note over T2: T2 读到了"脏数据" Note over T1,T2: 不可重复读 T1->>DB: 读取 age → 18 T2->>DB: UPDATE age=20 + COMMIT T1->>DB: 再次读取 age → 20 Note over T1: 两次读同一个数据，结果不同 Note over T1,T2: 幻读 T1->>DB: SELECT COUNT(*) → 100 T2->>DB: INSERT 一行 + COMMIT T1->>DB: SELECT COUNT(*) → 101 Note over T1: 两次查，行数不同
▶ 展开详解
脏读 vs 不可重复读 vs 幻读——本质区别：
  • 脏读：读到了其他事务未提交的修改——这是最严重的，因为那些修改可能被回滚
  • 不可重复读：同一行数据，两次读到的值不一样（被其他事务 UPDATE 并提交了）——针对已提交的修改
  • 幻读：同样的查询条件，两次读到的行数不一样（被其他事务 INSERT/DELETE 了）——针对新增/删除的行
InnoDB 如何在 RR 级别解决幻读？

快照读（普通 SELECT）：通过 MVCC 的 ReadView，只读事务开始前已提交的数据，看不到后来的 INSERT。
当前读（SELECT FOR UPDATE）：通过 Next-Key Lock（行锁 + 间隙锁）锁住查询范围，阻止其他事务在间隙中 INSERT。

RR 级别真的完全解决幻读了吗？

没有完全解决。如果 T1 先快照读（SELECT），然后 T2 INSERT 并提交，接着 T1 用当前读（SELECT FOR UPDATE 或 UPDATE），T1 会看到 T2 插入的行——这种情况下发生了幻读。解决办法是开启事务后立即执行一次当前读来锁住范围。

3. 四种隔离级别

隔离级别脏读不可重复读幻读实现
READ UNCOMMITTED可能可能可能不加锁

最低隔离级别，几乎没有并发控制。事务可以读到其他事务未提交的修改（脏读）。在生产环境中几乎不用。唯一的好处是并发性能最高，因为不需要任何锁或版本判断。

READ COMMITTED防止可能可能每次 SELECT 新建 ReadView

每次 SELECT 语句执行时创建新的 ReadView（快照），所以看不到未提交的数据（避免了脏读），但可以看到其他事务在两次 SELECT 之间提交的修改（不可重复读）。这是 Oracle 的默认隔离级别。许多互联网公司也使用 RC + 开启 binlog row 格式来避免主从不一致。

REPEATABLE READ防止防止防止 (MVCC 基本解决)事务首次 SELECT 建 ReadView + Next-Key Lock

InnoDB 的默认隔离级别。事务开始时创建 ReadView 并复用，所以一直看到相同的数据快照（可重复读）。通过 Next-Key Lock（行锁 + 间隙锁）阻止其他事务在查询范围内插入数据来防幻读。但要注意：RR 下如果两个事务同时 UPDATE 同一行，后提交的会触发"在可重复读之前数据已被修改"的错误。

RR vs RC 怎么选？RC 并发更好（间隙锁少），RR 一致性更好（可重复读）。互联网高并发场景常用 RC + row 格式 binlog。

SERIALIZABLE防止防止防止所有读加共享锁

最高隔离级别。所有普通 SELECT 自动转为 SELECT ... FOR SHARE（加共享锁），相当于所有读都是当前读且加锁。并发度最低，容易死锁。一般只在非常严格的场景（如金融账务核对）中使用。

InnoDB 默认 RR 级别，通过 MVCC（快照读）+ Next-Key Lock（当前读）基本解决幻读

MVCC 的三个隐藏列：

InnoDB 每行数据都有三个隐藏列：DB_TRX_ID（最近修改该行的事务 ID，6 字节）、DB_ROLL_PTR（回滚指针，指向 undo log 中的旧版本，7 字节）、DB_ROW_ID（隐藏主键，6 字节，当表没有显式主键时自动生成）。

ReadView 的可见性判断算法：
  • trx_id == creator_trx_id → 可见（自己修改的）
  • trx_id < min_trx_id → 可见（事务开始前已提交）
  • trx_id >= max_trx_id → 不可见（事务开始后才启动的事务修改的）
  • min_trx_id <= trx_id < max_trx_id → 如果在 m_ids 中则不可见（活跃事务修改的），否则可见（已提交的）
undo log 版本链的清理：

当没有任何 ReadView 需要某个历史版本时（所有活跃事务都不再需要），Purge 线程会清理这些旧的 undo log。长事务会导致 undo log 膨胀（因为旧版本不能被清理）。

RC vs RR 关键区别：
• RC：每次 SELECT 生成新 ReadView → 能看到其他事务已提交的修改 → 不可重复读
• RR：首次 SELECT 生成 ReadView，后续复用 → 看不到后来的提交 → 可重复读

5. 行锁类型

graph LR subgraph "Next-Key Lock 范围" A["索引值 10"] -->|"Gap Lock"| B["空隙"] B -->|"Record Lock"| C["索引值 20"] C -->|"Gap Lock"| D["空隙"] D -->|"Record Lock"| E["索引值 30"] end F["Next-Key Lock = Gap Lock + Record Lock
锁定区间: (10,20] 或 (20,30]"]
▶ 展开详解
Record Lock（记录锁）：

锁定单个索引记录。例如 SELECT * FROM user WHERE id=1 FOR UPDATE 会在 id=1 的聚簇索引记录上加 X 锁。如果没有索引或索引失效，会退化为表锁。

Gap Lock（间隙锁）：

锁定索引记录之间的间隙，阻止其他事务在间隙中 INSERT。只在 RR 级别下生效。例如已有 id=10 和 id=20 两行，间隙锁会锁住 (10, 20) 这个区间。Gap Lock 之间不冲突——两个事务可以同时持有同一个间隙的 Gap Lock。

Next-Key Lock（临键锁）：

Record Lock + Gap Lock 的组合，锁定一个左开右闭的区间。例如 (10, 20] 表示锁住 10-20 之间的间隙和 id=20 这条记录。这是 InnoDB 在 RR 级别下防止幻读的核心机制。

锁的退化：

当查询条件是唯一索引的等值查询（如 WHERE id=10）且记录存在时，Next-Key Lock 退化为 Record Lock。当查询条件是唯一索引等值查询但记录不存在时，退化为 Gap Lock。

锁作用备注
Record Lock锁定单条索引记录精确命中

Record Lock 锁的是索引记录，不是数据行本身。如果表没有索引，MySQL 会锁定所有扫描到的行（实际上退化为表锁）。当 WHERE 条件是唯一索引且命中唯一行时，只锁这一行。

Gap Lock锁定索引间隙阻止 INSERT，只在 RR 级别

Gap Lock 是 RR 级别特有的。RC 级别没有 Gap Lock（这也是 RC 并发性能更好的原因之一）。Gap Lock 只阻止 INSERT，不阻止 SELECT 或 UPDATE。多个事务可以同时持有同一个 Gap Lock（因为 Gap Lock 之间不互斥）。

Next-Key LockRecord + GapInnoDB 默认行锁形式

在 RR 级别下，InnoDB 默认使用 Next-Key Lock 来扫描索引（不仅是精确等值查询）。这是为了解决幻读——不仅当前存在的行被锁定，还会锁住间隙阻止插入。如果查询条件覆盖了多个记录和间隙，会产生多个 Next-Key Lock。

6. 死锁 & 解决

sequenceDiagram participant A as 事务A participant DB as MySQL participant B as 事务B A->>DB: UPDATE SET age=20 WHERE id=1 Note over A: 持有 id=1 的 X 锁 B->>DB: UPDATE SET age=30 WHERE id=2 Note over B: 持有 id=2 的 X 锁 A->>DB: UPDATE SET age=25 WHERE id=2 Note over A: 等待 id=2 的锁... B->>DB: UPDATE SET age=35 WHERE id=1 Note over B: 等待 id=1 的锁... Note over DB: 死锁！互相等待 DB-->>A: 回滚代价小的事务
▶ 展开详解
InnoDB 的死锁检测机制：

InnoDB 通过等待图（wait-for graph）检测死锁——事务 A 等待事务 B 的锁，事务 B 又等待事务 A 的锁，形成了循环。InnoDB 会回滚"undo log 量最小"的事务（代价最小的），另一个事务获得锁继续执行。

innodb_deadlock_detect=ON（默认开启）会在每次加锁等待时检查是否存在死锁。在高并发场景下，死锁检测本身也是性能瓶颈（O(n²) 复杂度），可以通过 innodb_lock_wait_timeout 来配合使用。

死锁预防策略：
  • 统一访问顺序：所有事务按相同的顺序访问资源（例如都先操作 id=1 再操作 id=2）
  • 缩短事务：把非数据库操作（RPC 调用、文件 IO）移出事务
  • 合理索引：索引可以缩小锁范围，避免不必要的 Gap Lock
  • RC 级别：RC 没有 Gap Lock，减少锁冲突概率
  • 分布式锁：在应用层先获取锁再操作，但引入了额外的复杂度
预防死锁：统一资源访问顺序、缩短事务、合理索引（减少锁范围）、死锁检测（innodb_deadlock_detect=ON）

三、日志系统

1. redo log vs undo log

redo logundo log
作用崩溃恢复，保证持久性事务回滚 + MVCC，保证原子性

redo log 和 undo log 的定位完全不同：redo 是向前滚动（重放已提交事务），undo 是向后滚动（撤销未提交事务）。崩溃恢复时，先 redo 把已提交的修复，再 undo 把未提交的回滚。

记录内容物理日志：页的修改逻辑日志：修改前的数据

redo log 记录的是"在某个表空间的某个页面的某个偏移量上写入了什么数据"——物理层面的修改。undo log 记录的是"把某行数据的某列从什么值改成了什么值"——逻辑层面的修改。物理日志回放快但不可读，逻辑日志可读但回放慢。

写入时机事务执行中写入 buffer修改数据前先写入

redo log 写入流程：事务修改数据 → 先写 redo log buffer（内存）→ 事务提交时 fsync 到磁盘。undo log 在修改数据之前就要写入（并记录对应的 redo 来保护 undo 本身），因为回滚和 MVCC 都需要它。

存储位置redo log file（循环写）undo 表空间

redo log 是固定大小的循环文件（默认 48MB x 2），写满后触发 checkpoint 把脏页刷盘，然后覆盖旧日志。undo log 存在 undo 表空间中（innodb_undo 目录），可以自动扩展，但长事务会导致 undo 膨胀。

2. WAL 机制

flowchart LR A["更新请求"] --> B["写 redo log buffer"] B --> C["写 binlog"] C --> D["写 redo log (prepare)"] D --> E["写 binlog 完成"] E --> F["redo log 标记 commit"] F --> G["返回客户端成功"] G -.->|"后台异步"| H["脏页刷盘"]
▶ 展开详解
为什么需要 WAL？

假设没有 WAL：每次 UPDATE 都要找到对应数据页（可能不在内存中，需要磁盘 IO），修改后立即刷盘。如果修改涉及多个页（比如更新了索引），就需要多次随机 IO。WAL 把随机写转为顺序写——只需把修改记录追加到 redo log 末尾，数据页可以等后台空闲时再批量刷盘。

redo log 的循环写入：

redo log 有固定大小（innodb_log_file_size），由两个文件组成（ib_logfile0 和 ib_logfile1），循环写入。write pos 是当前写入位置，checkpoint 是已刷盘位置。当 write pos 追上 checkpoint 时，必须等待 checkpoint 推进（脏页刷盘），此时数据库写入会暂停。

checkpoint 机制：

checkpoint 把 buffer pool 中的脏页批量刷到磁盘，然后更新 redo log 的 checkpoint 位置。InnoDB 有 sharp checkpoint（暴力刷所有脏页，发生在关闭时）和 fuzzy checkpoint（逐步刷，正常运行时的策略）。Master Thread 每隔一定时间自动触发 fuzzy checkpoint。

Write-Ahead Logging：先写日志再写磁盘。崩溃后靠 redo log 重放恢复。避免每次更新都随机写磁盘。

3. binlog vs redo log

binlogredo log
层级Server 层（所有引擎通用）InnoDB 引擎层

binlog 是 MySQL Server 层实现的，所有存储引擎（InnoDB、MyISAM、Memory）都支持。redo log 是 InnoDB 特有的，其他引擎没有。这意味着即使你用 MyISAM 引擎，binlog 仍然会生成，但没有 redo log 的保护。

内容逻辑日志（SQL/行变更）物理日志

binlog 的三种格式：
1. STATEMENT：记录 SQL 语句本身，量小但可能主从不一致（如使用了 NOW()）
2. ROW（推荐）：记录每行数据的具体变更，量大但精确
3. MIXED：默认 STATEMENT，特殊情况用 ROW

redo log 记录的是页的物理修改（比如"在页号 100 的偏移 200 处写入 4 字节数据"），与 SQL 无关。

用途主从复制、数据恢复崩溃恢复

binlog 用于：主从复制（从库重放 binlog）、基于时间点的数据恢复（用 mysqlbinlog 工具回放）、Canal/Maxwell 等 CDC 工具订阅数据变更。redo log 只在崩溃恢复时使用（将未刷盘的已提交事务的数据页恢复）。

写入方式事务提交时一次性写事务执行中持续写

binlog 在事务提交时一次性写入（这就是为什么两阶段提交是必要的——需要协调 redo log 和 binlog 的写入顺序）。redo log 在事务执行过程中持续写入 buffer，提交时 fsync 落盘。

4. 两阶段提交

sequenceDiagram participant T as 事务 participant R as redo log participant B as binlog T->>R: 写入 redo log，状态=prepare T->>B: 写入 binlog T->>R: redo log 状态改为 commit Note over T,R: 崩溃恢复判断： Note over T,R: redo prepare + binlog 完整 → 提交 Note over T,R: redo prepare + binlog 不完整 → 回滚
▶ 展开详解
为什么需要两阶段提交？

redo log 和 binlog 是两个独立的日志系统，需要保证它们的一致性。如果先写 redo 再写 binlog——写完 redo 后崩溃，binlog 没有这条记录，从库会丢失这条数据（主从不一致）。如果先写 binlog 再写 redo——写完 binlog 后崩溃，redo 没有记录，崩溃恢复后主库没有这条数据但从库有（还是不一致）。两阶段提交通过 prepare + commit 机制保证了要么都成功，要么都失败。

崩溃恢复的判断逻辑：
  • redo log 处于 prepare 状态 + binlog 完整 → 说明 binlog 已写入，事务应该提交，重做 redo
  • redo log 处于 prepare 状态 + binlog 不完整 → 说明 binlog 未写入，事务应该回滚
  • redo log 处于 commit 状态 → 正常提交了，redo 重放即可
组提交优化（Group Commit）：

MySQL 5.6+ 引入了组提交，将多个事务的 binlog 写入和 redo log fsync 合并为一次操作，大幅提升写入性能。三个阶段：flush（刷 binlog）、sync（fsync binlog）、commit（redo commit），每个阶段可以攒批处理。

目的：保证 binlog 和 redo log 的一致性，这样才能保证主从数据一致。

1. 一条 SQL 的执行过程

各阶段详解：

连接器：TCP 握手 → 身份认证（用户名密码）→ 权限读取（此后连接内的权限不会更新）。连接用完后通过连接池管理，长连接可能导致内存占用高（MySQL 在执行过程中使用的内存都分配在连接对象中）。

执行器：判断用户对表的权限 → 调用存储引擎接口逐行获取数据 → 对于带索引的查询，根据优化器的执行计划一条条取数据。

2. EXPLAIN 关键字段

字段含义好 / 坏
type访问类型system > const > eq_ref > ref > range > index > ALL
各类型含义：
  • system：表只有一行数据（系统表），几乎见不到
  • const：主键或唯一索引的等值查询，最多返回一行（WHERE id=1）
  • eq_ref：JOIN 时被驱动表走主键/唯一索引，每个驱动行匹配至多一行
  • ref：非唯一索引等值查询（WHERE name='张三'），可能返回多行
  • range：索引范围扫描（WHERE id > 100, BETWEEN, IN）
  • index：全索引扫描（扫整个索引而非全表，如 COUNT(*) 走二级索引）
  • ALL：全表扫描——需要优化的信号
key实际使用的索引有索引名 / NULL

key 显示 MySQL 实际选择使用的索引名称。如果是 NULL 说明没有使用索引。possible_keys 显示候选索引列表，如果 possible_keys 有值但 key 是 NULL，可能优化器认为全表扫描更快（例如表很小，或索引区分度不够）。可以用 FORCE INDEX 强制指定索引对比效果。

rows预估扫描行数越小越好

rows 是优化器根据统计信息估算的需要扫描的行数，不是实际行数。当 rows 远大于实际返回行数时，说明索引选择性差或统计信息过期。可以通过 ANALYZE TABLE 更新统计信息。rows 的估算基于索引基数（Cardinality），通过采样得到。

Extra额外信息Using index / Using filesort / Using temporary
关键 Extra 值：
  • Using index（好）：覆盖索引，不需要回表
  • Using index condition（中等）：索引下推（ICP），在索引层过滤后再回表
  • Using where（一般）：Server 层对存储引擎返回的数据做了 WHERE 过滤
  • Using filesort（坏）：需要额外排序（内存或磁盘），可优化的信号
  • Using temporary（坏）：需要临时表来处理查询（如 GROUP BY + ORDER BY 不同列），严重影响性能
  • Using join buffer（需关注）：JOIN 时使用了 join buffer（被驱动表没走索引）
key_len索引使用的字节数越大说明用的列越多

key_len 可以帮助判断联合索引实际用了多少列。例如联合索引 (name VARCHAR(50), age INT)，如果 key_len=152（UTF-8 下 VARCHAR(50) = 50*3+2），说明只用了 name 列；如果 key_len=156（加上 INT 的 4 字节），说明两列都用了。可以用这个值来验证最左前缀是否按预期工作。

▶ 展开详解
垂直拆分 vs 水平拆分的决策：

垂直分库：按业务域拆分（用户库、订单库、商品库），每个库独立部署。解决了单库的并发连接数和写入压力。但跨库 JOIN 变成了应用层聚合，引入了分布式事务问题。适合微服务架构。

垂直分表：把一张宽表的常用列和冷列分开。例如 user 表拆成 user_base（id, name, phone, email）和 user_detail（id, address, bio, avatar）。减少了单行大小，让 buffer pool 能缓存更多热点数据。

水平拆分：按某个字段（通常是主键或分片键）hash 取模或按范围拆分到多个库/表中。解决了单表数据量过大的问题（超过 500 万行查询开始变慢）。

分库分表带来的核心挑战：
  • 分布式 ID：不能用自增主键了，需要雪花算法（Snowflake）或其他分布式 ID 方案
  • 跨分片查询：ORDER BY、GROUP BY、分页变得复杂（需要从多个分片取数据后合并）
  • 跨分片事务：需要分布式事务（XA、TCC、Saga、AT 模式等）
  • 扩容：从 4 库扩到 8 库，数据需要重新分配，可能需要停服迁移
  • 中间件：ShardingSphere、MyCat、Vitess 等
垂直分库：按业务拆分 → 解决并发压力
垂直分表：冷热分离 → 减少单行大小
水平分库分表：按字段 hash 取模 → 解决单表数据量大的问题
带来的问题：分布式事务、跨库 JOIN、全局 ID（雪花算法）

五、主从复制

flowchart LR M["Master 主库"] S["Slave 从库"] M -->|"写 binlog"| M M -->|"IO 线程拉取 binlog"| S S -->|"写入 relay log"| S S -->|"SQL 线程重放"| S
▶ 展开详解
主从复制的三个线程：

1. Master 的 Binlog Dump 线程：主库为每个从库创建一个 dump 线程，负责把 binlog 中的事件发送给从库。
2. Slave 的 IO 线程：从库连接到主库，接收 binlog 事件并写入本地的 relay log。
3. Slave 的 SQL 线程：从库从 relay log 中读取事件并重放在本地数据库。

复制延迟的根源分析：
  • 主库并发写入，但从库 SQL 线程只有一个（5.7 之前），串行回放跟不上
  • 大事务（如批量 UPDATE 100 万行）在主库可能只执行几秒，但在从库回放可能需要相同时间
  • 从库机器性能不如主库（IO/CPU 瓶颈）
并行复制（5.7+）：

通过 slave_parallel_type=LOGICAL_CLOCK + slave_parallel_workers=N，从库可以按组提交（group commit）的方式并行回放 binlog。同一个组内的事务在从库可以并发执行，不同组之间串行。MySQL 8.0 引入了基于 writeset 的并行复制，可以跨组并行。

半同步复制：

默认是异步复制（主库不管从库是否收到）。半同步复制下，主库至少等一个从库确认收到 binlog 后才返回客户端成功。通过 rpl_semi_sync_master_wait_point=AFTER_SYNC 配置。可以减少数据丢失的风险但不能完全消除。

延迟原因：Slave 单线程回放、大事务、机器性能差异。
缓解方案：MySQL 5.7+ 并行复制（按库/按组提交）、升级硬件、拆分大事务。
半同步复制：Master 等至少一个 Slave 收到 binlog 后再返回成功。
