进程与线程的本质区别——资源 vs 执行：

进程是资源分配的基本单位（拥有独立的地址空间、文件描述符、信号处理表等），线程是CPU 调度的基本单位（拥有独立的程序计数器、寄存器、栈）。一个进程可以包含多个线程，所有线程共享进程的堆、代码段、数据段和打开的文件，但每个线程有自己的调用栈和寄存器上下文。这就是为什么线程间通信（通过共享内存）比进程间通信（需要通过管道/消息队列等内核机制）快得多。

上下文切换代价的量化理解：

线程切换：保存/恢复 PC、寄存器组、栈指针 → 通常在纳秒级。进程切换：除了保存线程上下文，还需要切换页表基址寄存器（CR3 on x86）、刷新 TLB（Translation Lookaside Buffer）→ 可能导致后续内存访问大量 TLB miss，代价在微秒级。这就是为什么线程被称为"轻量级进程"（LWP, Lightweight Process）。

银行后台场景：

在高并发交易系统中（如银行核心账务），通常使用线程池而非频繁创建进程——因为每次 fork 一个进程需要复制整个父进程的页表（即使有 COW），而创建线程只需分配栈空间和初始化寄存器上下文。对于每秒处理数万笔交易的系统，这个开销差异是决定性的。

对比维度进程 (Process)线程 (Thread)
资源单位资源分配的基本单位CPU 调度的基本单位

进程是 OS 进行资源分配（内存、IO 设备、文件句柄）的最小单元。线程是 CPU 时间片分配的最小单元——调度器不看进程，只看线程。在 Linux 内核中，进程和线程都是用 task_struct 表示的，只是共享的资源不同：clone() 系统调用通过不同的标志位（CLONE_VM, CLONE_FILES 等）决定父子是否共享地址空间和文件描述符，从而区分创建的是"进程"还是"线程"。

地址空间独立虚拟地址空间共享进程地址空间

每个进程拥有独立的页表，虚拟地址到物理地址的映射完全隔离。线程没有自己的页表——所有线程共享同一个进程的页表，这意味着线程 A 可以直接通过指针访问线程 B 在堆上分配的内存（当然需要同步机制保护）。这种共享是线程通信快的基础，也是线程不安全的根源。

上下文切换重（保存页表、刷新 TLB）轻（仅保存寄存器、PC、栈指针）

进程切换时，除了通用寄存器、程序计数器、栈指针之外，还必须切换地址空间——即加载目标进程的页表基址到 MMU 的页表基址寄存器（x86 上的 CR3）。CR3 的写入会触发 TLB 全刷新（除非使用 PCID 优化），导致该 CPU 核心的 TLB 缓存全部作废。后续的内存访问都会经历一次完整的页表遍历（4 级页表 × 4 次内存访问），直到 TLB 重新预热。线程切换不涉及地址空间切换，TLB 中的条目仍然有效。

通信方式管道、消息队列、共享内存、Socket 等 IPC直接读写共享内存（堆、全局变量）

进程间通信必须经过内核（即使是最快的共享内存也需要通过系统调用 mmap 建立映射），涉及用户态-内核态的切换和数据拷贝。线程间通信可以直接通过读写同一个进程地址空间内的变量完成，不需要系统调用。这就是为什么高并发程序中倾向于使用多线程+共享数据结构而非多进程——但也带来了竞态条件的问题，需要锁、信号量等同步机制来保护。

崩溃影响一个进程崩溃不影响其他进程一个线程崩溃通常导致整个进程崩溃

进程隔离是操作系统安全的基石——每个进程在独立的地址空间中运行，进程 A 的 SIGSEGV（段错误）不会波及进程 B。但线程没有这种隔离：一个线程的未捕获异常、空指针解引用或者栈溢出，会导致整个进程收到 SIGSEGV 信号并被终止。这就是为什么 Chrome 浏览器采用多进程架构——每个 Tab 一个进程，一个页面崩溃不会导致浏览器整体崩溃。Java 中线程的未捕获异常可以通过 UncaughtExceptionHandler 捕获，但如果异常导致 JVM 级错误（如 OOM），进程仍然会终止。

Java 映射JVM 进程本身Java Thread → OS 线程 1:1

HotSpot JVM 采用 1:1 线程模型——每个 java.lang.Thread 对象底层对应一个操作系统原生线程（Linux 上通过 pthread_create 创建）。这意味着 Java 线程的调度完全由操作系统内核的调度器决定，JVM 只负责管理 Thread 对象的生命周期和状态转换。早期 JVM 还有 N:1 绿色线程模型（所有 Java 线程映射到单个 OS 线程），但因为无法利用多核 CPU 而早已被淘汰。Project Loom 的虚拟线程则是 M:N 模型——大量虚拟线程被调度到少量 OS 线程上执行。

2. 进程间通信 (IPC)

方式特点适用场景
管道 (Pipe)半双工，单向，仅父子进程Shell 管道 ls | grep

管道是最古老的 IPC 机制，由内核维护一个环形缓冲区（Linux 上默认 64KB）。数据在管道中按 FIFO 顺序流动，写入端和读取端各占一端，单向传输。管道必须由父进程创建（pipe() 系统调用返回两个 fd），然后 fork 子进程，子进程继承 fd，父子各自关闭不需要的一端。管道是字节流协议——没有消息边界，发送方写 100 字节，接收方可能分两次读出 60+40 字节。匿名管道的生命周期与进程绑定，进程退出后管道自动销毁。

命名管道 (FIFO)有文件名，任意进程可访问无亲缘关系的进程间通信

命名管道通过 mkfifo 命令或 mkfifo() 系统调用在文件系统中创建一个特殊的文件节点（类型为 FIFO）。与匿名管道不同，命名管道有持久化的文件系统入口，任意两个知道路径的进程都可以打开它进行通信——不需要父子关系。但是数据仍然是内核内存中的缓冲区，不会真正写入磁盘。当所有读写端都关闭后，管道中的数据丢失。命名管道仍然是半双工的，如果需要双向通信，需要创建两个 FIFO。

消息队列消息有边界，异步，支持优先级异步解耦，生产-消费者模式

与管道不同，消息队列以消息为单位（每条消息有长度和类型），保留了消息边界——发送方发 3 条消息，接收方一定收到 3 条消息，不会出现"粘包"。消息队列支持按类型选择性接收（而不是严格的 FIFO），可以给不同消息设置不同优先级。消息队列的生命周期跟随内核，除非显式删除或重启系统，否则消息一直存在。缺点是每条消息有长度限制（Linux 默认 8192 字节），且消息队列的总容量有限。

共享内存最快（零拷贝），需同步高频大数据量交换

共享内存是最快的 IPC 方式，因为数据不需要在用户态和内核态之间拷贝。两个进程各自将同一块物理内存映射到自己的虚拟地址空间（通过 shmget + shmat 或 mmap + MAP_SHARED），之后就能像访问普通内存一样直接读写。但共享内存本身不提供任何同步机制——进程 A 正在写数据时进程 B 可能读到不完整的数据，这就是竞态条件。因此共享内存通常配合信号量使用。在银行系统中，高性能交易中间件（如 Tuxedo）常用共享内存作为进程间数据传输通道。

信号量 (Semaphore)同步原语，非数据传输控制共享资源的并发访问

信号量本身不传输数据，它是一个计数器，用于控制多个进程/线程对共享资源的访问。P 操作（减 1，如果结果为负则阻塞）和 V 操作（加 1，唤醒等待者）都是原子操作（由内核保证）。信号量在"生产者-消费者"问题中非常经典：empty 信号量初始为 N（缓冲区空位数），full 信号量初始为 0（已填充数），mutex 信号量初始为 1（互斥访问缓冲区）。注意信号量和互斥锁的区别：信号量可以允许多个访问者（计数 > 1），互斥锁只允许一个。

Socket跨主机，支持 TCP/UDP/Unix Domain网络通信、同一主机高效通信

Socket 是最通用的 IPC 方式——不仅能用于同一台主机上的进程通信（Unix Domain Socket），还能跨网络通信（TCP/UDP Socket）。Unix Domain Socket 在同一主机上的性能优于 TCP Socket（不需要经过网络协议栈，不走网卡驱动），但仍然不如共享内存。它常用于本地服务通信：Docker daemon 通过 /var/run/docker.sock 与客户端通信，MySQL 客户端通过 /var/run/mysqld/mysqld.sock 连接本地 MySQL 服务。

Signal异步通知，信息量小进程控制 (SIGKILL, SIGTERM)

信号是 Unix 系统中最古老的进程间异步通知机制。一个进程可以给另一个进程发送信号（如 kill -9 PID 发送 SIGKILL），信号的处理方式有三种：默认处理（大多终止进程）、忽略（SIGKILL 和 SIGSTOP 不能忽略）、自定义信号处理函数。信号携带的信息量极小——只能告诉目标进程"发生了某件事"（信号编号），几乎没有数据传输能力（除了 sigqueue 可以附带少量数据）。信号处理函数中不能调用非异步信号安全的函数（如 printf、malloc），否则可能导致死锁。

为什么需要用户态/内核态的分离？

现代 CPU 至少提供两个特权级（x86 的 Ring 0/3，ARM 的 EL0/EL1）。内核态代码可以访问所有硬件资源（内存、磁盘、网络、中断控制器），执行特权指令（如修改页表基址寄存器 CR3、关中断 CLI/STI）。用户态代码被严格限制——不能直接访问硬件，不能修改自己的页表，所有涉及硬件或全局状态的操作必须通过系统调用委托内核完成。这种隔离保证了恶意或有缺陷的用户程序不会破坏整个系统。

三种进入内核态的方式：

1. 系统调用（主动）：用户程序主动请求内核服务（read/write/open/fork），通过 syscall 指令（x86-64）触发，是同步的。2. 中断（被动，硬件触发）：外部设备（键盘、网卡、定时器）向 CPU 发出中断信号，CPU 暂停当前执行的用户代码，转入中断处理程序。3. 异常（被动，CPU 自身触发）：当前指令执行出错（缺页异常、除零错误、非法指令），CPU 自动转入异常处理程序。其中缺页异常是最常见的——虚拟内存机制的正常组成部分。

系统调用的开销构成：

一次系统调用至少包含：1) 触发 syscall 指令（保存用户态 RIP/RFLAGS 到 MSR 寄存器，加载内核态 RIP/RFLAGS），2) 保存通用寄存器到内核栈，3) 根据系统调用号（rax）跳转到对应的内核函数，4) 执行实际逻辑（可能涉及磁盘 IO、网络收发、内存分配），5) 恢复寄存器，6) sysret 返回用户态。其中第 2-6 步消耗几百个 CPU 周期。如果系统调用期间发生进程切换，TLB 刷新带来的间接开销可能远超系统调用本身。银行系统的零拷贝技术（Java NIO 的 DirectByteBuffer 底层用 mmap，Kafka 用 sendfile）就是为了减少这些用户态/内核态之间的数据拷贝。

4. 零拷贝技术（银行场景重点）

银行后台为什么关注零拷贝？ 核心交易系统每秒处理数万笔交易，每笔交易数据都要在进程间、内存与磁盘/网络之间传输。传统的 read() + write() 模式需要 4 次上下文切换 + 4 次数据拷贝（2 次 DMA + 2 次 CPU 拷贝），严重浪费 CPU。零拷贝技术将这个开销降到最低。
技术拷贝次数原理
传统 read + write4 次上下文切换 + 4 次拷贝read: DMA→内核缓冲区→用户缓冲区; write: 用户缓冲区→内核 Socket 缓冲区→DMA

传统文件传输的完整流程：1) 用户调用 read()→ 切换到内核态；2) DMA 将磁盘数据拷贝到内核读缓冲区；3) CPU 将数据从内核读缓冲区拷贝到用户缓冲区；4) 切换到用户态，read() 返回。5) 用户调用 write()→ 切换到内核态；6) CPU 将数据从用户缓冲区拷贝到内核 Socket 缓冲区；7) DMA 将数据从 Socket 缓冲区拷贝到网卡；8) 切换到用户态，write() 返回。其中步骤 3 和 6 的 CPU 拷贝是完全多余的——数据本来就在内核中，只是为了经过用户态的"中转站"。

mmap + write4 次上下文切换 + 3 次拷贝省去内核→用户缓冲区的 CPU 拷贝

mmap() 将内核读缓冲区的物理页直接映射到用户进程的虚拟地址空间，用户进程就像读写自己的内存一样访问文件数据——实际上读的是同一块物理内存。所以 read() 那一步的 CPU 拷贝（内核→用户）被省掉了，数据直接从内核读缓冲区"复制"到 Socket 缓冲区。Java NIO 的 MappedByteBuffer 就是基于 mmap 实现的。Kafka 早期版本大量使用 mmap 来读写日志文件，利用 OS 的页缓存来管理消息数据。

sendfile2 次上下文切换 + 2 次拷贝完全在内核态完成，不经过用户态

sendfile() 系统调用是最彻底的零拷贝方案。用户只需一次调用，指定源文件描述符和目标 Socket 描述符，内核直接在内部完成：DMA 拷贝磁盘→内核缓冲区，然后 CPU 拷贝内核缓冲区→Socket 缓冲区（如果需要），最后 DMA 拷贝 Socket 缓冲区→网卡。整个过程只有 2 次上下文切换（进入和退出 sendfile），用户态代码完全不需要接触数据。如果网卡支持 scatter-gather DMA（SG-DMA），连"内核→Socket 缓冲区"那一步 CPU 拷贝也能省掉——只传递描述符，由 DMA 引擎直接从内核缓冲区组装网络包。

splice不拷贝数据，管道间传递页引用两个 fd 间零拷贝传输

splice() 是 Linux 2.6.17 引入的更通用的零拷贝方案。它不拷贝数据本身，而是在内核中传递内存页的引用（相当于指针传递）。两个文件描述符之间传输数据时，内核只是把"这个物理页现在归你了"的引用从一个 fd 转移到另一个 fd。这要求其中一个必须是管道——因为管道的环形缓冲区是用页结构管理的，方便做页引用转移。Nginx 在高版本中逐渐从 sendfile 转向 splice 配合使用以支持更灵活的代理场景。

死锁的四个必要条件（缺一不可）：

死锁发生的四个条件必须同时满足，破坏其中任意一个就能预防死锁。在实际系统设计中，最常被打破的是"循环等待"——通过资源有序分配（所有线程按固定顺序获取锁）来避免环的形成。

银行家算法（Dijkstra, 1965）：

银行家算法是一种死锁避免策略（而非死锁预防）。它的核心思想与银行贷款审批如出一辙：系统维护当前可用资源量、每个进程已分配的资源量和最大需求量。当一个进程请求资源时，系统先试探性地假设分配给它，然后检查是否存在一个"安全序列"——按照这个序列依次满足所有进程的需求，每个进程都能运行完毕并释放资源。如果存在安全序列，则分配是安全的；否则拒绝分配，进程必须等待。算法复杂度 O(n² × m)，在大型系统中实时运行代价较高，更多用于教学和理论分析。

银行系统的实际做法：

真正的银行核心系统通常不会在运行时动态检测死锁——代价太高。而是通过严格的锁获取顺序约定（如：先账户 ID 小的锁，再账户 ID 大的锁）、锁超时机制（获取锁超时则回滚事务并重试）和数据库死锁检测（Oracle/MySQL 后台有死锁检测线程，定期检查等待图，发现环则回滚代价最小的事务）来综合应对。

条件含义破解措施可行性
互斥资源同一时刻只能被一个进程使用资源设计为可共享不可行

互斥是很多资源的固有属性——打印机不能同时打印两个文档、数据库的一行记录不能同时被两个事务修改、银行账户余额不能同时被两个线程更新。将资源设计为可共享虽然理论上能打破互斥条件，但大多数场景下做不到。例如 ReentrantReadWriteLock 允许读共享——多个读线程可以同时持有读锁，读操作之间不互斥，但读写之间仍然互斥。这算是部分打破了互斥条件。

占有并等待持有资源的同时等待其他资源一次性申请所有资源困难

一次性申请所有资源的思路：进程在开始执行前声明并获取所有需要的资源，如果任一资源不可用则释放已获得的所有资源并等待重试。问题在于：1) 很多场景下无法预知需要哪些资源（如转账交易在运行时才知道要锁哪两个账户）；2) 资源利用率低（提前占用了暂时不需要的资源）。数据库中 SELECT ... FOR UPDATE NOWAIT 就是这种思想——如果锁不可得直接返回错误而非阻塞，避免占着 A 等 B 的局面。

不可剥夺资源只能由持有者主动释放设计为可剥夺部分可行

CPU 是可剥夺的典型——调度器可以在任何时候把 CPU 从一个进程手中抢走分给另一个进程。但数据库锁、文件锁这类逻辑资源很难被剥夺——如果操作系统强行收回一个数据库事务的锁，该事务的所有修改都会处于不确定状态，可能导致数据不一致。实际上合理的做法是：如果一个线程占着 A 资源请求 B 资源失败，则主动释放已持有的 A 资源，等待一段时间后重新尝试获取 A 和 B——这绕过了"不可剥夺"的限制，但增加了"活锁"的风险。

循环等待存在 P1→P2→...→Pn→P1 的资源等待环资源有序分配（锁排序）最可行

资源有序分配（Lock Ordering）是工业界最常用的死锁预防手段。给所有资源分配一个偏序编号，要求每个线程按编号递增的顺序获取资源。例如转账场景：始终先锁 ID 小的账户再锁 ID 大的——线程 A 转账 100→200 和线程 B 转账 200→100，按这个规则都会先锁 100 再锁 200，从而避免了交叉等待形成的环。Java 中的 synchronized 或 ReentrantLock 配合 Lock.tryLock(timeout, unit) 是实现锁排序的标准方式。

Java 死锁检测实战

检测手段：

1. jstack -l <PID> 会扫描所有线程的锁等待关系，检测到死锁时输出 Found one Java-level deadlock 和详细的线程栈信息。2. ThreadMXBean.findDeadlockedThreads() 可在运行时编程式检测死锁。3. 数据库的死锁检测：MySQL InnoDB 有死锁检测线程，默认每 5 秒检查一次等待图，发现环后会回滚代价最小的事务（innodb_deadlock_detect 参数控制）。

虚拟内存的三大核心价值：

1. 隔离：每个进程拥有独立的虚拟地址空间（32 位系统 4GB，64 位系统理论上 256TB），进程 A 无法通过指针访问进程 B 的内存（除非主动通过共享内存映射）。这是操作系统安全性的基石。2. 超卖：所有进程的虚拟内存总和可以远超物理内存大小——操作系统利用磁盘上的 swap 区域将不活跃的内存页换出，物理内存只保留活跃的工作集（Working Set）。3. 简化程序加载：可执行文件的代码段和数据段可以被映射到虚拟地址空间而不需要全部加载到内存（demand paging），遇到缺页时再从磁盘加载。

地址转换流程（x86-64 4 级页表）：

虚拟地址被分割为多个字段（每级页表的索引 + 页内偏移）：PML4 索引(9bit) → PDPT 索引(9bit) → PD 索引(9bit) → PT 索引(9bit) → 页内偏移(12bit)。每次地址转换，MMU 从 CR3 寄存器拿到 PML4 表的物理基址，逐级查找。每一级都涉及一次内存访问，所以一次完整的 4 级页表遍历需要 4 次内存访问——这就是为什么 TLB 如此重要。TLB（Translation Lookaside Buffer）是 CPU 内部的硬件缓存，存放最近用过的虚拟→物理地址映射。典型 TLB 命中率 > 99%，命中时地址转换只需要 1 个 CPU 周期；不命中时需要完整的页表遍历，耗时 100+ 周期。

与 JVM 的关系：

JVM 运行在虚拟内存之上，JVM 的堆、栈、方法区、直接内存都是对操作系统虚拟内存的二次抽象。JVM 通过 -Xms / -Xmx 限制堆大小，但 JVM 进程的实际虚拟内存占用还包括：线程栈（每线程默认 1MB）、Metaspace（使用 native memory）、Direct ByteBuffer 分配的堆外内存、JIT 编译后的代码缓存等。理解虚拟内存是理解 JVM 内存管理和 OOM 问题的前提。

7. 分页 vs 分段

维度分页 (Paging)分段 (Segmentation)
划分方式固定大小（4KB），物理上划分可变大小，逻辑上划分

分页是物理视角的划分：不管程序逻辑结构如何，内存被切分为固定大小的页框（Page Frame，通常 4KB）。分段是逻辑视角的划分：代码段、数据段、堆、栈各自形成一个逻辑段，段的大小根据实际需要动态增长。这是两种截然不同的设计哲学——分页追求管理的简单高效，分段追求让程序的内存布局与其逻辑结构一致。

碎片类型内部碎片（最后一页用不完）外部碎片（空闲空间被割裂）

分页的内部碎片：如果程序需要 10KB 内存而页大小是 4KB，则分配 3 个页面（12KB），最后一页只用了 2KB → 内部浪费 2KB。分段的内部碎片为 0（因为段大小等于实际需要），但会产生外部碎片：随着段的分配和释放，物理内存中散布着很多不连续的、太小的空闲区域，无法满足大段的分配请求。这就是"碎片化"——总空闲 100MB，但最大的连续空闲块只有 10MB。

外部碎片无有（需要紧缩 compact）

分页完全消除了外部碎片——因为所有页面等大，任何空闲页面都可以分配给任何进程，不存在"这块空闲太小用不了"的问题。分段的外部碎片需要内存紧缩（Memory Compaction）来解决：操作系统暂停所有进程，移动已分配的内存块使其连续，腾出大的空闲空间。这非常耗时，类似于 JVM 的 Full GC 中的 compaction 过程。现代操作系统实际上使用段页式——程序暴露段接口给编译器（代码段、数据段），但底层用分页实现（段的基址和限长被转换为页表项）。

地址空间一维（所有地址连续）二维（段号 + 段内偏移）

分页下程序看到的地址是一维线性空间，任何合法地址都是一个数（0 到最大虚拟地址）。分段下地址是二维的 —— 程序员需要指定 "代码段的偏移 0x100" 而非简单的 "地址 0x40100"。这对编译器和程序员都不友好，是分段被淘汰的原因之一。x86 上遗留了分段的痕迹（CS/DS/SS/ES 段寄存器），但现代操作系统将所有段基址都设为 0（平坦模型），分段实际上被架空了。

现代应用所有现代 OS 的核心机制主流 OS 中已被淘汰

Linux、Windows、macOS 的虚拟内存管理全部基于分页实现。分段只在一些专用的嵌入式或实时操作系统中还有使用（如某些 RTOS）。Linux 的 mmap 允许将文件映射到一段虚拟地址——看起来像是"段"的概念，但底层仍然是分页管理（每个映射的文件页面按需加载）。可以说，分段的设计思想（逻辑上与程序结构对应的内存区域）仍然存在，但实现已经全部用分页替换。

LRU 的理论优势与实现困境：

LRU（Least Recently Used）利用程序的时间局部性——最近访问过的页面很可能在短期内再次被访问。理论上 LRU 接近最优（OPT），但实现代价高昂：每次内存访问都需要更新 LRU 排序信息。纯软件实现需要一个链表，每次页面访问都把它移到链表头部 → 链表操作本身就要几条指令的 CPU 时间，对每次内存访问都做这个操作完全不现实。硬件实现需要每个页面一个"访问时间戳"寄存器，并在每次访问时更新——面积和功耗代价都太大。

Linux 的 Clock 算法近似：

Linux 使用双链表 LRU 近似策略——维护 active_list（活跃页）和 inactive_list（非活跃页）。每个物理页有 PG_referenced 标志位（由 MMU 硬件在页被访问时自动置位）。内核定期扫描：如果 PG_referenced=1，清除标志并留在 active_list（给第二次机会）；如果 PG_referenced=0，将页从 active_list 移到 inactive_list。需要换出时，从 inactive_list 尾部取出页面。这就是 Clock（时钟）算法的变体——不是严格 LRU，但工程上非常高效（无每次访问的开销，只需定期扫描时读取标志位）。

Belady 异常：

FIFO 算法独有的反直觉现象——增加物理页框数量，缺页中断次数反而增加。根本原因是 FIFO 无视页面的访问频率——一个被频繁访问的页面可能仅仅因为"来得早"就被换出，即使它即将被再次访问。LRU 没有 Belady 异常，因为 LRU 满足栈特性（增加页框相当于在栈上做更浅的剪枝，不会影响已有剪枝的结果）。

IO 模型的本质区别——在哪一步阻塞：

IO 操作分为两个阶段：阶段一，等待数据从网络/磁盘到达内核缓冲区（数据就绪）；阶段二，将数据从内核缓冲区复制到用户缓冲区（数据拷贝）。阻塞 IO 在两个阶段都阻塞；非阻塞 IO 在阶段一不阻塞（立即返回 EWOULDBLOCK，需要反复轮询），但阶段二仍然阻塞；IO 多路复用在阶段一阻塞（在 select/epoll 上等待），阶段二在 recvfrom 中阻塞；异步 IO 两个阶段都不阻塞——发起 aio_read 后立即返回，内核完成所有工作后通知用户。只有异步 IO 是真正的"非阻塞"——Java AIO（NIO2）实现了真正的异步 IO。

select vs poll vs epoll 的技术差异：

select 将监控的 fd 集合从用户态拷贝到内核，内核遍历整个集合（O(n)），每次调用后需要 reinitialize 整个集合。fd_set 默认 1024 上限（可重新编译内核修改）。poll 将 fd 集合组织为链表，没有数量限制，但内核仍然 O(n) 遍历。epoll 通过 epoll_ctl 将 fd 注册到内核的红黑树 + 就绪链表中——每个 fd 只需要注册一次（不需要每次调用都传入整个集合），内核通过回调机制在 fd 就绪时将其加入就绪链表，epoll_wait 直接遍历就绪链表（O(1) 已就绪 fd 的数量而非 O(n) 总 fd 数）。这就是为什么 C10K 问题的解决依赖 epoll——一万个连接中只有几个活跃时，select 每次要扫描一万个 fd，epoll 只返回活跃的那几个。

IO 模型阶段一（等待数据）阶段二（拷贝数据）代表
阻塞 IO (BIO)阻塞阻塞传统 Socket read/write，JDBC

阻塞 IO 是最简单直观的模型：线程调用 read() 后挂起，直到数据到达并被拷贝到用户缓冲区才返回。在这期间线程不能做任何事情——对于单线程服务器，这意味着它无法处理任何其他连接。所以传统的阻塞 IO 服务器必须使用一个线程处理一个连接的模型（Thread-per-Connection），线程数量 = 并发连接数。在连接数很大的场景下，大量线程处于阻塞等待状态，浪费内存（每个线程栈 1MB），线程切换开销巨大。

非阻塞 IO (NIO)不阻塞（轮询）阻塞SocketChannel + 轮询

非阻塞 IO 模式下，read() 调用立即返回——如果数据没准备好就返回错误码（Linux 上是 EAGAIN/EWOULDBLOCK），应用程序需要反复调用 read() "轮询"直到数据就绪。虽然线程不用在阶段一阻塞，但轮询消耗了大量 CPU 时间——大部分 read() 调用都返回"没数据"，白白浪费 CPU 周期。纯非阻塞 IO 很少直接使用，通常结合 IO 多路复用——让内核帮忙轮询，有数据时才通知用户。

IO 多路复用在 select/epoll 上阻塞在 recvfrom 上阻塞Netty, Nginx, Redis

IO 多路复用的核心思想：用一个线程同时监控多个文件描述符，任何一个就绪就通知应用程序——而不是为每个连接分配一个线程。这就是 Reactor 模式。Netty 的 EventLoop 就是这种实现——一个 EventLoop 线程监控成百上千个连接，哪个连接有数据到达就处理哪个。阻塞 IO 和 IO 多路复用的本质区别不是"是否阻塞"——IO 多路复用也是阻塞在 select/epoll 上的——而是一个线程能处理多个连接 vs 一个线程只能处理一个连接。

信号驱动 IO异步（SIGIO）阻塞较少使用

信号驱动 IO 预先通过 sigaction 注册 SIGIO 信号处理函数，然后立即返回。当数据在内核中就绪后，内核发送 SIGIO 信号给进程，在信号处理函数中调用 recvfrom 完成数据拷贝。问题是：信号处理函数中能做的事情非常有限（不能调用非异步信号安全的函数），且信号合并问题导致多个 IO 事件可能只触发一个信号。在 epoll 成为主流后，信号驱动 IO 基本退出历史舞台。

异步 IO (AIO)不阻塞不阻塞（全异步）Java NIO2 (AsynchronousSocketChannel)

异步 IO 是唯一从头到尾都不阻塞的 IO 模型。应用发起 aio_read() 后立即返回（连内核拷贝数据都不等），内核完成所有工作（等待数据就绪 + 拷贝到用户缓冲区）后通过回调或信号通知应用。Java 的 AsynchronousSocketChannel 就是 AIO 的实现。理论上 AIO 性能最优，但在 Linux 上 AIO 的实现存在一些限制：内核 AIO 对网络 IO 的支持要等到 io_uring（Linux 5.1+）才真正完善。这也是为什么 Netty 团队经过评估后仍然坚定使用 IO 多路复用而非 AIO——epoll 足够高效且更成熟可控。

10. CPU 调度算法

算法原理优点缺点饥饿风险
FCFS先来先服务，非抢占简单公平短任务被长任务阻塞无

FCFS (First Come First Served) 是最简单的调度算法——按任务到达顺序依次执行，当前任务完成前绝不切换。问题出在护航效应（Convoy Effect）：一个长 CPU 密集型任务先到达并占用 CPU，后续大量的短 IO 密集型任务全部排队等待——而这些短任务本来可以很快完成并释放 IO 资源。在分时系统中 FCFS 几乎不被采用，但在批处理系统中仍然有意义（所有任务都是已知的，目标是最大化吞吐量而非响应时间）。

SJF预计运行时间最短优先最小化平均等待时间需预知运行时间有

SJF (Shortest Job First) 在数学上能最小化平均等待时间——可以证明，将短任务排在长任务前面能让"等待队列的长度"最小化。但问题是：操作系统如何知道一个任务需要运行多长时间？指数平均法可以近似：记录每个任务的历史运行时间，预测下一次运行时间 = α × 上一次实际运行时间 + (1-α) × 上一次预测值。α 取 0.5 时相当于加权平均。长任务可能被源源不断的短任务插队而永远得不到执行——这是典型的饥饿。

优先级按优先级调度，高优先级先执行灵活可控低优先级可能饿死有

优先级调度在实时系统中非常关键——紧急任务（如核电站的紧急停机信号）必须立刻获得 CPU。但静态优先级的问题是低优先级任务可能永远得不到执行（被高优先级任务持续抢占）。老化（Aging）机制是经典解决方案：随着等待时间增加，动态提高任务的优先级。Linux 的 CFS 调度器通过 vruntime（虚拟运行时间）隐式实现了老化——vruntime 越小的任务优先级越高，等待的任务 vruntime 不变而运行中任务的 vruntime 增长，自然达到平衡。

Round Robin时间片轮转，抢占式公平，响应时间可控时间片选择困难无

Round Robin (RR) 是分时系统的基础：每个任务分配一个固定的时间片（通常 10-100ms），用完后放到队列末尾，轮流执行。时间片选择是门学问——太小（如 1ms）会导致频繁的上下文切换（切换开销占比大，实际工作效率低），太大（如 500ms）则退化为 FCFS，交互式应用的响应变得迟缓。Linux 默认时间片在 10-20ms 左右（CFS 使用动态时间片）。Java 线程调度完全交给操作系统，Thread.yield() 只是建议调度器"可以考虑给别人机会"，但不保证。

多级反馈队列 (MLFQ)多队列 + 动态优先级调整兼顾交互与吞吐参数调优复杂低

MLFQ (Multi-Level Feedback Queue) 是现代通用操作系统的真正实现思路。它维护多个优先级队列，高优先级队列分配短时间片（保证交互响应），低优先级队列分配长时间片（提高吞吐量）。任务首次进入最高优先级队列，用完时间片后降级到下一级；如果在低优队列中长时间等待后获得了 CPU（说明它被饿到了），则升级。IO 密集型任务往往用不完时间片（主动让出 CPU），所以停留在高优先级；CPU 密集型任务会逐渐降级。MLFQ 不需要预知任务的运行时间——它通过反馈机制自适应。Linux 的 CFS 可以看作 MLFQ 的连续版本。

Linux CFS红黑树管理，vruntime 追踪完全公平，O(log N)—无

Linux CFS (Completely Fair Scheduler) 从 Linux 2.6.23 开始成为默认调度器。它的核心数据结构是一棵红黑树，按照每个任务的 vruntime（虚拟运行时间）排序。调度器总是选择红黑树最左边的节点（vruntime 最小的任务）。vruntime 的增长速度与任务的权重（nice 值）成反比——nice 值越低（优先级越高）的任务 vruntime 增长越慢，所以它"在树中往右移"的速度越慢，被调度的次数越多。CFS 不使用固定时间片，而是根据任务数量和权重动态计算目标延迟（targeted latency），在该延迟内所有可运行任务至少获得一次 CPU。

五、综合高频考点

11. 缺页中断 (Page Fault)

缺页中断是虚拟内存机制的正常组成部分，不是错误！ 实际上，进程启动时几乎没有页面在物理内存中——所有代码、数据页面都是"映射了但未加载"状态。进程执行第一条指令就触发缺页中断，操作系统从磁盘加载代码页，然后重新执行该指令。这个过程对用户程序完全透明。

缺页中断的完整流程：

1. MMU 查找页表，发现该页的 Present 位为 0（不在物理内存中）→ 触发缺页异常（trap 到内核）。

2. 内核检查该虚拟地址是否在进程的合法地址范围内（VMA 区域检查）——如果不是，则发送 SIGSEGV 信号给进程（这就是著名的"Segmentation Fault"）。

3. 如果是合法但未映射或已换出的页：找一块空闲物理内存页框（如果物理内存已满则根据页面置换算法选一个页换出到磁盘）。

4. 从磁盘（可执行文件或 swap 分区）读取目标页面数据到分配的物理页框中。

5. 更新页表——将该虚拟页映射到刚分配的物理页框，设置 Present=1 并清除其他标志。

6. 重新执行触发缺页的指令——这次 MMU 能成功翻译地址，程序继续执行，就像什么都没发生。

次缺页 (Minor Page Fault) vs 主缺页 (Major Page Fault)： 如果目标页面已经在物理内存的"页缓存"中（只是还没有映射到这个进程的页表中），则只需更新页表映射即可，不需要磁盘 IO——这叫 minor fault。如果需要从磁盘读取——这叫 major fault，性能昂贵。Java 应用中，频繁的 major page fault 可能意味着堆内存不足（GC 过于频繁地在堆和 swap 之间换页），需要通过 -Xmx 限制堆大小来避免。

12. 线程同步与互斥（银行场景）

银行转账的经典并发问题： A 向 B 转账 100 元，同时 B 向 A 转账 200 元。如果两个线程不加控制地并发执行：A=A-100, B=B+100, B=B-200, A=A+200——最后的余额取决于指令的交错顺序，而不是正确的业务逻辑。这就是竞态条件——两个线程的最终结果依赖于它们指令执行的相对时序。

操作系统的同步原语——互斥锁 vs 条件变量 vs 信号量：

互斥锁（Mutex）解决互斥访问问题——同一时刻只有一个线程能访问临界区。条件变量（Condition Variable）解决等待条件成立的问题——线程 A 等待"账户余额 >= 取款金额"这个条件，线程 B 存款后通知 A。信号量（Semaphore）既是互斥也是同步——可以控制"N 个资源同时被使用"。这三个构成了并发编程的基础，Java 中的 synchronized/Lock、wait/notify、Semaphore 分别对应了它们。

乐观锁 vs 悲观锁——银行系统的选择：

悲观锁（Pessimistic Lock）假设冲突一定会发生，先锁再操作——适合写密集场景。乐观锁（Optimistic Lock）假设冲突不会发生，操作完在提交时检查（CAS 或版本号）——适合读多写少场景。银行核心账务系统倾向悲观锁（资金安全性高于吞吐量），而用户余额查询系统可以使用乐观锁（高并发读取，冲突概率低）。Java 中 synchronized/ReentrantLock 是悲观锁，AtomicInteger.compareAndSet 是乐观锁（CAS 由 CPU 指令实现，如 x86 的 LOCK CMPXCHG）。
