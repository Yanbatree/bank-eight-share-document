一句话：Harness 是 Agent 的"操作系统内核"——LLM 负责"想"，Harness 负责"做"。它管理着 LLM 与外部世界交互的每一个环节：执行什么命令、谁能执行、执行前要做什么检查、执行后要做什么记录。

5.1 Harness 核心职责

职责做什么为什么重要
工具执行将 LLM 输出的 tool_call 翻译为真实系统调用LLM 不直接执行代码，需要中间层翻译和管控
权限控制allow / deny / ask 三级权限模型防止 Agent 越权操作（删库、访问敏感文件）
Hook 系统在工具执行前后、会话启停时注入自定义逻辑企业合规（提交前自动扫描、操作前二次确认）
沙箱隔离限制 Agent 可访问的文件系统、网络、进程安全第一——Agent 代码执行必须在受限环境
会话状态管理上下文窗口、后台任务、工作目录长任务需要状态持久化和任务编排

5.2 Hook 机制详解

sequenceDiagram participant L as LLM participant H as Harness participant PreH as PreToolUse Hook participant Tool as 实际工具 participant PostH as PostToolUse Hook L->>H: 决定调用 Read / Bash / Write ... H->>PreH: 触发 PreToolUse 事件 PreH-->>H: 返回 (允许/拒绝/修改参数) alt Hook 拒绝 H-->>L: 返回 "操作被阻止" else Hook 允许 H->>Tool: 执行实际工具调用 Tool-->>H: 返回执行结果 H->>PostH: 触发 PostToolUse 事件 PostH-->>H: 返回 (记录日志/发送通知/修改结果) H-->>L: 返回最终结果 end
常见 Hook 事件类型：
事件触发时机典型用途
PreToolUse工具调用之前审批卡点、参数校验、敏感操作拦截
PostToolUse工具调用之后审计日志、结果校验、自动提交
Notification特定通知事件权限请求转发、异常告警
SessionStart会话启动时环境初始化、加载项目配置
SessionEnd / Stop会话结束时清理临时文件、上报使用统计
PreCompact上下文压缩前保存关键信息防止被压缩丢失

5.3 权限模型（安全核心）

flowchart TD ToolCall["LLM 发出工具调用"] --> Check{权限检查} Check -->|"allow (白名单)"| Execute["直接执行"] Check -->|"deny (黑名单)"| Block["阻止 + 告知 LLM"] Check -->|"ask (需确认)"| Prompt["弹出确认框给用户"] Prompt -->|"用户同意"| Execute Prompt -->|"用户拒绝"| Block Execute --> SandboxCheck{沙箱检查} SandboxCheck -->|"在允许范围内"| Run["真正执行"] SandboxCheck -->|"越界"| Block
权限粒度设计： 
// 权限配置示例（settings.json）
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",        // 允许运行 npm 脚本
      "Bash(git diff *)",       // 允许查看 git diff
      "Bash(git log *)"         // 允许查看 git log
    ],
    "deny": [
      "Bash(rm -rf *)",         // 禁止递归删除
      "Bash(git push --force)", // 禁止强制推送
      "Read(.env)",             // 禁止读取环境变量文件
      "Read(*credentials*)"     // 禁止读取凭证文件
    ],
    "ask": [
      "Bash(git push *)"        // git push 需要确认
    ]
  }
}
银行场景的权限设计原则： • 默认拒绝：非白名单操作一律拒绝或需审批 • 最小权限：Agent 只能访问完成任务所需的最小资源集 • 读写分离：读操作可宽松，写操作必须严格管控 • 审批链路：高风险操作（数据修改、生产部署）必须走审批 Hook

二、MCP 协议

1. MCP 是什么？（Model Context Protocol）

graph LR subgraph "MCP 之前 - 每个 LLM 单独对接" L1["Claude"] --> DB1["数据库插件"] L2["GPT"] --> DB2["数据库插件"] L3["Gemini"] --> DB3["数据库插件"] end subgraph "MCP 之后 - 统一协议" L4["任何 LLM 客户端"] --> MCP["MCP 协议"] MCP --> Server1["DB Server"] MCP --> Server2["File Server"] MCP --> Server3["API Server"] end
一句话：MCP 是 AI 应用的"USB-C 接口"——定义了 LLM 如何与外部工具/数据源交互的统一标准协议，由 Anthropic 在 2024 年底开源。 类比：LSP（Language Server Protocol）标准化了 IDE 和语言工具的通信，MCP 标准化了 LLM 和外部工具的通信。

MCP 三大核心概念：
概念作用举例
Resources暴露数据（只读，类似 GET）文件内容、数据库记录、文档
Tools暴露操作（可执行动作）创建文件、发送消息、查询 API
Prompts预定义的提示模板代码审查模板、SQL 生成模板

3. MCP vs Function Calling

Function CallingMCP
定义方每个 LLM 厂商自己的规范统一的开源协议
生态各自为政社区共建 Server，一次开发到处用
发现机制开发者手动声明函数Server 自描述，Client 自动发现工具
安全性依赖各厂商明确的权限模型 + 沙箱
传输HTTP/gRPC（各厂商不同）标准 stdio / SSE
关系：MCP 可以理解为 Function Calling 的"标准化升级版"。

4. MCP 核心通信机制

sequenceDiagram participant Host as MCP Host participant Client as MCP Client participant Server as MCP Server Note over Host,Server: ① 初始化握手 Host->>Client: 启动 Server Client->>Server: initialize (协议版本 + 客户端信息) Server-->>Client: capabilities (支持的能力列表) Note over Host,Server: ② 发现工具 Client->>Server: tools/list Server-->>Client: [{name, description, inputSchema}] Note over Host,Server: ③ LLM 决定调用工具 Client->>Server: tools/call {name, arguments} Server-->>Client: {content: [...]} Note over Host,Server: ④ 读取资源（可选） Client->>Server: resources/read {uri} Server-->>Client: {contents: [...]}
面试加分点：能说出 MCP 是 LLM ↔ 外部世界的标准协议，类比 LSP，先说 Protocol 层（Resources/Tools/Prompts），再说 Transport 层（stdio/SSE），面试官会觉得你真正理解了这个东西。

2. Tool Use / Function Calling 原理

工作流程： 
① 开发者定义工具 schema（名称、描述、参数 JSON Schema）
② 用户输入问题 → LLM 判断是否需要调用工具
③ 需要 → LLM 输出 tool_call（工具名 + 参数）而非文本
④ 客户端执行工具调用，获取结果
⑤ 将结果传回 LLM → LLM 基于结果生成最终回答
关键：LLM 不直接执行工具，它是"决策者"——决定调用哪个工具和传什么参数，实际执行由客户端完成。

手写一个简单的 MCP Server demo： 
// 简易 MCP Server - 提供天气查询工具
import { Server } from "@modelcontextprotocol/sdk/server/index.js";

const server = new Server({
  name: "weather-server",
  version: "1.0.0"
}, {
  capabilities: { tools: {} }
});

server.setRequestHandler("tools/list", async () => ({
  tools: [{
    name: "get_weather",
    description: "查询指定城市的天气",
    inputSchema: {
      type: "object",
      properties: { city: { type: "string" } },
      required: ["city"]
    }
  }]
}));

server.setRequestHandler("tools/call", async (request) => {
  const { city } = request.params.arguments;
  return { content: [{ type: "text", text: `${city}: 晴, 25°C` }] };
});
